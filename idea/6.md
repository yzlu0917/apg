## 1. 方向地图（含主题覆盖）

> 范围：以 **2024-09 ~ 2026-03**（最近 6–18 个月）在 arXiv / OpenReview / 顶会（NeurIPS/ICLR/ICML/ACL/EMNLP）高讨论方向为主；本次深挖 **8 个方向**，同时在每个主题里指出“主线/痛点/重要性”。（**Step A：广撒网** 已执行）

### 1.1 推理与验证（reasoning / verifier / self-correction）

* 主线：从“输出级奖励/评测”转向 **过程级监督（PRM/过程验证器）+ test-time compute（TTS）**，用 verifier/PRM 选择或修正推理轨迹。([ICLR 会议录][1])
* 痛点：验证器易被 hacking、过程标签贵、不同 TTS 策略强依赖任务难度与训练范式。([arXiv][2])
* 重要性：推理与 agent 行为越来越像“长链决策”，验证/纠错将成为可控智能的核心基础设施。

### 1.2 可靠性与对齐（robustness / honesty / calibration / uncertainty）

* 主线：把“幻觉/不诚实”从口号变成 **可测量对象**：语义不确定性、诚实度基准、幻觉检测再评估。([Nature][3])
* 痛点：free-form 输出下的 uncertainty 难稳定估计；很多“诚实/幻觉”指标混杂了能力与风格；黑盒模型难用 logprobs。([NeurIPS][4])
* 重要性：**agent 场景**里错误会“执行到外部世界”，可靠性不只是文本质量。

### 1.3 表征与机制解释（mechanistic interpretability / representation engineering）

* 主线：从“手工电路”到 **特征发现（SAE/字典学习）+ 自动电路发现 + 表征编辑（RepE）**，同时出现反思：SAE 特征是否真比原模型更可解释？([NeurIPS][5])
* 痛点：可解释性评估本身不稳定；“自动解释”易宽泛；从相关到因果需要昂贵干预实验。([NeurIPS][6])
* 重要性：Best-paper 级别的“为什么有效”经常需要机制证据，而不是只报分。

### 1.4 训练与优化新范式（RLHF/RLAIF alternatives / off-policy RL / credit assignment）

* 主线：**DPO 家族爆发**（SimPO、KTO 等）+ **离策略/异步 RLHF/RLVR**（OAPL、BAPO 等），解决工程现实中的 policy lag、样本效率与稳定性。([arXiv][7])
* 痛点：偏好数据噪声与长度偏置；off-policy 校正难；RLVR 容易“奖励同质化/经验浪费”。([arXiv][8])
* 重要性：在不堆预训练算力的约束下，**post-training** 是最可控、最可复现的创新空间。

### 1.5 数据合成与评测（synthetic data / evaluation / contamination）

* 主线：对 benchmark 可信度“开刀”：**污染因果量化、污染检测、leaderboard 审计、动态/新鲜基准**。([NeurIPS][9])
* 痛点：污染检测可被规避；leaderboard 有激励扭曲；“评测即训练”导致过拟合。([NeurIPS][10])
* 重要性：NeurIPS best paper 常青主题：**测量科学（measurement）** 本身就是贡献。

### 1.6 多智能体/工具使用（agents / tool use / planning / search）

* 主线：tool agent 从 demo 走向 **标准化基准（τ-bench/BFCL/StableToolBench）、真实任务（SWE-bench）、真实噪声（AgentNoiseBench）**；规划结构从 chain-of-thought 走向图规划、状态建模。([arXiv][11])
* 痛点：现实中工具会变、会坏、会被注入攻击；模型强依赖命名/格式捷径；长程依赖导致 credit assignment 困难。([OpenReview][12])
* 重要性：你偏好 agent，这也是顶会最拥挤但也最可能出“范式级贡献”的赛道。

### 1.7 检索与记忆（RAG / long-context / memory）

* 主线：从“检索+生成”到“类记忆系统”：LongBench v2、FreshStack、HippoRAG 2 等强调 **长上下文推理、现实新鲜域、联想/常识记忆**。([arXiv][13])
* 痛点：长上下文并不等于会用；检索增加噪声；评测易污染、域迁移难。([OpenReview][14])
* 重要性：Agent 的“外部记忆/工具知识”几乎都落在 RAG 与记忆系统上。

### 1.8 结构化生成/程序归纳（code / formal methods / theorem proving）

* 主线：从 pass@k 走向 **污染可控与可验证**：LiveCodeBench（动态收题）、NeurIPS 2025 的代码验证/测试范式、Lean 里的端到端验证（CLEVER 等）。([OpenReview][15])
* 痛点：验证信号昂贵但强；形式化与自然语言之间鸿沟；工具链复杂影响复现。([NeurIPS][16])
* 重要性：对 agent 来说，“写代码→跑测试→修复”是最典型的可验证闭环。

## 3. 深入对比（每个方向：现状→瓶颈→争议→可突破点）

### 3.1 A1 推理与验证（PRM/TTS）

**现状范式**

* PRM/过程验证：给推理每一步打分或排序，结合 beam/MCTS/重采样。
* TTS：在推理时投入更多 compute（多样采样、搜索、验证器筛选），但“最优策略依赖训练方式与题目难度”。([arXiv][2])

**代表论文（10篇，6–18个月内为主）**

* Scaling Automated Process Verifiers（ICLR 2025）：自动化过程验证器 + test-time 扩展。([ICLR 会议录][1])
* PRIME（2025）：仅用 outcome label 做在线 PRM 更新，缓解 online PRM 易 hacking。([OpenReview][20])
* PURE（2025）：min-form credit assignment 统一 test-time 与 training-time 的过程信号。([OpenReview][21])
* PQM（Process Q-value Model）：把 PRM 放进 MDP/Q-value 排序框架。([OpenReview][22])
* ThinkPRM（2025）：生成式长CoT PRM，强调少监督与可扩展验证。([arXiv][23])
* PPRM（Preference-based PRM）：偏好驱动的 step-wise 指导。([OpenReview][24])
* PAG（NeurIPS 2025）：policy 与“生成式验证器”交替，自选择是否 revision。([NeurIPS][25])
* S2R（ACL 2025）：自验证/自纠错的两阶段训练+RL。([ACL Anthology][26])
* Step-level Verifier-guided Hybrid TTS（EMNLP 2025）：step verifier 指导的混合 TTS。([ACL Anthology][27])
* The Art of Scaling Test-Time Compute（2025）：大规模比较 TTS 策略，结论“无单一最优”。([arXiv][2])

**主要瓶颈**

* PRM/Verifier 的 **偏置与 reward hacking**：验证器更像“第二个模型”，会引入系统性错误。([OpenReview][20])
* TTS 方法“相互不可比”：不同任务/预算下策略最优点不同。([arXiv][2])

**争议/失败案例**

* “更强 TTS 是否只是把错误推到更晚？”：可能提高 pass@k 但降低 first-pass。([arXiv][28])
* PRM 泛化：数学强不代表工具/对话强（多域 PRM 仍在早期）。([OpenReview][29])

**可突破点（适合 best-paper 的支点）**

* 统一视角：把 PRM/TTS 视为 **“可控的推理计算分配”**，给出可检验预测（何时应 search、何时应 verify）。
* 新评测：把“推理正确”拆成“**错误发现率/纠错效率/预算曲线**”，而不是单点准确率。

### 3.2 A2 工具使用与Agent（规划/鲁棒/真实噪声）

**现状范式**

* 工具调用评测从静态走向：多轮、规则、状态、真实噪声与工具不可用。([arXiv][11])
* 训练数据侧：自动合成（ToolACE）+ 标准平台（ToolBench/StableToolBench）。([ICLR 会议录][30])

**代表论文/基准（12篇）**

* τ-bench（2024）：tool-agent-user 动态对话 + 领域规则约束。([arXiv][11])
* StableToolBench（ACL Findings 2024）：虚拟 API server + 稳定评测，解决 API 变化与随机性。([arXiv][31])
* ToolBench（ICLR’24 spotlight，平台）：开源训练/评测平台，GitHub 5.5k⭐。([GitHub][32])
* BFCL（ICML 2025）：函数调用 leaderboard，AST 评测可扩展。([ICML][33])
* AgentNoiseBench（2026.02）：把“真实世界噪声”注入 agent benchmark，平均显著降幅。([arXiv][34])
* FAIL-TALMS（NAACL 2025）：研究“用户查询欠指定/工具不可用”的系统性失败。([ACL Anthology][35])
* WASP（NeurIPS 2025）：Web agent prompt injection 安全评测。([NeurIPS][36])
* T1（NeurIPS 2025）：多轮对话中工具依赖与规划评测数据集。([NeurIPS][37])
* Graph-based Agent Planning, GAP（OpenReview/2025）：显式子任务依赖图，支持并行/串行工具执行。([OpenReview][38])
* DyMo + SVS（OpenReview/2025.10）：状态环境下的 dynamics modelling 与自验证采样。([OpenReview][39])
* Robust Function-Calling（2024）：指出命名约定过拟合；强调 robust function-calling。([OpenReview][12])
* Don’t Adapt SLMs for Tools / PA-Tool（2025.10）：主张“适配 schema 给模型”，用 peakedness 选重命名。([arXiv][40])

**主要瓶颈**

* **Schema/命名捷径**：模型对 tool name/arg name 高敏感，真实系统里 schema 会演化。([OpenReview][12])
* **鲁棒性欠评测**：过去工具基准默认“工具可用、无噪声、无注入”。([arXiv][34])

**争议/失败案例**

* “更强模型 ≠ 更稳定 agent”：在规则/噪声/注入下可能退化。([arXiv][34])
* “工具能力”与“规划能力”纠缠：很难定位是 tool selection、argument grounding、还是 state tracking 的错。([NeurIPS][37])

**可突破点**

* 新原则：把工具使用看成 **“语义意图 → 可执行合约（contract）”** 的映射，schema 表面是 nuisance。
* 新评测透镜：**工具演化（rename/reorder/deprecate）+ 噪声注入** 的系统化“稳健曲线”。

### 3.3 A3 可靠性/诚实/不确定性（hallucination/uncertainty/honesty）

**现状范式**

* 从 token-level entropy 转向 **语义不确定性** 与 “confabulation/hallucination” 可检测子类。([Nature][3])
* 基准把“诚实”拆成：自知边界/不欺骗/一致性。([OpenReview][41])

**代表论文/基准（10篇）**

* Nature 2024：语义熵/统计方法检测一类幻觉（confabulations）。([Nature][3])
* Kernel Language Entropy（NeurIPS 2024）：白盒/黑盒 LLM 语义不确定性估计。([NeurIPS 会议记录][42])
* NeurIPS 2025：多样性引导采样提升语义UQ样本效率。([NeurIPS][4])
* BEHONEST（OpenReview）：诚实三维度基准。([OpenReview][41])
* Evaluating Honesty in LLMs（OpenReview）：指出“更大更准≠更诚实”。([OpenReview][43])
* Epistemic Integrity（OpenReview）：测量 assertiveness 等语言层面的“认知完整性”。([OpenReview][44])
* LLM-Check（NeurIPS 2024）：幻觉性质分析与检测方法比较。([NeurIPS][45])
* MetaQA（ACM 2025）：基于 metamorphic relation 的自包含幻觉检测。([ACM Digital Library][46])
* MHAD（IJCAI 2025）：内部表征探测的幻觉检测。([IJCAI][47])
* Re-evaluating Hallucination Detection（EMNLP 2025）：对检测方法做再评估，指出脆弱性。([ACL Anthology][48])

**主要瓶颈**

* “诚实/不确定性”定义容易被 **风格/拒答倾向** 污染。([OpenReview][49])
* black-box 场景无法直接用 logprobs，需要可迁移指标。([arXiv][50])

**可突破点**

* 面向 agent：把可靠性指标升级为 **“执行风险”**（例如工具调用会造成外部影响时的校准/拒绝策略）。
* 把 uncertainty 变成可用信号：用于 **规划时的分支/验证预算分配**。

### 3.4 A4 评测科学（污染/leaderboard/动态基准）

**现状范式**

* 用对照预训练实验做污染因果量化；审计 leaderboard 的激励扭曲；构建新鲜域基准。([NeurIPS][9])

**代表论文（10篇）**

* NeurIPS 2025：因果量化 test contamination 对 generative eval 的影响（通过重复注入训练数据）。([NeurIPS][9])
* ConStat（NeurIPS 2024）：性能驱动的污染检测与量化。([NeurIPS][51])
* How Contaminated Is Your Benchmark?（OpenReview）：测量污染的必要性与方法。([OpenReview][52])
* Search-Time Data Contamination（2025）：把“检索/搜索阶段”也纳入污染讨论。([OpenReview][53])
* The Impact of Post-training on Data Contamination（OpenReview）：研究 post-training 与污染交互。([OpenReview][54])
* The Leaderboard Illusion（NeurIPS 2025）：审计 Chatbot Arena，指出系统性问题。([NeurIPS][10])
* Survey on Data Contamination（2025）：梳理污染定义/评测策略。([arXiv][55])
* ResearchCodeBench（NeurIPS 2025）：评测“实现新论文想法”的代码能力，强调未见过的新颖性。([NeurIPS][56])
* Quantifying contamination effect（arXiv 2026.02）：对 MATH 等复制注入做量化实验。([arXiv][57])
* Synthetic data orchestration（OpenReview 2025）：讨论合成数据权衡与编排。([OpenReview][58])

**争议/失败案例**

* “污染检测”可能变成猫鼠游戏；leaderboard 生态会主动适配指标。([NeurIPS][10])

**可突破点**

* 面向 agent：提出“**动态工具/动态文档**”的 contamination-free 协议（类似 Live benchmark 思路）。
* 关键支点：把“评测”做成 **可复现的科学实验**（对照组、因果推断、置信区间）。

### 3.5 A5 机制解释（SAE/电路/RepE）

**现状范式**

* SAE/字典学习特征发现成为主流；但出现反例与方法学反思：SAE 是否真抓到“模型学到的概念”？([arXiv][59])

**代表论文（10篇）**

* Measuring Progress in Dictionary Learning（NeurIPS 2024）：讨论字典学习进展与评测。([NeurIPS][17])
* KV memories vs SAE（NeurIPS 2025）：提出“原模型参数里的结构几乎同样可解释”。([NeurIPS][5])
* SAE monosemantic in VLM（NeurIPS 2025）。([NeurIPS][60])
* SAE interpret random transformers（2025）：随机网络也能“被解释”，挑战直觉。([arXiv][59])
* Revising/Falsifying SAE feature explanations（NeurIPS 2025）：讨论自动解释的失败与修正。([NeurIPS][6])
* Efficient Automated Circuit Discovery（OpenReview）：自动化电路构建。([OpenReview][61])
* IBCircuit（OpenReview）：信息瓶颈视角做整体电路发现。([OpenReview][62])
* Circuit discovery for jailbreaking（OpenReview 2025）：对 jailbreak 行为做机制分析。([OpenReview][63])
* Representation Engineering survey（2025）：RepE 作为可控行为的新范式。([arXiv][64])
* Taxonomy/Challenges of RepE（2025）：总结机会与风险。([OpenReview][65])

**主要瓶颈**

* 评测“解释质量”很难统一；很多工作停在相关性。
* 与实际能力提升之间缺桥梁（best paper 的机会点）。

**可突破点**

* 把 mech interp 嵌入 agent：解释 **工具选择/规则遵守/被注入劫持** 的内部机制，并给出可控干预。

### 3.6 A6 Post-training 对齐与优化（DPO+off-policy RL）

**现状范式**

* DPO 作为 RLHF 替代已成默认；大量变体围绕“参考模型、长度偏置、噪声鲁棒、多人比较”。([arXiv][19])
* off-policy RLHF/RLVR 强调现实系统中的异步与 buffer。([arXiv][66])

**代表论文（10篇）**

* DPO（2023，但仍是主干基线）。([arXiv][19])

## 4. 候选课题（3–5 个最可能冲击 best paper）

> **Step C：找 gap 的系统化工具箱**（至少4种）我将用于下面每个候选：
>
> 1. Failure modes map；2) Assumption stress test；3) Counterfactual baseline；4) New eval lens；5) Unifying principle（部分候选也用）

### C1（主推候选）ToolShift：**工具演化与噪声下的鲁棒工具调用 agent**

* 核心问题定义：当工具 schema（函数名/参数名/描述/可用性）发生真实演化或受噪声影响时，现有 tool agents 会系统性崩溃；如何让 agent 对 **schema 表面形式不敏感** 且能 **可靠恢复**？
* 关键缺口（gap）

  * 现有基准开始测噪声/不可用（FAIL-TALMS、AgentNoiseBench），但缺少“**可控、可分解、可复现的 schema 演化谱系**”与对应训练原则。([ACL Anthology][35])
  * 已有方法要么“适配 schema 给模型”（PA-Tool），要么做 robust function calling 但缺少机制解释与统一视角。([arXiv][40])
* best-paper 支点

  * **Unifying principle**：把 tool use 形式化为“语义意图 → 可执行合约”的映射；schema 变换是 nuisance group；做 group-robust learning。
  * **New eval lens**：提出 “**Robustness–Compute 曲线**”（不同噪声/演化强度下的成功率与恢复效率）。
  * **Mechanism**：证明模型从“命名捷径”转为“语义对齐表示”（表征一致性/因果干预）。
* Timing：2025–2026 agent 基准爆发（BFCL/τ-bench/AgentNoiseBench/WASP 等），社区正在从能力走向部署鲁棒。([ICML][33])

### C2 过程验证器用于工具规划：**Step-wise verifier for tool planning**

* 核心问题：工具规划失败往往来自中间步骤（选错工具/错参/状态追踪错），但主流评测只看终局。
* Gap：PRM/Verifier 主要在数学推理；迁移到 tool planning 缺少“过程标签定义”和“可验证分解”。([ICLR 会议录][1])
* 支点：把 tool planning 轨迹分解成可验证断言（合约/状态转移），构建过程奖励与 test-time search 的统一理论。
* Timing：TTS/PRM 热度高，向 agent 扩展是自然下一步。([arXiv][2])

### C3 Off-policy RLVR for agents：**把“现实异步/缓存”变成理论与算法优势**

* 核心问题：现实训练系统天然 off-policy（policy lag、异步采样）；如何在 agent 任务上把 off-policy 从 bug 变成 feature？
* Gap：OAPL/BAPO 等刚出现，但在 tool agent（多轮、噪声、规则）上的系统研究与理论预测不足。([arXiv][28])
* 支点：提出统一的 off-policy credit assignment + recoverability 指标；给出可证伪的 scaling/效率预测。
* Timing：2026 初 off-policy RL 工作密集出现，窗口期适合做“系统化+统一理论”。([arXiv][28])

### C4 机制解释驱动的 Agent 安全：**解释并修复 prompt injection/工具劫持**

* 核心问题：web agents 易被 prompt injection；我们能否定位内部“被劫持电路”，并做可控干预来提升鲁棒？
* Gap：WASP 给了评测，但“可解释修复”仍弱；电路分析已用于 jailbreak，但未系统落到 web/tool agent。([NeurIPS][36])
* 支点：机制因果证据 + 小干预（activation editing / feature suppression）带来安全提升。
* Timing：agent 安全成为热点，评测基准出现意味着可复现。([NeurIPS][36])

### C5 动态、无污染的 Agent Benchmark 协议：**把“评测可信度”扩展到工具/文档快速演化**

* 核心问题：agent 基准被快速过拟合；工具/文档变化导致 hidden contamination。
* Gap：污染研究多在 LM benchmark；工具/文档/检索链条的“search-time contamination”仍缺统一协议。([OpenReview][53])
* 支点：可复现实验 + 统一协议；给出对现有 leaderboards 的纠偏结论。
* Timing：Leaderboard Illusion 等让社区对 measurement 更敏感。([NeurIPS][10])

### 5.1 选择前必须回答的关键问题（可见）

**Q1：最接近的 3 类工作是谁？我们不可替代的增量是什么？**

1. **鲁棒函数调用/命名过拟合类**：指出命名捷径与跨基准波动（Robust Function-Calling / Hammer 等）。我们增量：把“命名捷径”提升为 **群不变性（group invariance）** 的统一学习目标 + 可控演化谱系评测 + 机制证据。([OpenReview][12])
2. **适配 schema 给模型类**：PA-Tool 主张重命名 schema 以贴合预训练熟悉度。我们增量：相反路线——**让 agent 对 schema 演化鲁棒**，并刻画“何时适配 schema 更好/何时适配模型更好”的可证伪边界。([arXiv][40])
3. **鲁棒评测/真实噪声类**：FAIL-TALMS、AgentNoiseBench 给出噪声与不可用的评测透镜。我们增量：提出 **ToolShift（schema 演化为核心轴）**，并提供训练原则（对比一致性 + 合约验证）让模型在该轴上显著改善。([ACL Anthology][35])

**Q2：贡献类型是什么？为什么足够强？**

* 组合型贡献：**新问题定义（tool schema evolution robustness） + 新评测（ToolShift suite） + 新机制/方法（schema-contrastive consistency + contract verification） + 半理论（group-robust risk 的可检验预测）**。

**Q3：算力约束下最小闭环是什么（2–3 周）？**

* 2–3 周信号：

  * 用 BFCL/StableToolBench/τ-bench 先做 **schema perturbation stress test**（rename/reorder/drop doc）形成 ToolShift-lite；
  * 对 7B 模型做 LoRA-SFT（+/- 我们的 consistency loss）并看鲁棒性曲线是否抬升。([ICML][33])

**Q4：best paper 级叙事一句话？**

* “**Agent 失败的根因不是不会用工具，而是把工具 schema 的表面形式当成语义；我们给出可控的工具演化评测，并用群不变学习与可执行合约把 agent 变得可部署、可解释、可验证。**”

### 5.2 一句话主张（thesis claim）

**通过对工具 schema 的可控群变换进行对比一致性训练，并用可执行合约做验证与约束解码，我们能在工具演化与真实噪声下显著提升 tool-using agents 的成功率，同时用机制分析证明模型从“命名捷径”转向“语义对齐的工具表示”。**

---

### 5.3 Novelty checklist（至少 5 条，含“最接近 3 类工作”逐项对比）

> 下面每条都对应“不可替代”的贡献点（避免被说成工程堆砌）

1. **新评测轴：Schema Evolution Spectrum（SES）**

   * 不是只注入噪声（AgentNoiseBench）或只测不可用（FAIL-TALMS），而是系统化覆盖：rename / reorder / paraphrase / deprecate / distractor injection / partial docs。([arXiv][34])
2. **新训练目标：Schema-Contrastive Consistency（SCC）**

   * 相比 Hammer 类“增强数据+训模型”，我们显式加入“跨 schema 变换的表示一致性”约束，目标是学到 **nuisance-invariant state**。([arXiv][88])
3. **合约验证 + 约束解码作为可验证组件（不是 TTS 堆采样）**

   * 区别于纯 test-time 重采样：我们把 tool call 视为满足 JSON schema + 额外约束的“程序”，提供 deterministic verifier。
4. **边界结论：何时“适配 schema”优于“适配模型”**

   * 与 PA-Tool 路线形成可证伪对照：在低资源/小模型/高演化强度时，我们预测“适配模型（SCC）”更稳；在极端熟悉度差异时，适配 schema 可能更划算。([arXiv][40])
5. **机制证据链（beyond correlation）**

   * 用 probing/因果干预证明：训练后工具选择对 name token 的依赖下降、对 description semantics 的依赖上升；并与性能曲线一致。
6. **可复现的“鲁棒性–预算曲线”报告标准**

   * 不是给单点 accuracy，而是给 “SES 强度 × 预算(采样/验证) × 成功率/违约率”三维面，利于社区复用。

---

### 5.4 核心洞察（为什么不是工程堆砌）

* 现有 agent/tool learning 隐含假设：**schema 是语义本身**（函数名/参数名足够稳定）。
* 现实：schema 是 **接口外观（nuisance）**，语义应来自“描述+约束+可观测行为”。
* 我们用一个可检验原则统一：

  * 把 schema 变换看成群 (G) 上的数据增强；
  * 训练目标逼近 **group-robust risk**：(\min_\theta \mathbb{E}*{g\sim G}, \ell(f*\theta(x, g(S)), g(y)))；
  * 若模型学到真正语义表示，则对 (g) 不敏感；否则鲁棒性曲线会塌。
* 这给出 **可证伪预测**：一致性训练应主要提升“高 SES 强度”区域，而对原始（无扰动）性能影响可控（甚至略降但可换取大幅稳健增益）。

### 5.5 方法清晰定义（伪代码级别 + 损失/超参/复杂度）

#### 5.5.1 任务定义

* 输入：用户请求 (x)，工具集合 schema (S={(t_i, desc_i, args_i, constraints_i)}_{i=1}^m)
* 输出：结构化调用 (y)（可为 JSON/AST），包含 tool 选择与参数。

#### 5.5.2 Schema 变换族（SES：Schema Evolution Spectrum）

* (g\in G) 包含可组合操作：

  * **Rename**：tool/arg 名同义改写或随机字符串（保持映射表）。
  * **Reorder**：参数顺序/字段顺序打乱（语义不变）。
  * **Paraphrase**：description 改写（LLM 生成 + 过滤）。
  * **Drop/Mask**：遮蔽部分字段/文档（模拟缺文档）。
  * **Distractor injection**：加入相似无关工具（hard negatives）。
  * **Deprecate/alias**：工具版本化、别名映射（模拟演化）。

#### 5.5.3 训练算法：SCC + Contract Verification Fine-tuning

**核心思想**：同一个语义任务，在不同 (g(S)) 下应产出等价调用 (g(y))，并且“决策表征”应一致。

**伪代码**

* 给定训练集 (D={(x,S,y)})
* 对每个样本：

  1. 采样 (K) 个变换 (g_1..g_K\sim G)
  2. 构造 ((x,S_k=g_k(S), y_k=g_k(y)))
  3. 前向得到 logits 与一个“决策表征” (h_k)（例如：生成 tool call 前最后一层的 hidden state，或 tool-choice head 的 pooled embedding）
  4. 计算损失：

     * **结构化监督**：(L_{\text{CE}}=\sum_k -\log p_\theta(y_k|x,S_k))
     * **一致性**（两种可选）

       * KL 一致性：(L_{\text{KL}}=\sum_{k<l} \mathrm{KL}(p_\theta(\text{tool}|x,S_k),|,p_\theta(\text{tool}|x,S_l)))
       * 表征对比：InfoNCE，让同一样本不同变换为正对，不同样本为负对
     * **合约违约惩罚**：若生成 ( \hat y_k) 不满足 contract（JSON schema/type/range/dependency），加 (L_{\text{con}}=\mathbb{1}[\neg \text{valid}(\hat y_k)])
  5. 总损失：
     [
     L = L_{\text{CE}} + \lambda_{\text{cons}} L_{\text{cons}} + \lambda_{\text{con}} L_{\text{con}}
     ]

**关键超参（建议初值）**

* (K=2)（先跑通），后续 (K=4) 做强鲁棒
* (\lambda_{\text{cons}} \in {0.1, 0.3, 1.0})
* (\lambda_{\text{con}} \in {0, 0.1})（先不加或弱加，避免训练早期不稳定）
* context length：4k–8k（取决于工具文档长度）
* LoRA rank：32/64（baseline），保持可复现

**复杂度估计**

* 训练成本约为 **×K**（同样本多视图）。K=2 时仍可控。
* 合约检查是 O(|y|) 的解析与规则验证，CPU 侧即可。

#### 5.5.4 推理流程（部署向）

1. 给定 (x,S)，生成 N 个候选 tool call（N=1/4/8，预算可控）
2. 对每个候选做 contract check（语法+类型+约束）
3. 若全部失败：触发 **repair prompt**（只修 JSON/参数，不重写计划）
4. 输出通过验证的最高分候选（或执行后再验 postcondition）

### 5.6 至少 3 个强基线（含最难打败 baseline）

> 我认为**最难打败**的是：**Hammer/robust function-calling**（已针对命名捷径做过增强训练）+ BFCL 强评测链路。([arXiv][88])

基线清单：

1. **Vanilla SFT**：在 ToolBench/ToolACE 风格数据上做指令微调（不加 SES、不加一致性）。([GitHub][32])
2. **SFT + SES augmentation only**：只做 schema 变换增强，不加一致性损失（验证 SCC 的增量）。
3. **Hammer / Robust Function-Calling**：已有针对命名约定过拟合的鲁棒训练。([arXiv][88])
4. **PA-Tool（schema adaptation）**：训练-free 侧重把 schema 改成模型熟悉形式。([arXiv][40])
5. **Template-based function calling**（EMNLP 2025）：用模板分解 function calling 步骤（可作为强prompt/finetune baseline）。([ACL Anthology][89])

### 5.7 Minimum viable experiments（至少 6 个必须做 + 预期现象，可证伪）

**E1：ToolShift-lite 基准构建与“脆弱性曲线”**

* 在 BFCL/StableToolBench/τ-bench 上做 SES 扰动（rename/reorder/paraphrase/distractors）。([ICML][33])
* 预期：现有模型在 rename/distractor 上显著掉点；且掉点与工具数量/相似度相关（可量化）。

**E2：SFT vs SFT+SES vs SCC（核心）**

* 预期：

  * SFT+SES 提升鲁棒但有限；
  * 加 SCC 后，高强度 SES 区域显著抬升；
  * 原始无扰动性能最多小幅波动（如果大幅下降则否定“语义不变性”假设，需改法）。

**E3：与 Hammer/PA-Tool 的对照边界**

* 预期：

  * 在轻度 rename 下，PA-Tool 可能强；
  * 在“持续演化 + distractors + partial docs”的复合扰动下，SCC 更稳；
  * 若结果相反，则说明“适配 schema”可能是主路线（推翻我们的核心主张）。

**E4：合约验证组件的作用（推理时约束 vs 训练时惩罚）**

* 比较：无约束解码 / 仅推理约束 / 训练时加 (L_{\text{con}}) / 两者都加
* 预期：推理约束提升 syntactic validity；训练时惩罚减少“靠 repair”次数并提升 first-pass。

**E5：机制验证 1——命名捷径依赖度下降**

* 指标：对 tool-name tokens 做遮蔽/随机化的敏感性；或计算“名称相似度 vs 误选率”的斜率。
* 预期：SCC 后斜率变小（更不依赖名字），且与鲁棒提升相关。

**E6：机制验证 2——表征一致性与泛化相关**

* 统计：不同 (g) 下表征 (h_k) 的 cosine 相似度 vs 成功率。
* 预期：更高的一致性对应更高鲁棒成功率；若无相关，说明一致性损失选点/定义错误。

**E7（加分）：跨域迁移**

* 在 FAIL-TALMS（工具不可用/欠指定）与 AgentNoiseBench 上测试是否“副作用鲁棒”。([ACL Anthology][35])
* 预期：SCC 主要提升 schema 类问题；对“不可用工具”需要额外策略（失败也有信息量）。

---

### 5.8 预期会失败的设置（至少 2 个，并说明失败价值）

1. **极端欠指定意图**（用户没说清楚需要哪个工具/参数缺关键字段）：

   * 预期失败原因：不确定性来自任务信息不足，不是 schema nuisance。
   * 价值：逼出“uncertainty-aware clarification”的必要性（可成为后续工作）。([ACL Anthology][35])
2. **强对抗 prompt injection（web agent 安全）**：

   * 预期失败原因：这是攻击者改变“目标函数”，不只是接口演化。
   * 价值：清晰划分鲁棒性边界，避免被 reviewer 说“你这不安全”。([NeurIPS][36])

---

### 5.9 风险与 Plan B / Plan C

**主要风险**

* R1：SCC 一致性损失选错表征点，导致“学不动/过约束”。
* R2：SES 变换若不够真实，会被质疑“你扰动不代表真实演化”。
* R3：提升主要来自“更多数据/更多合成”，被认为不够机制化。

**Plan B（保持方向但换支点）**

* 把核心从“训练法”转为“评测+科学发现”：

  * ToolShift 做成社区标准（像 LongBench v2 / LiveCodeBench 的地位），并给出强审计结论与建议协议。([arXiv][13])

**Plan C（方法换壳但保留主张）**

* 走“schema adaptation vs model adaptation 的最优边界”理论：

  * 系统比较 PA-Tool 与 SCC，提出可证伪预测：什么时候应 rename schema、什么时候应训模型。([arXiv][40])

---

### 5.10 Debug roadmap（诊断树，结果不如预期怎么迭代）

1. **先定位错误类型**

* tool selection 错？argument extraction 错？type/range 违约？多轮状态错？

2. 若 **selection 错**：

* 看 distractor 注入下 confusion matrix
* 增强 hard negatives（更相似工具）
* 把一致性表征点改到“选择工具的 head/pool”

3. 若 **argument 错**：

* 引入字段级对齐：把参数抽取做成 span/slot 任务
* 增加 contract-aware decoding（类型与依赖约束）

4. 若 **训练不稳定**：

* 降低 (\lambda_{\text{cons}})，先 warmup 只做 CE，再逐步加一致性
* K 从 2 开始；不要一上来 4–8

5. 若 **无扰动性能掉太多**：

* 做双目标：clean loss 与 robust loss 分开权重
* 或用 curriculum：SES 强度从弱到强

6. 若 **鲁棒不提升**：

* 检查 SES 是否真的“语义不变”（映射表是否正确）
* 检查模型是否仍能读到“泄漏线索”（例如 rename 时保留了可猜的前缀）

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

> 目标：**12 周冲刺**（可并行），前 2–3 周完成最小闭环信号；之后扩展到机制与论文级叙事。

### 6.1 数据与基准选择（含理由）

**主基准（强复现、覆盖面、社区认可）**

* **BFCL**：函数调用评测覆盖广、AST 评估可规模化。([ICML][33])
* **StableToolBench**：稳定工具评测，避免 API 漂移导致不可复现。([arXiv][31])
* **τ-bench**：多轮+规则，贴近真实 agent。([arXiv][11])

**鲁棒性外推（用来证明“不是只会刷一个基准”）**

* **FAIL-TALMS**（欠指定/不可用）。([ACL Anthology][35])
* **AgentNoiseBench**（真实噪声注入）。([arXiv][34])

### 6.2 方法/训练细节（工程可落地）

**模型选择**

* 主实验：7B–14B 开源指令模型（便于全量日志、激活分析、LoRA 复现）。
* 评测扩展：用闭源 API 模型做 “zero-shot ToolShift” 对照（你 API 预算充足）。

**训练配置（建议）**

* LoRA + bf16
* seq len：4k/8k
* optimizer：AdamW，lr 1e-4（LoRA），warmup 3% steps
* batch：按 16×80GB 做数据并行；梯度累积到等效 global batch 256–1024（取决于 seq len）
* K=2 起步（每样本两种 schema 视图），每周逐步加复杂度

### 6.3 评测协议（主指标/鲁棒性/显著性/机制验证）

**主指标**

* Tool selection accuracy（或 top-k）
* Argument exact match / AST match（BFCL 风格）([ICML][33])
* End-to-end success rate（τ-bench/StableToolBench）([arXiv][11])
* Contract violation rate（语法/类型/约束）
* Repair rate（需要二次修复的比例）

**鲁棒性指标**

* **Robustness curve**：SES 强度 α ∈ [0,1]（扰动比例/强度） vs success
* Worst-case drop：(\min_{g\in G_\alpha}) performance
* Noise interaction：在 AgentNoiseBench 上做“SES+噪声”交叉（看是否叠加）。([arXiv][34])

**统计显著性**

* 每个设置 ≥3 random seeds
* bootstrap over tasks，报告 95% CI
* 预注册（可选）：明确 primary endpoint 是“高强度 SES 区域的 AUC”

**机制验证**

* name-token ablation / randomization sensitivity
* 表征一致性与成功率的相关（E6）
* 可选：小规模 activation patching（只对 7B 做）

### 6.4 资源估算（显存/训练时间/GPU天数上限/并行策略）

**粗估（保守）**

* 7B LoRA：

  * SFT（200k–500k 样本，K=1）：~0.5–1.5 GPU-days（16卡并行）
  * SCC（K=2）：~1–3 GPU-days
* 14B LoRA：约翻倍
* 机制分析（激活缓存+probe）：额外 1–2 GPU-days（可边训练边跑）

**上限建议**

* 总预算：**≤ 120 GPU-days**（按 16×A800 约 7–8 天满载）
* 并行策略：

  * 机器1：训练（主线）
  * 机器2：评测 + ToolShift 数据生成 + 机制探测（probe/SAE 可选）

### 6.5 复现与工程组织（配置/日志/超参搜索）

* 配置：Hydra/W&B（或自建 JSON config），每次实验固定：seed、数据版本、SES 变换版本号
* 数据版本：ToolShift 生成脚本 deterministic（给定 seed 生成同一扰动集）
* 超参搜索：只对 (\lambda_{\text{cons}})、K、SES 强度 curriculum 做小网格；其余固定
* 产物：

  * 公开：ToolShift 数据、评测脚本、合约检查器、训练脚本
  * 论文：所有曲线+置信区间+失败案例可复现

### 6.6 按周里程碑（12周）

* **Week 1**：复现 BFCL/StableToolBench/τ-bench 跑通；实现 SES 扰动生成（rename/reorder/paraphrase/distractors）
* **Week 2**：ToolShift-lite 报告：现有模型鲁棒性曲线 + 失败模式 map（selection vs args）
* **Week 3**：7B：SFT vs SFT+SES vs SCC（K=2）第一版结果（是否抬升高强度区域）✅最小闭环
* **Week 4**：加入 contract verifier + 约束解码；完成 E4
* **Week 5**：对照 Hammer/PA-Tool/模板式 baseline（至少 2 个强 baseline）
* **Week 6**：扩展到 τ-bench 多轮规则；做 policy violation 分析
* **Week 7**：机制验证（E5/E6）：敏感性/一致性相关；整理关键图
* **Week 8**：外推到 FAIL-TALMS / AgentNoiseBench（E7），明确边界与失败也有价值
* **Week 9**：14B 复现实验（验证非偶然）；做计算预算曲线（N=1/4/8）
* **Week 10**：写作冲刺：方法/实验/消融补齐；补做 reviewer 可能要求的 counterfactual baseline
* **Week 11**：整理开源与复现说明；补齐附录（SES 细节、合约库、统计检验）
* **Week 12**：打磨叙事与图表；准备投稿版本

## 7. 写作框架（标题×5、摘要骨架、贡献点、图表、related work）

### 7.1 标题候选（5个）

1. **ToolShift: Measuring and Training Schema-Robust Tool-Using Agents**
2. **Beyond Names: Schema-Contrastive Consistency for Reliable Function Calling Agents**
3. **From Schema to Semantics: Contract-Verified Tool Use under Interface Evolution**
4. **Group-Robust Tool Learning: Invariance to API Renaming, Reordering, and Missing Docs**
5. **Reliable Agents in the Wild: A Controlled Spectrum of Tool Evolution and a Principled Fix**

### 7.2 摘要骨架（按 best-paper 风格）

* 现象/问题：tool agents 在标准基准上好，但在工具演化/噪声下崩溃（给一个定量结论与例子）
* 缺口：现有评测缺少 schema-evolution 轴；现有方法缺统一原则与机制解释
* 方法：提出 ToolShift（SES 生成协议）+ SCC（对比一致性训练）+ 合约验证推理
* 结果：跨 BFCL/StableToolBench/τ-bench 显著提升鲁棒曲线 AUC；并给出边界（在哪些设置失败）
* 机制：敏感性下降、表征一致性上升、因果干预支持“从命名捷径到语义对齐”
* 影响：提供可复现协议与训练原则，推动可部署 agent

### 7.3 贡献点 bullet（建议写法）

* **C1（Benchmark）**：ToolShift/SES：可控、可复现的 schema 演化谱系与评测协议
* **C2（Method）**：Schema-Contrastive Consistency（SCC）+ contract verification 的训练/推理框架
* **C3（Mechanism）**：定量与因果证据表明模型减少命名捷径依赖，学习更语义化的工具表示
* **C4（Science）**：给出可证伪预测：鲁棒增益主要出现在高 SES 强度区域，并与一致性度量相关

### 7.4 主要图表草图（建议 4 张“必杀图”）

1. **鲁棒性曲线面板**：SES 强度 × success（SFT / SES-only / SCC / Hammer / PA-Tool）
2. **失败模式桑基图**：selection / argument / contract violation / state error 占比随 SES 强度变化
3. **机制图**：name-token ablation 敏感性 vs 成功率；训练前后对比
4. **一致性–泛化相关图**：表征一致性指标 vs ToolShift 成功率（点云+回归+CI）

### 7.5 Related Work 组织方式（避免堆砌）

* Tool benchmarks：BFCL / StableToolBench / τ-bench / FAIL-TALMS / AgentNoiseBench（按“评测维度”组织）([ICML][33])
* Robust function calling：Hammer/robust FC、schema adaptation（PA-Tool）([arXiv][88])
* Reasoning/verifier/test-time compute（说明与我们“合约验证”的关系，但强调我们不是堆TTS）([arXiv][2])
* Mechanistic/representation（解释我们如何做机制证据）([NeurIPS][5])
