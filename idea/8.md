## 1. 方向地图（含主题覆盖）

### 1.1 我先怎么筛

* **Step A｜广撒网**：我先把你给的 10 个主题空间全部扫了一遍，不只看“热门”，还看**最近 6–18 个月是否有明确的范式迁移**、是否有**高信号 benchmark / theory / mechanism**、以及是否能在 **16×80GB A800** 内做出扎实结果。
* **Step B｜可冲 best paper 评分**：我重点看 6 个维度：`Novelty potential / Mechanistic insight / Tractability / Community relevance / Benchmarkability / Moat`。
* **Step C｜系统挖 gap**：我显式用了 6 套工具：`failure modes map / assumption stress test / counterfactual baseline / unifying principle / new eval lens / complexity lens`。后面第 4 节会把“用哪一套工具找到哪个 gap”写清楚。

### 1.2 10 个主题空间：现在主线、痛点、重要性

1. **推理与验证（reasoning / verifier / self-correction）**

   * 主线：2025 的范式迁移很明显——从“更大模型”转向“**RL reasoning + test-time compute + verifier**”。DeepSeek-R1 让 RL-only reasoning 成为主线；Snell 等把“compute-optimal test-time scaling”推成 ICLR 2025 Oral；同时 ProcessBench / VerifyBench 把“verifier 到底有没有真会验”这个问题推到台前。
   * 痛点：**self-verification 常常失效**；很多 verifier 在普通 benchmark 上强，但可能只是在学“答案像不像对”，不是“每一步是否成立”。
   * 重要性：这是 2025–2026 最靠近“下一代 reasoning system”中枢的位置。([arXiv][1])

2. **可靠性与对齐（honesty / calibration / uncertainty）**

   * 主线：从“helpful/harmless”转向更细粒度的**诚实、校准、认知边界表达**；既看 token confidence，也看 semantic uncertainty、verbal confidence、decision-theoretic uncertainty。
   * 痛点：模型经常**高置信地错**；而且 verbal confidence 在攻击下会失真，诚实与能力也常被混淆。
   * 重要性：如果 verifier / judge / agent 不知道自己何时不确定，很多上层系统都站不住。([开放评论][2])

3. **表征与机制解释（mechanistic interpretability / representation）**

   * 主线：**SAE、circuit discovery、reasoning representation probing** 正在从“解释 feature”走向“解释 reasoning mechanism”。
   * 痛点：很多 mech interp 结论还停留在相关性；真正能做**因果干预、可证伪机制**的工作仍少。
   * 重要性：这是最适合做 best-paper 叙事的“为什么有效”支点。([开放评论][3])

4. **训练与优化新范式（RLHF/RLAIF alternatives, RLVR, off-policy, direct alignment）**

   * 主线：从 PPO-era 转到 **DPO / online preference / RLVR / GRPO / off-policy guidance**。
   * 痛点：一边是**reward overoptimization**，一边是**on-policy RLVR 很贵且不稳**；同时社区正在争论 RL 到底是“学了新能力”还是“压缩了 pass@k 到 pass@1”。
   * 重要性：这是 reasoning post-training 的方法学底座。([开放评论][4])

5. **数据合成与评测（synthetic data, evaluation, contamination, benchmark design）**

   * 主线：从静态 benchmark 转向**动态、抗污染、judge-aware、test-time-scaling-aware** 的评测。
   * 痛点：静态题库太容易被污染；LLM-as-a-judge 自己也不稳；很多 benchmark 测到的是“记忆/格式适配”而不是能力。
   * 重要性：这是最容易做出**新评测 lens**、形成论文护城河的位置。([开放评论][5])

6. **高效学习（parameter-efficient / modularity / compositionality）**

   * 主线：从“让模型会想”走向“**让模型少想但不掉点**”；还有 modular distillation、Self-MoE、LoRA modularity。
   * 痛点：很多提升更像系统优化；真正带机制解释的工作少。
   * 重要性：很实用，但若只做省 token / 省显存，best-paper 叙事会弱。([开放评论][6])

7. **多智能体 / 工具使用（agents, tool use, planning, search）**

   * 主线：评测从 toy tool use 转向**真实 API 生态、真实对话、MCP、browser / search agents**。
   * 痛点：真实 benchmark 上表现仍低；而且安全面暴涨。
   * 重要性：社区热度高，但实验噪声也高。([开放评论][7])

8. **检索与记忆（RAG, long-context, memory systems）**

   * 主线：两条线并行：一条是**long-context inference efficiency**，一条是**memory / sufficient-context / selective abstention**。
   * 痛点：即使给了“足够上下文”，模型仍会 hallucinate；而超长上下文成本高、memory 机制不稳定。
   * 重要性：有工业价值，但想做出“新原理”比 verifier 线稍难。([开放评论][8])

9. **结构化生成 / 程序归纳 / theorem proving**

   * 主线：**formal verification + theorem proving + verified code generation** 成为高信号场景，因为 reward 可精确。
   * 痛点：formalization bottleneck、搜索空间大、domain transfer 难。
   * 重要性：如果能把 formal signal 迁移回自然语言 reasoning，是很强的 NeurIPS 叙事。([开放评论][9])

10. **安全与红队（jailbreak / prompt injection / adversarial robustness）**

* 主线：从单轮 jailbreak 转向**agent prompt injection / judge robustness / multimodal injection**。
* 痛点：judge 和 verifier 自己也会被 attack；“一 token 就能骗 judge”这类结果很扎眼。
* 重要性：热度高，现实风险强，但如果没有机制层新意，很容易沦为 benchmark 工程。([开放评论][10])

### 1.3 我后续优先推进的 8 个方向

* 我进入第 2–4 节深入比较的，是这 8 个：

  1. 推理-verifier-test-time compute
  2. honesty / calibration / uncertainty
  3. mechanistic interpretability of reasoning
  4. post-training / RLVR / off-policy / direct alignment
  5. synthetic data / evaluation / contamination / judges
  6. agents / tool use / planning
  7. formal reasoning / verified code / theorem proving
  8. safety / jailbreak / prompt injection
* **暂不列入前 8 优先**的两个主题是：`long-context/RAG` 与 `efficiency/modularity`。不是它们不重要，而是**在你这个算力约束下，更容易做成系统 paper，而不是 best-paper 风格的“新机制 + 新评测 + 可证伪” paper**。([开放评论][8])

## 3. 深入对比（每个方向：现状 → 瓶颈 → 争议 → 可突破点）

### 3.1 方向一：推理 / verifier / test-time compute

**代表论文（6–8 篇）**

* **DeepSeek-R1**：把 RL-only reasoning post-training 推成主线。
* **Scaling LLM Test-Time Compute Optimally…**：固定 FLOPs 下，自适应 test-time compute 可胜过更大模型。
* **On the self-verification limitations…**：自我批判常塌；外部 sound verifier 才稳。
* **Reasoning Models Know When They’re Right**：hidden states 里有很强 correctness signal，可做 early exit。
* **Process Reward Models That Think (ThinkPRM)**：让 verifier 生成显式 verification trace。
* **Process Reward Model with Q-value Rankings (PQM)**：把 PRM 从独立分类改成序列 Q-value ranking。
* **RL Tango**：联合优化 generator 与 verifier。
* **On Learning Verifiers…**：第一次给 sequential reasoning verifier 的 PAC / impossibility 图景。 ([arXiv][1])

**现状**

* 主范式已经从“让模型多想一点”变成“**生成 + 搜索 + 验证**”的组合系统。
* verifier 不再只是辅助模块，而是 reasoning RL、best-of-N、search、early exit 的核心。 ([开放评论][11])

**主要瓶颈**

* 现有 PRM / judge 往往在自然数据上学到**答案正确性的表面相关**。
* process label 常来自有限人工标注或窄域数据，跨任务泛化差。
* verifier 本身也有“过拟合常见错误形态”的问题。 ([arXiv][12])

**争议 / negative results**

* “verification 比 generation easier” 这一直觉，被 self-verification failure 强烈挑战。
* hidden-state probing 说明模型**可能知道自己何时是对的**，但又没有把这种信号转成可靠文本验证器。
* 理论上也不是任何 verifier 目标都可学。 ([开放评论][13])

**可突破点**

* 关键不是再造一个更大 verifier，而是**把“过程正确”从“结果正确”里解耦出来**。
* 最有潜力的切口：**反事实、答案不变、过程可区分** 的训练与评测。

---

### 3.2 方向二：honesty / calibration / uncertainty

**代表论文（6–8 篇）**

* **Alignment for Honesty**：把 honesty 定义成“知道就答，不知道就承认不知道”。
* **Evaluating Honesty in Large Language Models**：把 accuracy 与 honesty 拆开测。
* **Improving UQ via Semantic Embeddings**：semantic uncertainty 线。
* **Rethinking Uncertainty Estimation in NLG**：质疑现有 token-level UE。
* **Efficient Semantic Uncertainty Quantification**：更便宜的 semantic UE。
* **DeLLMa**：把 classical decision theory 引进 inference-time reasoning。
* **On the Robustness of Verbal Confidence…**：confidence 表达在攻击下会失真。
* **Epistemic Integrity in LLMs**：assertiveness 与真实内部确定性错位。 ([开放评论][2])

**现状**

* 可靠性研究在从“概率校准”向“语言表达校准”“语义不确定性”“诚实行为”扩展。
* judge / verifier / agent 时代，这条线不再是 peripheral topic。 ([开放评论][2])

**瓶颈**

* calibration 往往只做 post-hoc，不进入模型内部机制。
* honesty benchmark 常把“不知道”“撒谎”“sycophancy”混在一起。
* 很少工作把 uncertainty 和 reasoning process 联起来。 ([开放评论][14])

**争议**

* verbal confidence 到底能不能作为可信代理？
* token likelihood / entropy 是否真能表达“语义层不知道”？
* honesty 是能力问题、偏好问题，还是监督目标不对？ ([开放评论][15])

**可突破点**

* 把 uncertainty 放到 **process verifier / abstention / selective reasoning** 里，而不是只做输出后处理。

---

### 3.3 方向三：mechanistic interpretability of reasoning

**代表论文（6–8 篇）**

* **Scaling and Evaluating Sparse Autoencoders**：SAE 成为机制解释标准工具之一。
* **Emergent Symbolic Mechanisms Support Abstract Reasoning**：提出 symbolic-style internal architecture。
* **Editing Reasoning Patterns in LLMs with Circuit Reshaping**：可编辑 reasoning circuits。
* **Interpreting and Controlling LLM Reasoning through IPG**：用梯度揭示/调控 reasoning causal structure。
* **CircuitTuning**：只更新关键 circuit 提升数学推理。
* **Revising and Falsifying SAE Feature Explanations**：把 feature explanation 变成可证伪对象。
* **Measuring SAE Feature Sensitivity**：检验 feature 稳定性。
* **Reasoning or Pattern Exploitation?**：用 SAE probe 看 RLHF/GRPO 改善是不是表面模式利用。 ([开放评论][3])

**现状**

* mech interp 从“解释语言建模 feature”转到“解释复杂 reasoning / RL post-training 后的表征变化”。
* 2025 一个新趋势是：**不只解释，还要控制**。 ([开放评论][16])

**瓶颈**

* 很多解释依然是 correlational。
* reasoning trace 很长，定位关键 circuit 难。
* “解释出来的 feature”是否稳定、是否可复现，是新争议点。 ([开放评论][17])

**争议**

* RL reasoning 提升到底是“真结构推理”，还是“更会 exploit 表面规律”？
* symbolic-looking 机制是否真的普适，还是只在某些 benchmark 成立？ ([开放评论][18])

**可突破点**

* 最强切口不是“再解释一个电路”，而是**拿机制解释去证明某个新训练目标为什么有效**。

---

### 3.4 方向四：post-training / RLVR / off-policy / direct alignment

**代表论文（6–8 篇）**

* **COPO**：count-based online preference optimization。
* **LUFFY**：off-policy guidance for reasoning RLVR。
* **M2PO / stale off-policy data**：问 off-policy 到底能走多远。
* **Asynchronous RLHF**：generation 与 learning 解耦。
* **χPO**：用 χ² regularization 抗 overoptimization。
* **Mitigating Reward Over-optimization in Direct Alignment**：DPO/IPO 过拟合 reward 的问题。
* **GRESO**：跳过信息量低的 rollout。
* **Towards demystifying the optimization landscape of RLVR**：分析 GRPO 成功原因。
* **Does RL Really Incentivize Reasoning…**：系统质疑 RLVR 的能力边界。 ([开放评论][4])

**现状**

* RL post-training 仍热，但社区从“会不会更强”转向“**为什么强、何时强、代价是什么**”。
* off-policy / asynchronous / selective rollout 都在围绕同一件事：**降低 RL 成本，提高信用分配效率**。 ([开放评论][19])

**瓶颈**

* 大量方法依赖较重训练预算。
* 同样的改法，在不同 base model 上差异大。
* 很多结果与 verifier 质量强耦合。 ([开放评论][19])

**争议**

* RL 是否真正带来“capability gain”，还是只是把 pass@k 压成 pass@1？
* direct alignment 是否天然容易 reward hacking？ ([开放评论][20])

**可突破点**

* 不去跟大规模 RL 直接拼卡，而是**从 verifier / data / credit assignment 的根本偏差下手**。

---

### 3.5 方向五：synthetic data / evaluation / contamination / judges

**代表论文（6–8 篇）**

* **LiveCodeBench**：动态、抗污染 code benchmark。
* **JudgeBench**：测 judge 在知识/推理/数学/代码上的客观性。
* **JETTS**：judge 对 test-time scaling 的帮助有多大。
* **DyCodeEval**：动态 code benchmark。
* **On the Fragility of Benchmark Contamination Detection…**：污染检测本身会被轻量 RL 掩盖。
* **CoDeC**：通过 in-context demonstrations 检测 contamination。
* **Detecting Benchmark Contamination Through Watermarking**。
* **VerifyBench**：专门测 reference-based reward systems。 ([开放评论][5])

**现状**

* benchmark 设计正在从“题库”升级成“**能力审计工具**”。
* judge 与 reward system 本身也成为被评测对象。 ([开放评论][21])

**瓶颈**

* 只做 benchmark 很容易被当成 resource paper。
* 如果 benchmark 没有新原理，它的寿命不长。
* contamination 与 shortcut 极难完全排除。 ([开放评论][22])

**争议**

* 静态 benchmark 还有没有意义？
* LLM-as-a-judge 能不能做可靠自动评测？
* contamination detection 自己会不会被优化掉？ ([开放评论][21])

**可突破点**

* 只要 benchmark 同时带来**新评测 lens + 新机制发现 + 对方法有直接指导意义**，它就不只是资源。

---

### 3.6 方向六：agents / tool use / planning

**代表论文（6–8 篇）**

* **τ-bench**：带用户交互和 domain rules 的 agent benchmark。
* **WildToolBench**：真实用户行为分布下的 tool use。
* **ToolComp**：多工具、多步过程监督 benchmark。
* **LiveMCPBench**：MCP 生态下的大规模 server/tool retrieval。
* **BrowseMaster**：planner-executor web browsing。
* **Benchmarking LLM Agents on Consequential Real World Tasks**：模拟真实公司环境。
* **GOAT**：human-free training framework for tool-use agents。
* **ToolPlanner**：多工具规划。 ([开放评论][7])

**现状**

* agent 评测已经在从“会不会调用工具”进化到“会不会在**复杂、动态、带约束规则**的环境里可靠完成任务”。
* benchmark 和安全问题同时升级。 ([开放评论][7])

**瓶颈**

* 基础设施重。
* 复现噪声大。
* 很多结果受 specific tool stack 影响。

**争议**

* 现有 agent benchmark 是否足够代表真实部署？
* agent improvement 是规划更强，还是仅仅 prompt-engineering 更好？

**可突破点**

* 如果把 agent 评测与**可验证过程监督**结合，才更有学术硬度。

### 3.7 方向七：formal reasoning / verified code / theorem proving

**代表论文（6–8 篇）**

## 4. 候选课题（3–5 个）

### 4.0 我用了哪些“找 gap”工具箱，以及各自找到什么 gap

1. **Failure modes map**

* 在 verifier 线，看到 3 类反复出现的失败：

  * self-verification collapse；
  * ProcessBench 上对难数学 process error 定位差；
  * judge / verifier 对答案 token、表面格式、prompt 攻击敏感。
* 这指向一个共同缺口：**当前监督没有把“过程是否成立”从“答案像不像对”里分开**。 ([开放评论][13])

2. **Assumption stress test**

* 被击穿的隐含假设有 4 个：

  * “verification 一定比 generation easier”；
  * “step label 训练出来的 PRM 就是在学 step validity”；
  * “benchmark contamination 可以后验稳定检测”；
  * “RL reasoning gain = 新能力 gain”。
* 这些假设都被近 18 个月的 negative/critical papers 挑战过。 ([开放评论][13])

3. **Counterfactual baseline**

* 我构造了最关键的反事实：**固定最终答案，改中间过程**。
* 如果 verifier 真的学到“过程正确”，它就该在“答案相同但过程不同”的配对样本上稳健区分；如果做不到，说明它在走 shortcut。
* 这条 baseline 在现有公开 benchmark 里基本缺席。 ([arXiv][12])

4. **Unifying principle**

* 我把几条线统一成一个原则：
  **robust reasoning oversight 的核心，不是更强 judge，而是把 latent state validity 从 outcome / style / confidence 的表面相关中剥离出来。**
* 这能统一 verifier、honesty、judge robustness、formal supervision 四条线。

5. **New eval lens**

* 我建议的新的评测 lens 是：
  **Answer-Matched Counterfactual Discrimination (AMCD)**：在 final answer 持平时，测 verifier 能否只靠过程分辨好坏。
* 这会让很多旧方法直接暴露 shortcut。

6. **Scaling / complexity lens**

* compute-optimal test-time scaling、off-policy RL、formal signal 都在告诉我们：
  **别再靠更大模型和更长 rollout 获胜；真正的高性价比改进来自更干净的监督和更对的归因。** ([开放评论][11])

---

### 4.1 候选 A（主推）：**Counterfactual, Answer-Invariant Process Verifier**

* **核心问题定义**
  现有 verifier 是否真正学会了“过程对错”，还是只学会了“答案/风格/置信度的相关性”？
* **关键 gap**
  公开 PRM / judge 评测几乎都没有系统控制 **final answer** 与 **过程合法性** 的解耦。
* **best-paper 支点**

  * 新机制：answer-invariant process discrimination
  * 新评测：AMCD / CRAFT benchmark
  * 新理论：在反事实环境中恢复 causal step-validity predictor
  * 机制验证：hidden-state answer leakage vs step-validity disentanglement
* **为什么现在做合适**
  2025 刚好是 verifier 爆发年，但也刚好有足够多负结果与理论工作，说明赛道成熟到可以做“反直觉纠偏”。

### 4.2 候选 B：**Latent Uncertainty for Selective Reasoning and Honest Abstention**

* **核心问题定义**
  模型内部已经知道自己何时大概率是对的/错的，为什么文本层推理仍然过度思考、过度自信？
* **关键 gap**
  hidden-state correctness probe 与 uncertainty / honesty / abstention 三条线还没被统一。
* **best-paper 支点**
  新机制 + 新 inference policy；最适合做“少算力但强 insight”。
* **时机**
  test-time compute 已成大热点，cost-aware reasoning 正在起飞。 ([开放评论][26])

### 4.3 候选 C：**Formal-to-Natural Process Supervision**

* **核心问题定义**
  能否用 formal / symbolic verifier 生成高质量自然语言过程监督，提升自然语言 reasoning verifier？
* **关键 gap**
  formal signal 很干净，但目前多被困在 theorem proving / verified code 小圈子。
* **best-paper 支点**
  新数据构造原则 + transfer story。
* **时机**
  Lean / verified code benchmark 正成熟。 ([开放评论][9])

### 4.4 候选 D：**Dynamic, Contamination-Resistant Verifier Benchmark**

* **核心问题定义**
  如何评测 verifier / judge 的真实泛化，而不是题库记忆和格式适配？
* **关键 gap**
  现有 contamination-resistant benchmark 多聚焦 final answer 或 code，不聚焦 process verifier。
* **best-paper 支点**
  新 benchmark + 新 metric + 对旧方法的系统性 failure analysis。
* **时机**
  contamination 问题在 reasoning model 时代更严重。 ([开放评论][22])

### 4.5 结论：我建议主推 **A**

* 它同时踩中 4 个 best-paper 要素：

  * **主赛道正热**
  * **gap 明确且可证伪**
  * **可以有 theory，也可以有 mechanism**
  * **算力要求主要在 verifier 和数据构造，不在超大 pretraining**

## 5. 主推 idea（完整研究蓝图）

### 5.0 先回答 4 个关键问题（按你的要求）

#### Q1. 最接近的 3 类工作是谁？我们到底多了什么不可替代的东西？

**最近的 3 类工作**

1. **标准 PRM / verifier 线**：ThinkPRM、PQM、RL Tango、ProcessBench / VerifyBench。
2. **hidden-state self-verification / efficient reasoning 线**：Reasoning Models Know When They’re Right、Training Language Models to Reason Efficiently。
3. **negative/theory 线**：On the self-verification limitations、On Learning Verifiers。 ([开放评论][27])

**我们多出来的“不可替代”东西**

* 不是再换一个 PRM loss。
* 不是再做一个 early-exit probe。
* 不是只说“self-verification 不行”。
* **我们同时做三件现有工作没闭环做到的事：**

  1. **新问题定义**：把 verifier 的目标改成 **answer-invariant process discrimination**。
  2. **新数据原理**：用 symbolic / formal state trajectory 生成**答案匹配的反事实 reasoning traces**。
  3. **新评测与机制验证**：用 **AMCD** 直接测 shortcut，并用表示分析证明 answer leakage 被压低。

#### Q2. 这个贡献是新机制 / 新理论 / 新评测 / 新问题定义中的哪一种？

* **组合型贡献**，而且是强组合：

  * **新问题定义**：verifier 应测“过程正确”，不是“结果像对”。
  * **新评测**：AMCD / CRAFT。
  * **新机制**：answer-invariant 训练目标，让 verifier 表示从 answer correctness 中解缠。
  * **新理论**：在反事实环境下，invariant predictor 恢复 causal step-validity 的条件。

#### Q3. 在算力约束下，2–3 周的最小闭环是什么？

* **第 1 周**：做出一个小型 CRAFT-math 数据集；跑标准 PRM / judge；确认在 **答案匹配反事实对** 上显著掉点。
* **第 2 周**：训一个 1.5B–3B answer-invariant verifier；看 AMCD 是否明显提升。
* **第 3 周**：把 verifier 用到 best-of-N trace selection 上，看 ProcessBench / held-out math/code 上是否转化成实用增益。

#### Q4. 这个方向的 best-paper 级叙事是什么？

* **一句话**：
  **“Reasoning verifiers fail not because verification is inherently too hard, but because current supervision conflates process validity with outcome correctness; once that shortcut is broken by counterfactual, answer-invariant supervision, reliable verification becomes learnable, interpretable, and useful.”**

---

### 5.1 一句话主张（thesis claim）

**CIVIC-PRM：用由 symbolic / formal state trajectory 生成的“答案匹配反事实 reasoning traces”训练 answer-invariant process verifier，可以在不增加预训练规模的前提下，显著提升 verifier 的跨域泛化、校准与 test-time selection 效果，并揭示现有 verifier 的核心 shortcut 是 outcome leakage。**

---

### 5.2 创新点列表（novelty checklist，含与最接近 3 类工作的逐项对比）

1. **新问题定义：answer-invariant verification**

* 现有 PRM 大多默认“过程正确”和“结果正确”高度同线。
* 我们显式要求 verifier 在 **final answer 固定** 时也能分辨 reasoning 过程优劣。

2. **新数据构造：从可验证状态轨迹到自然语言反事实 CoT**

* 不是纯 LLM 造负例。
* 我们从 exact symbolic/formal trajectories 出发，生成**可控、可校验、可反事实**的自然语言 reasoning trace。

3. **新训练目标：step supervision + matched-pair ranking + answer adversary + group-DRO**

* 相比 ThinkPRM / PQM 这类“更聪明地建模 step score”，我们直接打掉 **answer leakage**。
* 相比 hidden-state probe 工作，我们不是只做分析，而是把它变成**可训练的 verifier 目标**。
* 相比 negative/theory 工作，我们给出**实用可跑的方法闭环**。

4. **新评测：AMCD（Answer-Matched Counterfactual Discrimination）**

* 现有 ProcessBench / VerifyBench 很有价值，但还没有系统把“答案保持一致、过程改变”设成主指标。
* 我们把这个直接升格成 verifier 的第一指标。

5. **强机制解释**

* 我们会直接测 verifier 表示里对 `answer correctness` 的可预测性。
* 预期现象：标准 PRM 的内部表示更容易泄漏 outcome；CIVIC-PRM 泄漏更低，但 step-validity AUROC 更高。

6. **理论抓手**

* 在多环境反事实设置下，worst-group / invariant 风格训练可恢复更稳健的 causal process predictor；这比单纯 empirical trick 更像 NeurIPS best-paper 叙事。

---

### 5.3 核心洞察：为什么这不是工程堆砌？

* 这不是“多加一个 loss、再加点数据增强”。
* 真正的新视角是：
  **verifier 的根本问题不是 capacity 不够，而是训练分布把“答案对”当成了“过程对”的 shortcut。**
* 一旦问题定义错了，再大的 verifier 也会学歪。
* 所以我们的核心贡献不是“更强模型”，而是**更正确的监督不变量**。

---

### 5.4 方法设计

#### 5.4.1 问题定义

* 输入：问题 `x`，推理轨迹 `τ = (s1, ..., sT)`，最终答案 `a`。
* 我们关心的真实变量不是 `a 是否正确`，而是每一步的
  `z_t ∈ {0,1}`：**step t 是否保持了与正确解状态一致的合法转移**。
* verifier 的目标是预测 `p(z_t=1 | x, s_{1:t})`，并据此对整条轨迹打分。

#### 5.4.2 数据构造：CRAFT（Counterfactual Reasoning Audits for Faithful Traces）

**核心思想**

* 先从可验证环境拿到**精确状态轨迹**，再把状态转移 verbalize 成自然语言步骤。
* 然后做反事实干预，构造“**答案匹配但过程不同**”的 trace 对。

**候选任务族**

* 代数/方程化简（SymPy 可验）
* 图搜索 / shortest path / logical deduction（环境状态可验）
* 规则规划 / 算法题（模拟器可验）
* 代码题（编译/单测/partial state 可验）
* 小规模 formal / verified code 子集（tool-assisted 可验）

**反事实操作**

* `Invalid-Step`：在第 t 步做一个局部错误变换
* `Delayed-Repair`：中间错，后面修回来，最终答案仍对
* `Lucky-Answer`：过程错，但强行把 final answer 改成对
* `Answer-Swap`：过程不变，只换 final answer
* `Paraphrase`：语义不变、表述变化

#### 5.4.3 模型与损失

* verifier backbone：7B 级 open model，优先 **LoRA / QLoRA**；必要时 1.5B/3B 先跑 MVE。
* 输入格式：`[problem] + [reasoning prefix up to step t] + [candidate current step]`
* 输出：当前 step validity logit；整条轨迹分数为 step logit 的和或加权和。

**损失函数**

* `L_step`：step-level BCE
* `L_pair`：在答案匹配的正负轨迹对上做 pairwise ranking
* `L_adv`：gradient reversal，让 verifier 表示尽量**不含 final-answer correctness** 信息
* `L_gdro`：按环境分组（natural / invalid-step / delayed-repair / lucky-answer / answer-swap）做 worst-group optimization

总损失：
[
L = L_{step} + \lambda_{pair}L_{pair} + \lambda_{adv}L_{adv} + \lambda_{gdro}L_{gdro}
]

#### 5.4.4 伪代码

```text
for each task instance x:
    sigma = exact_solver.solve(x)              # symbolic/formal state trajectory
    tau_pos = verbalize(sigma)

    C = []
    for op in [Paraphrase, InvalidStep, DelayedRepair, LuckyAnswer, AnswerSwap]:
        tau_cf, step_labels, answer_label, env = intervene_and_verbalize(sigma, op)
        C.append((x, tau_cf, step_labels, answer_label, env))

    build answer-matched pairs P from C
    add (x, C, P) to training set

train verifier f_theta:
    minimize
      L_step(step_validity) +
      λ_pair * L_pair(answer-matched ranking) +
      λ_adv  * L_adv(remove answer-correctness leakage) +
      λ_gdro * L_gdro(worst-group risk)
```

#### 5.4.5 训练超参（第一版建议）

* backbone：1.5B / 3B 做 MVE；7B 做主结果
* LoRA rank：32–64
* seq len：4k
* optimizer：AdamW
* lr：`2e-4`（LoRA），`1e-5`（全量微调时）
* batch：按 step 数对齐，global batch 128–256 steps
* epochs：2–3
* `λ_pair = 0.5`, `λ_adv = 0.1`, `λ_gdro = 0.2` 起步
* calibration：held-out dev 上 temperature scaling + split conformal thresholding

#### 5.4.6 复杂度估计

* **数据生成**：主要吃 API 和 CPU / symbolic checker，不吃大训练卡时。
* **训练**：与普通 PRM 近似同阶；额外 adversary head 和 group-DRO 成本很小。
* **推理**：

  * best-of-N 打分：`O(NT)` verifier forward
  * 若做 prefix rejection，可节省后续 generator token
* **总体**：远低于重新训 reasoning policy。

---

### 5.5 至少 3 个强基线（含最难打败的）

1. **标准 PRM（最难打的训练型 baseline）**

   * 同 backbone、同数据量，只用 `L_step` 或常规 cross-entropy。

2. **PQM / ranking-style PRM**

   * 检验“仅仅换更聪明的 PRM loss”是否足够。

3. **强 reasoning evaluator / LLM-as-a-judge（最难打的 inference-only baseline）**

   * 用 32B 级 reasoning model 或强 frontier API judge 做 zero-shot / few-shot verification。
   * 这是最能体现我们“小模型 + 正确监督”优势的 baseline。

4. **隐藏状态 probe verifier**

* 检验“是不是不用训练 verifier，只做 hidden-state probe 就够了”。

5. **Outcome verifier / reference-based reward baseline**

* 检验只看 final answer 是否足够。

---

### 5.6 至少 2 个预期会失败的设置（而且失败也有价值）

1. **短、一步到位、几乎没有过程歧义的任务**

* 预期：CIVIC-PRM 增益很小，甚至因额外正则稍微掉点。
* 价值：证明方法确实是在修“process-vs-outcome 混淆”，不是万能加成。

2. **没有精确状态可验、且 verbalization 风格过窄的训练数据**

* 预期：对真实自然语言 traces 的迁移差。
* 价值：说明高质量 counterfactual supervision 的关键不是规模，而是**干净的 latent-state grounding**。

---

### 5.7 Minimum Viable Experiments（至少 6 个，带可证伪预期）

1. **E1｜Shortcut 审计：现有 verifier 在 AMCD 上是否崩**

* 设定：对标准 PRM / judge / hidden-state probe 跑普通 step accuracy 和 AMCD。
* 预期：普通指标还行，但 **AMCD 明显掉**。
* 可证伪：如果 AMCD 不掉，说明“outcome leakage”不是主要问题。

2. **E2｜核心 ablation：去掉 answer-invariance 还有效吗**

* 设定：`step-only` vs `+pair ranking` vs `+answer adversary` vs `+group-DRO`。
* 预期：AMCD 和 OOD transfer 最依赖 `pair + adversary`。
* 可证伪：若纯 step BCE 已足够，主张失效。

3. **E3｜best-of-N 选择增益**

* 设定：对同一 generator 采样 N 条轨迹，用不同 verifier 选最佳。
* 预期：CIVIC-PRM 在同样 N 下提升更大，尤其在“答案偶然对/过程错”的集合上。
* 可证伪：若 selection 不提升，说明 verifier 学到的 process signal 不能转成系统价值。

4. **E4｜校准与 selective abstention**

* 设定：比较 ECE / Brier / AURC / selective risk。
* 预期：CIVIC-PRM 更适合做 abstain。
* 可证伪：若更稳健但不更校准，就需要把 uncertainty head 纳入主方法。

5. **E5｜跨域泛化**

* 设定：train on algebra + graph + planning；test on held-out task family / code / formal subset。
* 预期：标准 PRM OOD 掉得更厉害；CIVIC-PRM 掉得更慢。
* 可证伪：若无 OOD 优势，说明方法只学了合成数据格式。

6. **E6｜机制验证：answer leakage probe**

* 设定：从 verifier hidden states 训练 probe 预测 final-answer correctness。
* 预期：标准 PRM 的 `answer-predictability` 更高；CIVIC-PRM 更低，但 step-validity 更高。
* 可证伪：若 answer leakage 没下降，mechanistic claim 站不住。

7. **E7｜反事实敏感性**

* 设定：只改 final answer token，不改过程；或只改中间关键步骤，不改答案。
* 预期：标准 PRM 更受 final answer token 影响；CIVIC-PRM 更受过程变化影响。
* 可证伪：若两者没差，核心叙事变弱。

---

### 5.8 潜在风险与备选路线（Plan B / C）

**风险 1：合成 verbalization 太模板化，迁移到真实 CoT 不够**

* **Plan B**：加入多模型 paraphrase、多风格改写、少量真实模型 CoT 对齐蒸馏。

**风险 2：step-level exact labeling 在 code / formal 上工程量过大**

* **Plan B**：先做 `math + graph/planning` 的强结论，再把 code/formal 作为附加实验。

**风险 3：answer adversary 把有用信号也删掉**

* **Plan C**：从对抗去信息改成 conditional HSIC / orthogonality regularization；或者只在 matched-pair 上用 invariance。

**风险 4：主结果只在 AMCD 上强，转不成系统指标**

* **Plan C**：把论文重心调整成
  “**新的 verifier evaluation principle + benchmark + failure analysis**”，
  方法作为 supporting result。
* 这仍可能是一篇很强的 NeurIPS benchmark+mechanism paper。

---

### 5.9 Debug roadmap（如果结果不如预期，怎么迭代）

**症状 A：AMCD 没拉开**

* 检查：

  * 反事实操作是不是太弱
  * 正负对是否真的 final-answer matched
  * step label 是否 noisy
* 处理：

  * 增强 `Delayed-Repair / Lucky-Answer` 占比
  * 提高 pair difficulty
  * 做人工 spot-check 100 对样本

**症状 B：AMCD 提升了，但 best-of-N 不涨**

* 检查：

  * 轨迹总分 aggregation 是否合理
  * 生成器采样多样性是否不足
* 处理：

  * 从 sum-step-score 改成 earliest-error-aware score
  * 提高 sampling temperature / diversify prompts

**症状 C：自然 benchmark 掉点太多**

* 检查：

  * adversary 是否过强
  * synthetic-to-real style gap
* 处理：

  * 降 `λ_adv`
  * 混入少量真实 CoT + weak labels 做 fine-tune

**症状 D：机制分析不支持“answer leakage”**

* 检查：

  * probe 容量是否过强或过弱
  * step representation 提取位置是否合适
* 处理：

  * 层扫描
  * token span 对齐
  * 做 answer-swap intervention 而不只做 probe

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

### 6.1 数据与基准选择

**训练数据**

* **CRAFT-train（自建）**

  * 由 symbolic / simulator / formal tool 产生 exact state trajectory
  * 再 verbalize 成自然语言 steps
  * 再做反事实干预
* 原因：

  * 干净
  * 可控
  * 可无限扩展
  * 最适合做 answer-matched counterfactual

**主评测**

* **ProcessBench**：process error 定位的标准 math benchmark。([arXiv][12])
* **VerifyBench / VerifyBench-Hard**：专门测 reference-based reward systems。([开放评论][28])
* **LiveCodeBench**：动态、抗污染 code benchmark。([开放评论][5])
* **CLEVER / VerifyThisBench（小规模 formal 子集）**：测 formal/verified code transfer。([开放评论][29])

**附加评测**

* 自建 **CRAFT-test / CRAFT-hard**
* 这是你论文最强的“新 lens”部分。

## 7. 写作框架（标题 / 摘要 / 图表 / related work）

### 7.1 标题候选 × 5

1. **Verifiers Need Counterfactuals: Answer-Invariant Process Verification for Reliable LLM Reasoning**
2. **Beyond Right Answers: Counterfactual Process Verifiers for Faithful Language Model Reasoning**
3. **When Correct Answers Mislead Verifiers: Answer-Invariant Oversight for LLM Reasoning**
4. **CRAFT: Counterfactual Reasoning Audits and Answer-Invariant Verifiers for Language Models**
5. **Reliable Verification is Learnable: Counterfactual and Causal Oversight for LLM Reasoning**

### 7.2 摘要骨架

* **背景句**：reasoning models increasingly rely on verifiers / judges / test-time search, but current oversight quality is poorly understood.
* **问题句**：existing verifier training and evaluation conflate process validity with outcome correctness.
* **方法句**：we introduce CRAFT, a benchmark and data-generation pipeline for answer-matched counterfactual reasoning traces, and CIVIC-PRM, an answer-invariant verifier trained with pairwise counterfactual ranking and adversarial de-leakage of final-answer information.
* **结果句**：CIVIC-PRM improves AMCD, cross-domain generalization, calibration, and best-of-N trace selection across ProcessBench, VerifyBench, LiveCodeBench, and formal subsets.
* **机制句**：representation analyses show that standard PRMs encode stronger outcome leakage, while CIVIC-PRM better isolates step validity.
* **结论句**：reliable verification is limited less by model size than by shortcut-prone supervision.

### 7.3 贡献点 bullet

* 提出 **answer-invariant process verification** 问题定义
* 提出 **CRAFT**：从 exact state trajectory 出发的反事实 reasoning 数据构造
* 提出 **CIVIC-PRM**：pairwise + adversarial + worst-group 的 verifier 训练目标
* 提出 **AMCD**：新的 verifier 核心评测指标
* 给出 **机制证据 + 理论解释**：现有 PRM 的主要 shortcut 是 outcome leakage

## 最终建议

### 我最推荐的主线

**做 A：Counterfactual, Answer-Invariant Process Verifier（CRAFT + CIVIC-PRM）**

### 原因

* 它最符合你的约束：

  * **不用拼预训练规模**
  * **16×80GB 足够**
  * **API 预算能变成高质量数据优势**
  * **既能做 theory，也能做 mechanism**
  * **有很强、很清晰、可证伪的 ablation**
* 它的 best-paper 叙事也最完整：

  * 不是“我又把 verifier 做强了”
  * 而是“**社区一直在测错 verifier 这件事**，所以方法也学歪了；一旦把问题定义纠正，可靠 verification 就真正可学了。”

如果要我给出一句最浓缩的判断：
**这条线最像 2025–2026 的“因果纠偏版 PRM/Verifier”——既踩中热点，又足够新，而且最有机会用机制解释把实验结果钉牢。**
