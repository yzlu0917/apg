## 1. 方向地图（含主题覆盖｜Step A：广撒网）

> 时间窗口：我按“过去 18 个月内（约 2024-09 ～ 2026-03）”来筛选热点与高影响力线索；并优先引用 NeurIPS/ICLR/ICML/ACL/EMNLP/OpenReview 与 arXiv 的主来源。

### 1.1 推理与验证（reasoning / verifier / self-correction）

* 主线：**过程级监督（PRM/过程验证器）+ 测试时搜索/采样**，用更好的 credit assignment 把“会想”做成“能稳定做对”。([OpenReview][1])
* 痛点：过程标注贵、自动标注噪声大；PRM 容易 **false positive** 导致 reward hacking、长链误导。([OpenReview][2])
* 重要性：能把“推理提升”从 prompting 经验转为可学习、可验证的训练与推理范式。

### 1.2 可靠性与对齐（robustness / honesty / calibration / uncertainty）

* 主线：从“让模型少胡说”走向“**知道自己不知道**（校准/置信/选择性回答）”，并把 honesty 当作可训练目标而非仅评测。([OpenReview][3])
* 痛点：评测难（微妙不诚实/迎合并非显式错误）；LLM-as-a-judge 在此类任务上不稳。([arXiv][4])
* 重要性：Agent 要做动作与调用工具时，校准与诚实是“安全阀”。

### 1.3 表征与机制解释（mechanistic interpretability / representation）

* 主线：以 **Sparse Autoencoder（SAE）** 为核心的特征分解、特征解释与可干预性；并开始强调“可证伪的解释”。([arXiv][5])
* 痛点：解释往往过泛；跨模型/跨 checkpoint 稳定性有限；评价解释质量本身是开放问题。([arXiv][6])
* 重要性：Best-paper 级叙事常需要“为什么有效”的机制证据链。

### 1.4 训练与优化新范式（RLHF/RLAIF alternatives, direct preference）

* 主线：**DPO 系列与其变体**（更稳、更简单、更少 RL 工程），以及对 DPO 失效模式的系统分析与修补。([arXiv][7])
* 痛点：likelihood over-optimization、长度偏置/啰嗦、泛化不稳；“更强的 DPO”常被快速复现。([OpenReview][8])
* 重要性：你们算力适中但实验密度高，**对齐算法**非常适配。

### 1.5 数据合成与评测（synthetic data / eval / contamination）

* 主线：评测从静态 QA 走向 **动态、可执行、可验证**（如 IR/RAG 的 uncontaminated benchmark、可追踪 grounding）。([NeurIPS][9])
* 痛点：污染（contamination）、leaderboard 过拟合、指标与真实需求错位；评测脚本本身可能有 bug。([arXiv][10])
* 重要性：Best paper 往往来自“**新评测透镜** + 揭示旧方法问题”。

### 1.6 高效学习（PEFT / modularity / low-data）

* 主线：LoRA/QLoRA + 数据/轨迹蒸馏 + 自动化 ablation；把“系统能力”拆解为可模块化训练。
* 痛点：模块之间接口不一致；多组件系统难做严谨归因。
* 重要性：与你们硬件约束契合，能做大量可复现实验。

### 1.7 多智能体/工具使用（agents / tool use / planning / search）

* 主线：从 toy env 走向 **真实交互环境**（Web/OS/Repo），并以可执行评测推动进展。OSWorld、WebArena、SWE-bench 属于标志性基准。([os-world.github.io][11])
* 痛点：成功率低且错误模式复杂（GUI grounding、长程依赖、工具噪声、指令注入）。OSWorld 报告“人类 72%+ vs 最好模型 ~12%”。([os-world.github.io][11])
* 重要性：你们偏好 agent；这是当前顶会最“想看到系统性突破”的方向之一。

### 1.8 检索与记忆（RAG / long-context / memory）

* 主线：RAG 从“能检索”走向“**多源综合、冲突消解、全局查询**”；并更重视鲁棒性与 grounding 标注。([arXiv][12])
* 痛点：检索错误会主导生成质量；模型对外部信息的信任需要被校准（situated faithfulness）。([OpenReview][13])
* 重要性：Agent 本质上是“多次 RAG + 工具调用”的序列决策。

### 1.9 结构化生成/程序归纳（code / formal methods）

* 主线：代码 Agent、可执行验证（单测/CI）、以及更“可证明/可检查”的推理链条。SWE-bench 成为核心载体。([arXiv][14])
* 痛点：数据与评测噪声（错误 patch/脆弱脚本）；真实 repo 分布漂移。([arXiv][10])
* 重要性：可验证信号强、复现实验友好，非常适合你们“可验证可复现”的目标。

### 1.10 安全与红队（jailbreak / adversarial / prompt injection）

* 主线：从 jailbreak 走向 **Agent 场景的间接提示注入（IPI）与工具/网页注入**；出现系统化基准与多层防御框架。([ACL Anthology][15])
* 痛点：攻防是动态对抗，“更强自适应攻击绕过防御”已成共识之一。([arXiv][16])
* 重要性：Agent 一旦可执行动作，安全从“文本质量问题”变为“系统风险问题”。

## 3. 深入对比（现状→瓶颈→争议→可突破点｜Step C：gap 工具箱显式使用）

### 3.0 我用的 gap 工具箱（至少 4 种）与“找到的 gap”

* **(G1) Failure modes map**：把 agent/PRM/RAG 的失败模式按“可观测症状→根因→可测指标”画成图；用于定位“缺了什么监督/接口”。
* **(G2) Assumption stress test**：逐条挑出隐含假设（工具输出可信、环境无对抗、评测脚本正确、judge 稳定…），构造破坏这些假设的设置。
* **(G3) Counterfactual baseline**：设计“反事实更强基线”（如：同任务但工具输出被最小扰动；或同策略但只替换净化器/门控器），逼出真正增益来源。
* **(G4) Unifying principle**：把“RAG 信任校准”“agent prompt injection”“过程验证器”统一为**“不可信观测下的序列决策/因果不变性”**。
* **(G5) New eval lens**：提出新评测维度（如 counterfactual sensitivity / attack-resilience 曲线 / utility–security Pareto）。

下面每个方向我会标注用到哪些策略（G1–G5）。

---

### 3.1 D1 过程验证器/PRM（推理 credit assignment）

**现状（范式）**

* PRM：对每一步打分；配合 Best-of-N/MCTS/搜索，或用于 RL fine-tuning。([OpenReview][1])
* 新趋势：把“过程奖励”定义为 **progress**（对未来正确率的提升），而非“局部像不像对”。([OpenReview][1])

**瓶颈（G1 failure map）**

* FP（false positive）累积：长链条中“错一步但给高分”会把搜索带偏。([OpenReview][2])
* Reward hacking：PRM+RL 时模型学会“讨好 PRM”。([OpenReview][20])
* 自动标注噪声：用模型自标注会把错误模式固化。

**争议/失败案例**

* PRM 增益常依赖强 base policy；弱模型可能被 PRM 误导（“评分器比执行者更强”假设失效）。
* 搜索带来的增益究竟来自“更长思考”还是“更大采样”？（需要反事实基线 G3）

**可突破点（G2/G4/G5）**

* **(Gap-1)** 把 PRM 从“推理链”推广到 **tool-use/agent 轨迹**：对 action 的过程奖励定义为“对任务成功概率的提升”。（G4）
* **(Gap-2)** 用 **counterfactual** 的方式校准 PRM：同一步在“等价观测”下应打相同分（不变性），在“关键信息翻转”下分数应变化（敏感性）。（G3/G5）

---

### 3.2 D2 真实环境 Agent 基准（Web/OS/Repo）

**现状**

* WebArena：真实 web app，自托管，可复现。([ICLR 会议录][23])
* OSWorld：真实 OS/应用任务；指出当前系统与人类差距巨大。([os-world.github.io][11])
* SWE-bench：repo 级 bugfix，单测可验证；并派生 Lite/Verified/Live 以降低成本与保持新鲜度。([arXiv][14])

**瓶颈（G1 failure map）**

* 误差多源：规划错、执行错、GUI grounding 错、工具噪声、长上下文遗忘。
* 评测脚本/数据质量：SWE-bench 排名会因评测修补而变化。([arXiv][10])

**争议/失败案例**

* “跑分提升”可能来自对 benchmark 的特殊工程（prompt/工具链）而非可泛化能力。
* 环境基准之间不一致：同一 agent 在 WebArena/OSWorld/SWE-bench 的成功因素不同。

**可突破点（G2/G5）**

* **(Gap-3)** 给 agent 引入 **统一的鲁棒性评测透镜**：不仅看 success@1，还看“在工具输出扰动/注入下的退化曲线”。（G5）
* **(Gap-4)** 引入“可解释的失败归因协议”：把失败分解为 observation failure / planning failure / action failure，并用最小反事实定位根因。（G3/G5）

---

### 3.3 D3 Agent 安全（提示注入/工具注入）

**现状**

* 攻击：网页注入、工具文档注入、工具选择劫持、日志→泄露链等。([ACL Anthology][15])
* 防御：系统化设计模式、token-level 净化、分层 guardrails、多阶段验证。([arXiv][33])
* 趋势：自适应攻击强调“攻击者后手更强”，使静态防御容易过时。([arXiv][16])

**瓶颈（G1/G2）**

* 隐含假设：模型能区分“数据 vs 指令”。现实中 LLM 天生“混淆 deputy”。（G2）
* 防御常与 utility 冲突：过滤太强就做不成任务；过滤太弱就被劫持。

**可突破点（G4/G5）**

* **(Gap-5)** 把安全目标从“分类攻击/非攻击”升级为 **决策不变性原则**：对“与任务无关/不可执行的指令”应保持 action-invariant。（G4）
* **(Gap-6)** 提出 **utility–security Pareto** 的标准评测：同等任务成功率下最小化 ASR；或同等 ASR 下最大化 success。（G5）

---

### 3.4 D4 RAG 鲁棒性与无污染评测

**现状**

* 新 benchmark 强调：grounding 标注、多源综合、全局查询、无污染/现实语料。([ACL Anthology][38])
* 新观点：模型需“situated faithfulness”——在内外知识冲突时校准信任。([OpenReview][13])

**瓶颈（G1/G2）**

* 检索错误主导：oracle context 下很好，真实检索下崩。
* 默认假设：检索内容可信、格式友好、没有对抗提示（对 agent 明显不成立）。（G2）

**可突破点（G4/G5）**

* **(Gap-7)** 将 RAG 的“信任校准”与 agent 的“注入防御”统一成：**不可信外部观测下的策略学习**。（G4）
* **(Gap-8)** 新评测：在同一 query 下采样多种检索结果（含误检/对抗/过时）做 **counterfactual robustness**。（G5）

---

### 3.5 D5 DPO 家族（对齐算法）

**现状**

* SimPO/AlphaDPO 等让 direct preference 更易用；同时出现对 DPO 失效机制（likelihood over-optimisation）的分析。([arXiv][7])

**瓶颈（G1/G2)**

* 隐含假设：偏好数据分布代表真实部署分布；在 agent 场景（工具噪声/对抗）易失效。
* “容易复现”：算法改动小、竞品多，护城河弱。

**可突破点（G4/G5）**

* **(Gap-9)** 把 DPO 的目标从“偏好”扩展为 **偏好 + 鲁棒性约束（不变性/敏感性）**，并给出可证伪预测。（G4/G5）

---

### 3.6 D6 诚实/校准/不确定性

**现状**

* 低标注 honesty alignment（EliCal）、sycophancy 新评测（SycEval）与更强 judge 辅助。([OpenReview][3])

**瓶颈（G2/G3）**

* judge 可靠性：评测者也会被 framing 影响。
* 反事实检验不足：同一事实在不同表达/检索上下文下，模型置信应可预测变化。（G3）

**可突破点（G5）**

* **(Gap-10)** 面向 agent 的“行动置信”：不仅对答案置信，还对**是否执行动作/调用工具**给可校准的风险估计。

---

### 3.7 D7 机制可解释（SAE 等）

**现状**

* SAE 特征解释走向“可证伪”；并逐步形成评测与代码生态。([NeurIPS][53])

**瓶颈（G1/G2）**

* 解释质量评估难；跨模型迁移难。
* 做到 best-paper 需要非常强的“机制→预测→干预→验证”闭环。

**可突破点（G4）**

* **(Gap-11)** 把 agent 安全/鲁棒性的问题用 mechanistic lens 描述为“某些可激活特征导致执行越权指令”，形成可干预证据链。

---

### 3.8 D8 多智能体（debate/committee）

**现状**

* Debate vs vote 的拆解与更便宜的 test-time scaling。([OpenReview][57])

**瓶颈**

* 很容易被归因成“多采样”；机制解释薄。
* 成本随 agent 数线性涨，且不保证鲁棒。

**可突破点（G3/G5）**

* **(Gap-12)** 反事实基线：固定采样次数，把“交互”替换为“独立采样”，检验 debate 的净增益。（G3）
* **(Gap-13)** 用 debate 作为“自动化对抗者/审计者”服务 agent 安全评测，而不是直接提分。（G5）

## 4. 候选课题（3–5 个最可能冲 best paper）

> 我优先挑选：**agent 相关** + **新评测透镜** + **机制/原则解释** + **在你们算力内可做严谨实验**。

### T1（主推候选）Counterfactual-Robust Tool Agents：用“不变性原则”统一注入防御、RAG 信任与 agent 决策

* 核心问题：**当工具/网页/检索内容不可信时，agent 如何学到“该忽略什么、该相信什么、该怎么行动”的可验证策略？**

* Best-paper 支点：

  * **Unifying principle**：把外部信息当作不可信观测，优化 **action invariance / sensitivity**
  * **New eval lens**：Counterfactual Tool Sensitivity（CTS）+ utility–security Pareto 曲线
  * 机制解释：不变性→抑制“可执行指令特征”的因果依赖（可做 mechanistic probe）
* Timing：真实环境基准成熟（OSWorld/WebArena/SWE-bench），安全问题爆发式增长，社区强需求。([os-world.github.io][11])

### T2 Agent 的过程奖励模型（Agent-PRM）：把 PRM 从 CoT 推广到“工具轨迹”，解决长程 credit assignment

* 核心问题：**如何给 agent 的每一步 action 赋予可泛化的“进度奖励”，避免仅靠终局成败学习？**
* gap：PRM 多在纯文本推理；agent 环境缺少统一的过程奖励定义与反事实校准。([OpenReview][1])
* 支点：progress-based process reward + 自动化评测脚本生成 dense signal；可给出 RL/MDP 形式化与可证伪预测。
* Timing：PRM 线热；agent 基准提供可执行验证信号。([OpenReview][1])

### T3 Latency/Cost-Aware Agents：在 OSWorld-Human 启发下做“成功率–成本”最优的决策策略

* 核心问题：**如何在同等成功率下最小化调用次数/延迟，或在成本预算下最大化成功率？**
* gap：多数工作只看 success@1；缺少系统性的“效率-准确”机制与训练目标。([arXiv][63])
* 支点：新指标（efficiency-weighted success）、可证伪预算曲线、与 test-time compute 统一。

### T4 评测可信性：自动发现并修复 agent benchmark 的“伪通过/脚本漏洞”（SWE-bench 方向）

* 核心问题：**leaderboard 是否在奖励“钻评测脚本空子”？如何系统纠正？**
* gap：已有工作表明脚本修补会改变排名，但缺少“可推广的 benchmark auditing 方法学”。([arXiv][10])
* 支点：新评测透镜 + 可复现审计工具链（对整个社区有公共品价值）。

## 5. 主推 idea（完整研究蓝图）：T1 Counterfactual-Robust Tool Agents

### 5.0 选择前的关键自问自答（按你要求“可见”）

**Q1：最接近的 3 类工作是谁？我们不可替代的是什么？**

1. **注入防御/安全模式**：Design Patterns、CommandSans、Securing AI Agents…([arXiv][33])

   * 我们新增：把防御从“检测/过滤工程”升级为**可优化的决策不变性目标** + **counterfactual 评测协议**（不仅 ASR，还看 CTS 与 Pareto）。
2. **RAG 信任校准**：situated faithfulness（内外知识冲突下校准信任）。([OpenReview][13])

   * 我们新增：把“信任”落到 agent 的 **action-level**，并在 Web/OS/Repo 环境给出可执行验证闭环。
3. **PRM/过程监督**：progress-based PRM、PQM/OmegaPRM…([OpenReview][1])

   * 我们新增：把“progress/credit assignment”用于**不可信观测下的策略鲁棒性**：用 counterfactual 一致性来训练/校准过程信号。

**Q2：贡献类型是什么？为什么足够强？**

* 组合：**新原则（Unifying principle）+ 新评测（New eval lens）+ 新训练目标（可证伪）**
* 强点：把安全、RAG、agent 统一到同一个“可优化/可检验”的数学对象（不变性/敏感性），能讲清“为什么有效”。

**Q3：算力约束下最小闭环（2–3 周信号）是什么？**

* 只做 WebArena（或 SWE-bench Lite）上的：

  1. 构造 counterfactual 注入/扰动；
  2. 用不变性损失做 LoRA 微调；
  3. 看 **鲁棒退化曲线是否显著变缓**，以及 CTS 是否预测真实失败。
* 不需要大模型训练，只需中模型 + 大量评测。

**Q4：best-paper 级叙事一句话？**

* **“把 agent 的外部信息（网页/工具/检索）视为不可信观测：用 counterfactual 不变性原则训练与评测，从而在不牺牲效用的情况下系统性提升鲁棒性与安全性。”**

---

### 5.1 一句话主张（thesis claim）

**通过对工具输出构造“反事实等价/反事实冲突”版本，并以“行动不变性/敏感性”为训练与评测目标，我们能让工具型 agent 在真实基准上同时获得更高鲁棒性（抗注入/抗误检）与可解释的机制证据。**

---

### 5.2 Novelty checklist（≥5 条，含与最接近 3 类工作逐项对比）

1. **从“检测攻击”到“约束决策”**：

   * 现有：过滤/分类/guardrails（易过拟合攻击样式）。([arXiv][33])
   * 我们：优化 **π(a|o)** 在 counterfactual 等价观测下保持不变（action-invariant），对关键事实翻转保持敏感（action-sensitive）。

2. **新评测维度 CTS（Counterfactual Tool Sensitivity）**：

   * 现有：ASR、success@1、静态鲁棒集。([arXiv][34])
   * 我们：逐步测量“对无关注入的 action 稳定性 / 对关键事实的响应性”，并画 utility–security Pareto。

3. **统一 RAG 信任校准与 agent 注入防御**：

   * 现有：RAG 讨论“信任冲突”；安全讨论“注入”。([OpenReview][13])
   * 我们：用同一个 counterfactual 框架同时覆盖“误检/过时/对抗注入”。

4. **可证伪的机制预测**（不是工程堆砌）：

   * 预测 A：加入 invariance loss 后，模型对注入 token 的注意力/依赖（或可分离特征激活）应下降；且下降幅度与鲁棒提升相关。
   * 预测 B：若只做过滤不做训练，CTS 提升有限；若只做训练不做门控，ASR 降低但 utility 损失更大（Pareto 劣）。

5. **反事实基线体系**（更难被一两周复制）：

   * 提供“同等推理预算/同等过滤强度/同等采样次数”的对照，保证结论可复现且归因清晰。

---

### 5.3 核心洞察（为什么不是工程堆砌）

* 注入与误检的共同点：它们都在改变观测 **o** 的“表面形式”，但不应改变最优行动 **a**（除非关键信息真的变了）。
* 因此可把安全目标写成：

  * **不变性**：对“不可执行/与任务无关”的内容扰动，策略分布不变；
  * **敏感性**：对“关键事实”扰动，策略应改变。
* 这给出一个**可以训练、可以评测、可以被反事实证伪**的原则；比“加一条系统提示/多一层过滤”更接近科学贡献。

---

### 5.4 方法定义（伪代码级别）与训练/推理细节

#### 5.4.1 定义

* 任务为部分可观测序列决策：每步得到工具输出/网页内容/检索结果 **o_t**，历史 **h_t**，产生动作 **a_t**（tool call / click / patch / answer）。
* 构造两类反事实：

  * **等价反事实** (o_t^{eq})：加入注入文本、格式扰动、无关段落、同义改写（语义不变）。
  * **冲突反事实** (o_t^{cf})：对关键事实做最小翻转（数值/实体/步骤约束改变）。

#### 5.4.2 训练目标（核心）

* 基础行为克隆/离线 RL（任选其一作为主线，建议先 BC 再小规模 RL）：
  [
  \mathcal{L}*{BC} = -\log \pi*\theta(a_t^* \mid h_t, o_t)
  ]
* **不变性损失**（等价反事实下策略分布保持一致）：
  [
  \mathcal{L}*{inv} = D*{KL}\big(\pi_\theta(\cdot|h_t,o_t)\ |\ \pi_\theta(\cdot|h_t,o_t^{eq})\big)
  ]
* **敏感性约束**（关键事实翻转应改变“原行动”的优势；用 margin）：
  [
  \mathcal{L}*{sens} = \max\big(0,\ m - (\log \pi*\theta(a_t^*|h_t,o_t) - \log \pi_\theta(a_t^*|h_t,o_t^{cf}))\big)
  ]
* 总损失：
  [
  \mathcal{L}=\mathcal{L}*{BC}+\lambda*{inv}\mathcal{L}*{inv}+\lambda*{sens}\mathcal{L}_{sens}
  ]
* 可选：加入“行动置信”校准头（对是否执行高风险动作输出 calibrated confidence），借鉴 honesty alignment 思路。([OpenReview][3])

#### 5.4.3 反事实生成器 (G)

* 用 API LLM 生成 (o^{eq}, o^{cf})（并固定模板/随机种子/版本号以保证可复现）。
* 注入样式参考已知攻击面：网页注入、工具选择注入等。([ACL Anthology][15])

#### 5.4.4 推理时门控（可选但强烈建议）

* 训练一个轻量 **Action-Gate**（或用规则）做“两人制”：

  1. policy 提议 action；
  2. gate 在“最小上下文”下判定 action 是否与用户目标一致、是否可能被注入诱导（类似“上下文最小化”的原则）。
* gate 的设计灵感来自系统化防御模式，而我们的贡献在于：gate 的效果用 CTS+Pareto 严格量化，而不是只给案例。([arXiv][33])

#### 5.4.5 关键超参（可落地起点）

* Base model：开源 7B–14B 指令模型（LoRA/QLoRA）。
* (\lambda_{inv})=0.5 起扫；(\lambda_{sens})=0.5；margin (m)=1.0（log-prob 差）。
* 每条轨迹生成：(K_{eq}=2)，(K_{cf}=1) 个反事实版本（先小后大）。
* 优先做 **step-level**（每一步 action）而不是 episode-level，便于 credit assignment 与可解释分析。

#### 5.4.6 复杂度估计

* 训练：与普通 SFT 同阶，额外是对每条样本多跑 (K_{eq}+K_{cf}) 次 forward；总体约 ×(1+K) 计算。
* 评测：需要额外跑 counterfactual suite（可并行；适合你们 16×80GB）。

---

### 5.5 至少 3 个强基线（含最难打败 baseline）

1. **ReAct/标准工具调用 agent**（同 base model、同工具集、同提示）作为主对照。
2. **纯过滤/净化防御**（例如 token-level sanitizer 思路）但**不做 invariance 训练**：对应“安全工程基线”。([arXiv][35])
3. **多采样/自一致性**（best-of-N / majority）但不做反事实训练：对应“test-time compute 基线”。
4. （最难打败）**强闭源模型 agent**（仅用于对比上界；不作为主要贡献依赖），并报告同样 CTS/Pareto。

---

### 5.6 预期会失败的 2 个设置（失败也有价值）

1. **“合法指令嵌在工具输出里”的任务**：例如网页内容本身就是操作规程；过强不变性/净化会误杀关键信息 → 直接揭示“data vs instruction 分离”的本质张力。
2. **多模态注入**（指令写在截图/图片里）：我们的文本反事实起步方案会弱 → 值得作为 limitation，并推动下一步多模态不变性。

---

### 5.7 Minimum Viable Experiments（≥6，含可证伪预期）

> 我按“先小闭环→再扩展”的顺序列；每项都给 falsifiable 现象。

1. **主结果：鲁棒退化曲线**

   * 基准：WebArena + SWE-bench Lite（先二选一）。([ICLR 会议录][23])
   * 设置：无攻击 / 注入扰动强度 α 从小到大。
   * 预期：我们方法的 success(α) 曲线显著更平缓；纯过滤或纯多采样在某段 α 会崩。

2. **新指标 CTS 与真实失败相关性**

   * 计算：对每个 state 生成 (o^{eq}, o^{cf})，看 action 分布变化。
   * 预期：CTS 低（更不该变时不变）且 sensitivity 高（该变时变）能预测更高成功与更低 ASR。

3. **反事实基线（G3）：同等过滤强度 vs invariance 训练**

   * 预期：仅过滤只能降低部分 ASR，但 utility 损失更大；加入 invariance 训练能把 Pareto 前沿整体外推。

4. **消融：(\mathcal{L}*{inv})、(\mathcal{L}*{sens})、gate、反事实生成策略**

   * 预期：去掉 (\mathcal{L}*{inv}) → 注入鲁棒性明显下降；去掉 (\mathcal{L}*{sens}) → 对关键事实反应迟钝（任务成功率下降）。

5. **跨基准泛化：WebArena → TheAgentCompany（或 OSWorld 子集）**([arXiv][28])

   * 预期：鲁棒性增益比绝对 SOTA 更稳定（说明学到原则而非过拟合基准）。

6. **机制验证：依赖度下降的可观测证据**

   * 指标：注入 span 的 attention/gradient attribution；或用 SAE/线性 probe 看“可执行指令特征”激活变化。
   * 预期：invariance 训练后，对注入 span 的依赖下降且与鲁棒性增益相关（可证伪）。

7. （加分）**对抗者后手：自适应攻击评测**（参考“Attacker Moves Second”的精神）([arXiv][16])

   * 预期：我们方法相对纯过滤更稳，因为目标不是识别某种攻击样式，而是约束决策不变性。

---

### 5.8 风险与 Plan B/C

* **风险 1：反事实生成质量不稳定**

  * Plan B：用规则 + 小模型生成 + 人工 spot-check 的混合；固定攻击模板集合（可复现）。
* **风险 2：utility–security trade-off 过强，提升鲁棒性但掉成功率**

  * Plan B：把目标改成 Pareto 优化（约束 success≥基线-ε 下最小化 ASR/CTS）。
* **风险 3：被审稿人认为“安全工程”**

  * Plan C：强化“原则化目标 + 新评测 + 可证伪机制预测”，并展示跨基准泛化与反事实基线不可替代性。

---

### 5.9 诊断树（debug roadmap）

1. **主分数不升**

   * 检查：BC 数据是否覆盖关键 action；是否 teacher 轨迹质量差；先做“只训 BC 不加反事实”对齐上限。
2. **鲁棒性不升**

   * 检查：(o^{eq}) 是否真的“语义等价”；注入是否被模型当成相关信息；调大 (\lambda_{inv})。
3. **对关键事实不敏感**

   * 检查：(o^{cf}) 的 fact-flip 是否足够影响最优 action；调大 (\lambda_{sens})/margin。
4. **ASR 下降但 utility 大掉**

   * 检查：不变性过强误杀合法信息；引入“可执行指令白名单”或让 gate 而非 sanitizer 承担部分责任。
5. **结果不稳定/方差大**

   * 固定采样预算、固定随机种子、报告置信区间；并用 CTS 解释方差来源。

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

### 6.1 数据与基准选择（含理由）

**首选最小闭环（2–3 周）**

* **WebArena**：可复现、任务是“工具型交互”的经典。([ICLR 会议录][23])
* **SWE-bench Lite**：低成本、强可验证（单测）。([SWE-bench][25])

**扩展（第 4–8 周）**

* **SWE-bench Verified / Live**：更干净 & 更新分布（抗过拟合）。([SWE-bench][26])
* **OSWorld 子集**：验证跨模态/真实 OS 交互，强化贡献外延。([os-world.github.io][11])

### 6.2 方法/算法步骤（可工程落地）

1. 采集 teacher trajectories：用闭源强模型在基准上跑（保存每步 obs、action、reward、是否成功）。
2. 反事实生成：对每步 obs 生成 (o^{eq}, o^{cf})（注入/误检/事实翻转）。
3. 训练：LoRA SFT（BC）+ invariance/sensitivity loss；小规模再做 offline RL（可选）。
4. 推理：policy 提议 action；gate 低上下文审计；必要时要求确认或再检索。
5. 评测：主任务成功率 + 注入鲁棒曲线 + CTS + Pareto。

### 6.3 评测协议（主指标+鲁棒性+显著性+机制验证）

* 主指标：success@1（每基准官方脚本）
* 鲁棒性：

  * 注入强度 α 的 success 曲线（AUC）
  * ASR（攻击成功率：偏离用户目标/泄露/越权动作）
  * CTS（你们提出的新指标）
* 统计：bootstrap CI；同任务多随机种子；报告效应量。
* 机制：attention/attribution +（可选）SAE/probe 相关性检验。

### 6.4 资源估算（在 16×80GB A800 内）

* **轨迹生成**：主要走 API（预算充足）；本地只做少量回放与评测。
* **LoRA 训练**：

  * 7B–14B：单次训练可用 4–8 张 80GB；多组 ablation 并行跑满两台机。
  * 预计 GPU-days：

    * 最小闭环（WebArena 或 SWE-bench Lite）：~30–80 GPU-days（含多组消融与反事实 suite）
    * 扩展到 3 个基准 + 机制分析：~120–250 GPU-days
* 并行策略：

  * 一台机做训练（多实验网格），一台机做大规模评测/反事实生成回放（vLLM + 多进程）。

### 6.5 复现与工程组织（强建议）

* 配置：Hydra / jsonnet；每次实验生成不可变 config hash。
* 日志：W&B/MLflow + 保存全部反事实生成 prompt、模型版本、随机种子。
* 数据版本：DVC/LFS；反事实数据集按（基准/攻击类型/强度/种子）分桶。
* 超参搜索：先粗扫 (\lambda_{inv},\lambda_{sens})，再固定后做 gate/生成器消融。

### 6.6 按周里程碑（可直接照着做）

* **Week 1**：搭 WebArena/SWE-bench Lite pipeline；跑 2 个强基线；定义注入模板与 CTS 计算脚本。
* **Week 2**：teacher 轨迹 1k–5k steps；反事实生成 V1；跑“只 BC”上限。
* **Week 3**：加入 (\mathcal{L}*{inv},\mathcal{L}*{sens})；得到第一条鲁棒退化曲线（是否值得继续的信号）。
* **Week 4**：系统消融（inv/sens/gate/生成器）；做反事实基线（同过滤强度）。
* **Week 5–6**：扩展到 SWE-bench Verified 或 OSWorld 子集；做跨基准泛化实验。
* **Week 7**：机制验证（attention/attribution；可选 SAE/probe）；形成“为什么有效”的证据。
* **Week 8**：自适应攻击评测（更强对抗者）；画 Pareto 前沿。
* **Week 9–10**：写作图表定稿 + 复现实验（重跑关键点、统计显著性）。
* **Week 11–12**：补充实验（失败案例、限制、多模态注入初探）+ 开源整理。

## 7. 写作框架（标题×5｜摘要骨架｜贡献点｜图表草图｜related work）

### 7.1 标题候选（×5）

1. **Counterfactual Robustness for Tool-Using Agents**
2. **Action Invariance as a Principle for Secure and Reliable LLM Agents**
3. **From Prompt Injection to Untrusted Observations: A Counterfactual Evaluation and Training Framework**
4. **CTS: Measuring and Optimizing Counterfactual Tool Sensitivity in Agentic Systems**
5. **Unifying RAG Trust Calibration and Agent Security via Counterfactual Decision Consistency**

### 7.2 摘要骨架（按 NeurIPS 风格）

* 背景：真实环境 agent（Web/OS/Repo）强依赖外部信息，但外部信息可能**错误、过时、对抗注入**。
* 问题：现有方法要么侧重过滤/检测、要么侧重提分，缺少统一原则与严格评测。
* 方法：提出 **counterfactual 等价/冲突** 构造；定义 **action invariance/sensitivity** 目标；给出 CTS 指标与 Pareto 协议；在多基准上训练与评测。
* 结果：在不牺牲（或较小牺牲）效用前提下显著降低 ASR、提升鲁棒曲线；CTS 预测失败；机制分析支持“依赖注入特征下降”。
* 贡献：原则化统一 + 新评测透镜 + 可复现 benchmark suite + 机制证据链。

### 7.3 贡献点 bullets（写在 introduction 的那种）

* 提出 **Counterfactual Tool Robustness** 框架：把工具输出视为不可信观测，统一注入防御与信任校准。
* 定义并系统化评测 **CTS** 与 **utility–security Pareto**，揭示传统 success@1 的盲点。
* 提出可优化训练目标（(\mathcal{L}*{inv}, \mathcal{L}*{sens})）并给出可证伪预测与反事实基线。
* 在 WebArena / SWE-bench（+可选 OSWorld）上验证并提供严格消融与统计显著性。
* 提供机制证据（依赖度/特征激活变化）解释“为什么有效”。
