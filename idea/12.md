## 1. 方向地图（含主题覆盖）

下面我先按你给的 10 个主题空间做 **Step A：广撒网**。
窗口默认取 **2024-09 到 2026-03**；我把其中 **8 个主题列为“重点候选”**，另外 **2 个主题列为“观察但不主推”**。

### 1.1 推理与验证（重点候选）

* **主线**：answer-only reward 正在被 step-level verifier / PRM / self-correction / reasoner-verifier joint training 取代；社区在追问“test-time compute 怎么花得更值”。
* **痛点**：PRM 标注贵、校准差；self-correction 常常“会改，但改错”；验证到底在什么条件下真正有收益，开始被系统性追问。
* **为什么重要**：这是当前 reasoning LM 的正中心，也是最适合做出 **新机制 + 新评测 + 可证伪实验** 的入口。 ([开放评论][1])

### 1.2 可靠性与对齐（重点候选）

* **主线**：从 verbal confidence 走向 semantic / latent uncertainty，再走向 process-level calibration。
* **痛点**：多数方法只给“最终答案是否靠谱”的分数，不能告诉系统“此时该继续、回退、还是分支搜索”。
* **为什么重要**：如果你想做 best-paper 级叙事，“模型什么时候知道自己不知道”是非常强的科学问题。 ([开放评论][2])

### 1.3 表征与机制解释（重点候选）

* **主线**：SAE 已从“找特征”进到“研究 scaling law、跨模型特征空间、推理特征、可干预性”。
* **痛点**：很多工作仍停在相关性；feature 是否真实、是否可证伪、是否能解释推理控制，仍未解决。
* **为什么重要**：这是把“为什么有效”讲清楚的最强支点之一。 ([开放评论][3])

### 1.4 训练与优化新范式（重点候选）

* **主线**：社区从“DPO 替代 RLHF”转向“offline / semi-online / online / off-policy / token-level credit assignment”更细粒度比较。
* **痛点**：reward hacking、likelihood displacement、credit assignment 粗糙、online RL 成本高。
* **为什么重要**：这里很适合出 **理论+机制**，但也最容易被“再调几个超参”稀释。 ([开放评论][4])

### 1.5 数据合成与评测（重点候选）

* **主线**：从静态 benchmark 转向 dynamic / contamination-aware / time-sliced / verifier-backed evaluation；同时 synthetic environments 开始成为 reasoning 训练场。
* **痛点**：静态榜单饱和、污染检测脆弱、RL 后污染信号甚至会被掩蔽。
* **为什么重要**：best-paper 不一定非得是模型；一个能改变整个领域评价方式的新视角也很强。 ([开放评论][5])

### 1.6 高效学习（观察，不主推）

* **主线**：参数高效、模块化、自蒸馏、自混合专家、组合泛化。
* **痛点**：很多工作最后变成“更省显存/参数”的工程题，机制洞察弱；而且 moat 往往不高。
* **为什么暂不主推**：在你给的算力下当然能做，但更难做出 best-paper 级“不可替代的科学叙事”。 ([开放评论][6])

### 1.7 多智能体 / 工具使用（重点候选）

* **主线**：benchmark 正从 function-calling 走向 realistic multi-step / asynchronous / cross-tool coordination。
* **痛点**：agent 最常见失败不是“不会调用”，而是“在不确定时不知该澄清、等待、并行还是直接做”。
* **为什么重要**：这是 reasoning 控制问题最贴近真实系统的一层。 ([开放评论][7])

### 1.8 检索与记忆（重点候选）

* **主线**：long-context 与 RAG 的边界正在重画；memory agents 开始强调 selective recall / revisitable memory / conflict handling。
* **痛点**：forward-only memory 容易覆盖信息；冲突证据与时效性证据仍是 RAG 的硬伤。
* **为什么重要**：这是“模型为何出错”从 parametric knowledge 走向 evidence management 的关键场景。 ([开放评论][8])

### 1.9 结构化生成 / 程序归纳 / 形式化推理（重点候选）

* **主线**：formal theorem proving、proof repair、thought-then-prove、agentic proof search 快速升温。
* **痛点**：搜索空间爆炸；informal thought 到 formal proof 的桥接仍脆；sample efficiency 很差。
* **为什么重要**：形式化验证提供最干净的 verifier，是做强机制解释和强反证实验的理想试验田。 ([开放评论][9])

### 1.10 安全与红队（观察，不主推）

* **主线**：自动红队、组合 jailbreak、multi-turn jailbreak、防御式 steering。
* **痛点**：评测面太易变，policy 依赖强；很多结果难脱离“新攻击/新防御”循环。
* **为什么暂不主推**：社区 relevance 极高，但更难做出“best paper 风格的统一原理”。 ([开放评论][10])

### 2.1 影响力证据摘记

* **推理/验证**：SCoRe 的搜索结果显示约 **373 citations**，Scaling Automated Process Verifiers 约 **217 citations**，ThinkPRM **65**，PQM **70**；同时 2025–2026 又迅速出现 PAG、Hard2Verify、UHeads、solution-verifier analysis 等跟进。 ([开放评论][1])
* **动态评测/污染**：LiveCodeBench 搜索结果显示约 **1163 citations**；MMLU-Pro 是 **NeurIPS 2024 Datasets & Benchmarks Spotlight**；FrontierMath 约 **165 citations**；2026 的污染脆弱性论文进一步指出 RL/GRPO 会掩蔽污染信号。 ([开放评论][5])
* **机制解释/SAE**：Scaling and evaluating SAEs 是 **ICLR 2025 Oral**，搜索结果显示约 **476 citations**；Universal Feature Spaces 约 **32**，TD-computation SAE 约 **25**，2025 下半年开始出现 reasoning-specific SAE 与 falsification work。 ([开放评论][3])
* **形式化推理**：Lean-STaR 是 **ICLR 2025 Spotlight**，搜索结果约 **63 citations**；DeepSeek-Prover-V1.5 约 **124 citations**；APOLLO 强调在 7B 级模型上用 proof repair 获得大幅提升。 ([开放评论][9])
* **训练优化**：VinePPO 搜索结果约 **79 citations**，Likelihood Displacement in DPO 约 **60 citations**；Bridging Offline and Online RL 则直接比较 offline / semi-online / online，表明社区正从“方法名”转向“训练 regime 本质”。 ([开放评论][11])
* **RAG/memory**：Inference Scaling for Long-Context RAG 约 **93 citations**，MemAgent **76**，RAG with Conflicting Evidence **40**，说明“检索/长上下文/记忆”仍在高密度迭代。 ([开放评论][12])

## 3. 深入对比（每个方向：现状 → 瓶颈 → 争议 → 可突破点）

### 3.1 方向 A：推理与验证 / self-correction / process reward

**代表论文（6）**

* **SCoRe** — 用多轮 online RL 学 self-correction；关键结论是：离线 correction traces 往往学不出真正的 test-time 自纠错。 ([开放评论][1])
* **Scaling Automated Process Verifiers for LLM Reasoning** — 把 PRM/automated process verifier 做大规模训练与系统评估，推动“过程奖励”成为主流 reasoning 技术栈。 ([开放评论][13])
* **ThinkPRM** — 让 PRM 自己“先想再判”，用 verbalized verification trace 提升 data efficiency。 ([开放评论][14])
* **PQM** — 把 PRM 重写成 Q-value ranking，而不是逐步二分类，强调 step interdependence。 ([开放评论][15])
* **PAG** — 统一 policy 与 generative verifier，并加入 selective revision。 ([开放评论][16])
* **Hard2Verify** — 面向 open-ended 数学题的 step-level verifier benchmark，目标是检验“前沿级”验证器。 ([开放评论][17])

**现状**

* answer-level reranking 正被 step-level verification 取代。
* verifier 不再只是后处理，而开始进入训练目标和 test-time control。
* community 已默认“验证”是 reasoning pipeline 的核心组件。 ([开放评论][13])

**主要瓶颈**

* PRM 标签昂贵；自动标签噪声大。
* 现有 verifier 多是“标量打分器”，校准弱。
* self-correction 容易把原本正确的轨迹改坏。 ([开放评论][18])

**争议点**

* “verification 到底什么时候 pay off？”已有系统研究开始表明 verifier gain 很依赖 solver/verifier family、任务、预算。 ([开放评论][19])
* “self-correction 是不是比简单 CoT / self-consistency 真强？”近期 benchmark 显示，复杂 self-correction 不总是比简单 CoT 更划算。 ([开放评论][20])

**我看到的可突破点**

* 现有方法把“错误”和“不确定”混成一个分数。
* 真正该做的是：**把 process error 与 process uncertainty 分开建模，再据此决定 continue / revise / branch。**

---

### 3.2 方向 B：可靠性 / 诚实性 / calibration / uncertainty

**代表论文（6）**

* **OpenEstimate** — 把“在不确定下做概率估计”单独立成 benchmark，而不是只测确定性问答。 ([开放评论][2])
* **Learn to Be Honest / Mitigate LLMs’ Overconfidence** — 把 hallucination probing 与 overconfidence 绑定起来，强调“会错且很自信”是系统性问题。 ([开放评论][21])
* **Atomic Calibration of LLMs in Long-Form Generations** — 从 response-level calibration 走向 claim/atomic fact level calibration。 ([开放评论][22])
* **Query-Level Uncertainty** — 不生成答案前就估计“这题我能不能答”，强调 internal confidence。 ([开放评论][23])
* **Reasoning with Confidence (UHeads)** — 用冻结 backbone 上的小 uncertainty head 做 step verification，成本远低于大 PRM。 ([开放评论][24])
* **Uncertainty Calibration of PRMs** — 直接指出 PRM 常把 partial trace 的成功概率估得过高，并提出 calibration 方案。 ([开放评论][18])

**现状**

* uncertainty 已从“输出层困惑度”转到 semantic / latent / hidden-state signal。
* calibration 不再只为 trustworthiness，也开始服务 inference-time scaling。 ([开放评论][24])

**主要瓶颈**

* 多数方法仍停在 final answer，而不是过程状态。
* calibration 方法常能“更会拒答”，但未必提升推理策略本身。
* verbal confidence 易被对抗扰动或表达方式影响。 ([开放评论][25])

**争议点**

* verbal confidence vs probabilistic confidence：哪个更接近真实 uncertainty？
* black-box sampling methods vs hidden-state probes：前者贵，后者可能过拟合。 ([开放评论][26])

**我看到的可突破点**

* uncertainty 该从“最终答案要不要信”升级成“当前 prefix 值不值得花更多 compute”。
* 这和 verifier/control 是一个问题，不应分开做。

---

### 3.3 方向 C：机制解释 / SAE / reasoning features

**代表论文（6）**

* **Scaling and evaluating sparse autoencoders** — ICLR 2025 Oral；给出 SAE scaling law 与 feature quality evaluation。 ([开放评论][3])
* **RouteSAE** — 通过跨层 routing 提取 multi-layer features，缓解单层 SAE 的局限。 ([开放评论][27])
* **How does CoT Think?** — 用 SAE + activation patching 研究 CoT 是否反映真实内部推理。 ([arXiv][28])
* **I Have Covered All the Bases Here** — 提出 ReasonScore，尝试自动找 reasoning moments 中的稀疏特征。 ([开放评论][29])
* **Revising and Falsifying SAE Feature Explanations** — 把 SAE 解释从“生成描述”推进到“可证伪解释”。 ([开放评论][30])
* **Sparse Autoencoders Reveal Universal Feature Spaces** — 探索不同 LLM 之间 feature universality。 ([开放评论][31])

**现状**

* SAE 从“好玩的可视化”进入“能否支持因果干预和跨模型结论”的阶段。
* reasoning features 正成为 interpretability 的新重点。 ([arXiv][28])

**主要瓶颈**

* feature 真伪和 monosemanticity 仍不稳定。
* 单条 CoT 常不足以解释真实计算；需要 distributional / resampling 视角。 ([开放评论][32])

**争议点**

* CoT 是否 faithful？
* SAE 特征是不是“解释”而不是“后验命名”？ ([arXiv][28])

**我看到的可突破点**

* 不只是“看见 reasoning feature”，而是把 feature 与 **revision / branching / abstention** 的控制决策因果挂钩。
* 一旦做到，interpretability 就不再是附属分析，而是方法核心。

---

### 3.4 方向 D：训练与优化 / DPO / RL / off-policy / credit assignment

**代表论文（6）**

* **Bridging Offline and Online RL for LLMs** — 系统比较 offline / semi-online / fully online，在 verifiable 与 non-verifiable 任务上都做对照。 ([开放评论][4])
* **VinePPO** — 用更精细的 trajectory-level credit assignment 提高 LLM RL。 ([开放评论][11])
* **Likelihood Displacement in DPO** — 揭示 DPO 中 preferred response likelihood 反而下降的现象。 ([开放评论][33])
* **On the Generalization of Preference Learning with DPO** — 从泛化理论角度理解 DPO。 ([开放评论][34])
* **LLMs Can Learn to Reason Via Off-Policy RL (OAPL)** — 直接拥抱 off-policyness，而不是勉强“伪 on-policy”。 ([arXiv][35])
* **CAPO** — 用 fine-grained credit assignment 改进 RL reasoning。 ([开放评论][36])

**现状**

* 社区正从“DPO vs PPO”转向“什么时候 online 真必要、credit 应该多细”。
* 理论和训练制度正在重新接近。 ([开放评论][4])

**主要瓶颈**

* reward hacking / overoptimization。
* token-level 或 step-level credit 仍不够因果化。
* 真正强的 online RL 代价依然大。 ([arXiv][37])

**争议点**

* online RL 是否总显著优于 offline DPO？近期工作并不支持一个简单结论。 ([开放评论][4])
* DPO 的标量偏好目标是否足够表达多步 reasoning 的价值结构？ ([开放评论][33])

**我看到的可突破点**

* 把 reward 从“局部正确性”改成“对最终可解性的反事实边际贡献”。
* 这比继续发明新的 PO loss 更像 best-paper 问题。

---

### 3.5 方向 E：数据合成 / 动态评测 / contamination

**代表论文（6）**

* **LiveCodeBench** — ICLR 2025 论文；time-segmented、contamination-free、覆盖 code generation / self-repair / execution 等。 ([开放评论][5])
* **MMLU-Pro** — NeurIPS 2024 Datasets & Benchmarks Spotlight；把 MMLU 拉回到更高难度、更低 prompt sensitivity。 ([开放评论][38])
* **FrontierMath** — 用专家原创题和自动验证构造极难数学 benchmark，并强调低污染风险。 ([arXiv][39])
* **IMProofBench** — 面向 research-level proof generation 的 private, evolving benchmark。 ([开放评论][40])
* **On the Fragility of Benchmark Contamination Detection in Reasoning Models** — 指出 RL/GRPO 可掩蔽污染检测信号。 ([开放评论][41])
* **Evading Data Contamination Detection for Language Models** — 进一步说明现有 contamination detection 有可被规避的假设漏洞。 ([开放评论][42])

**现状**

* 领域已默认：静态 benchmark 不再可信。
* 时间切片、动态生成、私有更新、formal validation 正成为新共识。 ([开放评论][5])

**主要瓶颈**

* benchmark construction 成本高。
* contamination detection 方法本身脆弱。
* 许多榜单只看 final score，不看 compute policy / verifier ROI。 ([开放评论][41])

**争议点**

* contamination 到底多大程度决定 leaderboard？
* private benchmark 与可复现性之间如何平衡？ ([开放评论][41])

**我看到的可突破点**

* 不是再做一个更难题库；而是做 **process-control aware evaluation**：评估模型何时该继续、何时该回退、何时该分支。

---

### 3.6 方向 F：agents / tool use / planning

**代表论文（6）**

* **MCP-Bench** — 面向 realistic multi-step tool use，覆盖 28 个 MCP servers、250 tools。 ([开放评论][7])
* **ToolTree** — 用 MCTS 风格做 tool planning。 ([开放评论][43])
* **OrchestrationBench** — 强调 agentic planning 与 tool orchestration。 ([开放评论][44])
* **Robotouille** — 异步规划 benchmark；同步任务还行，异步任务显著更难。 ([开放评论][45])
* **Structured Uncertainty Guided Clarification** — 把“先问清楚再做”变成结构化控制问题。 ([开放评论][46])
* **Building Math Agents with Multi-Turn Iterative Preference Optimization** — 说明 multi-turn preference optimization 对 tool-integrated reasoning 有效。 ([开放评论][47])

**现状**

* 工具调用 benchmark 正快速从“会不会调 API”转到“会不会规划/并发/澄清”。
* uncertainty 已开始被当成 action policy 的控制信号。 ([开放评论][46])

**主要瓶颈**

* benchmark 碎片化。
* 环境噪声和工具时延让结果不稳定。
* 很多 agent failure 其实是 ambiguity handling failure。 ([开放评论][45])

**争议点**

* 现有 benchmark 是不是仍太“实验室化”？
* multi-turn RL 是否真能稳定迁移到真实工具环境？ ([开放评论][48])

**我看到的可突破点**

* 把“clarify / act / wait / parallelize”统一成 uncertainty-aware control。
* 但这条线更分散，不如 reasoning/verifier 主线收敛。

---

### 3.7 方向 G：RAG / long-context / memory

**代表论文（6）**

* **Retrieval-Augmented Generation with Conflicting Evidence** — 把 conflict-aware RAG 单独拎出来。 ([开放评论][8])
* **Inference Scaling for Long-Context RAG** — 研究 long-context RAG 的 inference scaling。 ([开放评论][12])
* **Sparse RAG** — 通过 sparsity 降低解码时的检索文档负担。 ([开放评论][49])
* **LaRA** — 系统比较 long-context LLM 与 RAG。 ([开放评论][50])
* **MemAgent** — 用 memory workflow 处理长文本，强调 extrapolation。 ([开放评论][51])
* **ReMemR1** — 在 long-context agent 中加入 revisitable memory。 ([开放评论][52])

**现状**

* 社区已不再把“长上下文”和“检索”看成替代关系，而是 trade-off design space。
* memory agent 正从 append-only 走向 selective recall。 ([开放评论][50])

**主要瓶颈**

* conflicting evidence。
* forward-only 记忆覆盖。
* faithfulness detection 与 actual control 仍断裂。 ([开放评论][8])

**争议点**

* 长上下文是否终将淘汰 RAG？现有 benchmark 不支持这个简单结论。 ([开放评论][50])

**我看到的可突破点**

* 把“不知道该相信哪段证据”重写成 process uncertainty，而不是 retrieval score 问题。
* 但从 best-paper 角度，故事仍不如 verifier/control 主线聚焦。

---

### 3.8 方向 H：形式化推理 / theorem proving / structured reasoning

**代表论文（6）**

* **Lean-STaR** — 在 formal proof 每步前生成 informal thoughts，ICLR 2025 Spotlight。 ([开放评论][9])
* **DeepSeek-Prover-V1.5** — proof assistant feedback 驱动的 formal theorem proving strong baseline。 ([开放评论][53])
* **Formal Theorem Proving by Rewarding LLMs** — 用 formal verifier 直接做 RL reward。 ([开放评论][54])
* **Prover Agent** — informal reasoner + formal prover + Lean feedback + auxiliary lemmas。 ([开放评论][55])
* **APOLLO** — 低 sampling budget 的 proof repair pipeline，在 7B 级模型上很有代表性。 ([arXiv][56])
* **Steering Language Models for Theorem Proving** — 把 activation steering 引入 theorem proving。 ([开放评论][57])

**现状**

* theorem proving 已经成为 reasoning + verifier 最“干净”的闭环场景。
* community 正把 thought, repair, RL, steering 全都往 formal setting 搬。 ([开放评论][9])

**主要瓶颈**

* 需要复杂 infra。
* 搜索和 repair space 巨大。
* informal-to-formal bridge 仍脆。 ([arXiv][56])

**争议点**

* “是不是必须 long-CoT？”、“planner 和 prover 要不要拆开？” 都还在快速演化。 ([开放评论][57])

**我看到的可突破点**

* 如果主推 idea 需要一个最强机制验证场景，formal reasoning 是非常好的第二战场。
* 它尤其适合作为 **Plan B / transfer setting**。

### 4.1 候选 1（主推）：**把 process error 和 process uncertainty 分开建模，用它控制 continue / revise / branch**

* **核心问题定义**：一个 reasoning prefix 之所以不该直接继续，可能是“这一步错了”，也可能是“这一步没错，但后续仍高度不确定”。这两种情况需要完全不同的动作。
* **关键 gap**：PRM、UHeads、self-correction policy 大多输出单标量，没把“错”与“未知”分开。
* **best-paper 支点**：

  * 新机制：二元 process state（local validity + continuation uncertainty）
  * 新理论：额外 compute 的最优分配依赖这两个量，而不是单标量
  * 新评测：action regret / revision harm / compute value calibration
  * 可做强机制解释：用 SAE 找到 error feature vs uncertainty feature
* **为什么现在做合适**：PRM/self-correction 已经热到足够多 work 可以对比，但 Hard2Verify、PRM calibration、solution verifier analysis 又刚好暴露出“标量 verifier 不够”的窗口。 ([开放评论][17])

### 4.2 候选 2：**用“反事实边际贡献”替代“局部正确性”做 RLVR credit assignment**

* **核心问题定义**：一个 reasoning step 可能局部正确但对最终解没贡献，也可能看似冒险却打开了正确分支。
* **关键 gap**：PQM/VinePPO/CAPO 虽然在细化 credit assignment，但大多仍没直接建模“这一步对 solvability 的反事实价值”。
* **best-paper 支点**：

  * 新原理：credit 应该基于 marginal solvability improvement
  * 可给理论：与 process Q-value / influence / Shapley 风格联系
  * 实验可证伪：local-correctness reward 与 counterfactual-value reward 在高 branching 任务上应系统分化
* **为什么现在做合适**：online vs offline RL 之争正在转向“credit 到底怎么给”，这个 timing 很好。 ([开放评论][15])

### 4.3 候选 3：**用 mechanistic signals 预测“哪类 base model / prefix 会从 RLVR 与 self-correction 中获益”**

* **核心问题定义**：为什么有些 base model 一经 RLVR 就爆发，有些却很快 plateau 或 collapse？
* **关键 gap**：现有研究多是 post-hoc 经验判断，缺少训练前可观测的机制指标。
* **best-paper 支点**：

  * 新机制解释：latent soundness / uncertainty features 预测可训练性
  * 新实验：训练前 probe 能否预测训练后 gain
  * 强 falsifiable claim：如果机制成立，feature-level intervention 应能改变 revision/branch propensity
* **为什么现在做合适**：SAE 研究刚好成熟到能做 reasoning features，而 RLVR/self-training failure 也已经被明确报告。 ([arXiv][28])

### 4.4 候选 4：**建立“process-control aware”的动态 reasoning benchmark**

* **核心问题定义**：现有 benchmark 大多只评 final answer，不评“何时继续、何时回退、何时花 compute”。
* **关键 gap**：leaderboard 已经进入 contamination 与 policy gaming 时代，静态 final-score benchmark 不足以区分真正进步。
* **best-paper 支点**：

  * 新评测问题定义
  * 动态、污染鲁棒、带 oracle-action label
  * 可导出强发现：一些高分模型 action policy 可能很差
* **为什么现在做合适**：LiveCodeBench/MMLU-Pro/FrontierMath/污染脆弱性工作已经把 benchmark 更新到新阶段。 ([开放评论][5])

## 5. 主推 idea（完整研究蓝图）

### 5.1 选择前，我先回答 4 个关键问题

#### Q1. 最接近的 3 类工作是谁？我们多了什么“不可替代”的东西？

1. **PRM / verifier / uncertainty-head 类**

   * 代表：ThinkPRM、PQM、UHeads、PRM Calibration。
   * 它们多输出 **单标量 step score**；我们输出 **分解后的 process state**：

     * `P(step invalid | prefix)`
     * `P(eventual success | prefix)` 的 **分布**，而不只是均值。
   * 不可替代点：只有分解后才能自然决定 **revise** 和 **branch** 这两个不同动作。 ([开放评论][14])

2. **self-correction / online RL 类**

   * 代表：SCoRe、PAG。
   * 它们学“会不会改”，但没有显式可校准的 process-state model。
   * 不可替代点：我们不是学一个 heuristic policy，而是给出 **可解释、可校准、可测 regret** 的控制变量。 ([开放评论][1])

3. **mechanistic interpretability of reasoning 类**

   * 代表：How does CoT Think?、I Have Covered All the Bases Here、Falsifying SAE Explanations。
   * 它们解释 reasoning feature，但很少把 feature 和 **compute-control decision** 绑定。
   * 不可替代点：我们把机制分析变成方法核心，而不是论文附录。 ([arXiv][28])

#### Q2. 这是哪种贡献？

* **组合型贡献**：**新机制 + 新理论 + 新评测**。
* 新机制：error–uncertainty decomposition。
* 新理论：单标量 verifier 不足以最优分配额外 compute。
* 新评测：action regret / revision harm / compute value calibration。

#### Q3. 在我们的算力约束下，最小闭环是什么？

* **2–3 周 go/no-go 最小闭环**：

  1. 搭 synthetic verifiable reasoning 环境。
  2. 训练一个冻结 7B backbone 上的 dual-head controller。
  3. 在 synthetic + 小规模数学集上验证：

     * 是否比 PRM-only / UQ-only 更好预测 **oracle action**；
     * 是否在等 token 预算下提高 accuracy；
     * 是否降低 revision harm。
* 这一步如果没信号，就不值得继续堆 full-scale study。

#### Q4. 这个方向的 best-paper 级叙事是什么？

* **一句话叙事**：
  **“Reasoning models fail for two different reasons—being wrong and being unsure. Separating them turns test-time compute from brute force into a calibrated control problem.”**

---

### 5.2 一句话主张（thesis claim）

> **把 process-level 的“局部错误概率”和“后续成功不确定性”分开建模，并用它来决定 continue / revise / branch，可以在相同 test-time 预算下同时提升 reasoning accuracy、calibration 和 compute efficiency；而且这一收益对应可观测、可干预的内部机制。**

---

### 5.3 创新点列表（novelty checklist）

#### 与最接近的 3 类工作逐项对比

1. **相对 PRM / ThinkPRM / UHeads**

   * 不是一个 step score。
   * 是 **二元过程状态**：`local-invalidity` + `continuation-uncertainty`。
   * 目标不是“打分更准”，而是 **动作选择更低 regret**。

2. **相对 SCoRe / PAG**

   * 不是隐式学习“改不改”，而是显式建模 **为什么改 / 为什么分支**。
   * 可以单独评估 calibration、action regret、revision harm。

3. **相对 SAE reasoning interpretability**

   * 不只解释 feature。
   * 要求 feature 对 **控制决策** 具有因果作用，并能用 intervention 验证。

#### 额外 5 条创新

4. **新监督构造**：用 prefix rollouts 得到 `Beta(success | prefix)`，把 continuation uncertainty 变成可学目标。
5. **新控制器**：三动作 policy —— `continue / revise / branch`，不是 yes/no revise。
6. **新评测透镜**：

   * `Action Regret@Budget`
   * `Revision Harm Rate`
   * `Compute Value Calibration`
7. **新理论命题**：存在简单 reasoning POMDP，使得任何基于单标量 verifier score 的 policy 都有常数量级 regret，而双统计量 policy 可达零 regret。
8. **新机制验证**：预测 error 的 sparse features 和预测 uncertainty 的 sparse features 应分离，并对应不同干预效果。

---

### 5.4 核心洞察：为什么这不是工程堆砌？

* 当前很多 work 默认：

  * **低分 = 错了 = 该重做**。
* 但真正的系统里至少有两类 prefix：

  1. **错的 prefix**：该回退。
  2. **不明显错，但后续高度不确定的 prefix**：该分支或加 compute。
* 如果把两者压成一个标量：

  * 你要么过度 revise；
  * 要么过度 sample；
  * 要么错过最该花 compute 的地方。
* 所以真正的新视角不是“更强 verifier”，而是 **把 verifier 从评分器升级成过程状态估计器**。

---

### 5.5 方法定义（伪代码级）

我建议方法名先用工作名：**TriVer**（Triaged Verification by Error–Uncertainty Decomposition）。

#### 5.5.1 问题形式化

* 输入：问题 (x)
* 推理轨迹：(z_{1:T})
* 前缀：(p_t=(x,z_{1:t}))
* 动作：(a_t \in {\text{continue}, \text{revise}, \text{branch}_K})

我们要估计两个量：

1. **局部错误概率**
   [
   q_t = P(\text{step } t \text{ invalid} \mid p_t)
   ]

2. **前缀后续成功分布**
   [
   S_t \sim \text{Beta}(\alpha_t,\beta_t) \approx P(\text{eventual success} \mid p_t)
   ]
   其均值 (\mu_t=\alpha_t/(\alpha_t+\beta_t))，方差 (\nu_t) 表示 continuation uncertainty。

#### 5.5.2 监督信号构造

* **local validity label** (\ell_t)：

  * synthetic/formal：由确定性 verifier 给出；
  * math：由符号变换 checker / equation equivalence / weak executor 给出。

* **continuation target**：

  * 从 prefix (p_t) 采样 (M) 个 completion；
  * 用 final verifier 判断成功数 (c_t)；
  * 目标分布设为：
    [
    \text{Beta}^*_t = \text{Beta}(1+c_t,\ 1+M-c_t)
    ]

#### 5.5.3 训练目标

* Backbone 冻结或只做轻量 LoRA。
* 在若干层 hidden states 上接双头：

[
L = L_{\text{valid}} + \lambda_\mu L_{\mu} + \lambda_\nu L_{\nu} + \lambda_{\text{rank}}L_{\text{rank}} + \lambda_{\text{cal}}L_{\text{cal}}
]

其中：

* (L_{\text{valid}} = \text{BCE}(q_t,\ell_t))
* (L_\mu = (\mu_t-\mu_t^*)^2)
* (L_\nu = (\log \nu_t-\log \nu_t^*)^2)
* (L_{\text{rank}})：让高 continuation value prefix 排在前面
* (L_{\text{cal}})：Brier / differentiable calibration surrogate

#### 5.5.4 推理控制规则

* 若 (q_t > \tau_{\text{bad}})：**revise**
* 否则估计 branch 的 expected improvement：
  [
  EI_t(K)=\mathbb{E}[\max(Y_1,\dots,Y_K)]-\mu_t,\quad Y_i\sim \text{Beta}(\alpha_t,\beta_t)
  ]
* 若 (EI_t(K) > \lambda \cdot \text{cost}_t)：**branch_K**
* 否则：**continue**

```text
for each problem x:
    prefix = []
    for t in 1..T:
        step = LM.generate_next_step(x, prefix)
        h_t = hidden_state(prefix, step)

        q_t = invalidity_head(h_t)
        alpha_t, beta_t = beta_head(h_t)
        mu_t, var_t = BetaStats(alpha_t, beta_t)

        if q_t > tau_bad:
            prefix = backtrack_and_regenerate(prefix, r=1)
            continue

        EI_t = expected_improvement(alpha_t, beta_t, K)

        if EI_t > lambda_cost * step_cost:
            candidates = sample_K_completions_from(prefix + [step], K)
            prefix = select_best(candidates, calibrated_verifier)
            break
        else:
            prefix.append(step)

    return final_answer(prefix)
```

---

### 5.6 关键超参、损失、复杂度

#### 关键超参

* `M`（每个 prefix 的 rollout 数）：pilot 取 4，full 取 8
* `K`（branch sample 数）：4 / 8 / 16
* `τ_bad`：局部错误阈值
* `λ_cost`：compute cost trade-off
* `r`：backtrack 步数，先从 1 开始
* `L_head`：接 head 的层数，优先试 top-4 layers concat 或 top-1 residual

#### 训练细节

* backbone：7B-class open reasoner
* 冻结 backbone + 2-layer MLP head 为主；增强版再加 top-layer LoRA
* optimizer：AdamW
* head lr：1e-4 左右
* LoRA lr：5e-6 到 1e-5
* LoRA rank：16 或 32
* 序列长度：4k–8k
* 3 seeds 起步

#### 复杂度

* **训练标签生成**：(O(PM))，其中 (P) 是选中的 prefix 数
* **head 推理开销**：相对 LM 前向可忽略
* **整体 test-time**：

  * uniform best-of-K：(O(KT))
  * TriVer：(O(T + \rho KT'))，其中 (\rho) 是触发 branch 的 prefix 比例；核心目标是让 (\rho \ll 1)

---

### 5.7 至少 3 个强基线（含最难打败的 baseline）

1. **Calibrated PRM**（最难打败的传统 baseline）

   * ThinkPRM / PQM + post-hoc calibration
   * 强，因为已经兼顾过程打分与 calibration。 ([开放评论][14])

2. **UHeads / uncertainty-only verifier**

   * 强，因为和我们一样轻量，而且直接用 hidden states 做 verification。 ([开放评论][24])

3. **SCoRe / PAG 类 self-correction policy**

   * 强，因为直接优化 revise 行为。 ([开放评论][1])

4. **Uniform best-of-N / self-consistency + final verifier**

   * 强，因为简单但常很难打。
   * 必须做等 token budget 对比。

---

### 5.8 至少 2 个预期会失败的设置（失败也有价值）

1. **没有可靠 local verifier 的知识型开放问答**

* 预期：`q_t` 学不稳，TriVer 会退化成 uncertainty-only。
* 价值：明确方法边界——它更适合 **verifiable reasoning**，而非纯开放世界 QA。

2. **极短、极易题目**

* 预期：branch/revise 的开销超过收益。
* 价值：验证理论——当 prefix 已高置信、低不确定时，额外 compute 本就不该带来收益。

---

### 5.9 最少 6 个必须做的实验（含预期现象，且可证伪）

1. **Action prediction / regret**

* 目标：构造小规模 oracle-action set，比较 TriVer、PRM-only、UQ-only、always-revise。
* 指标：`Action Regret@Budget`。
* **预期**：TriVer regret 最低。
* **可证伪**：若 PRM-only 相同或更低，则“单标量不足”不成立。

2. **Equal-token budget accuracy curves**

* 任务：synthetic + math + formal transfer。
* 指标：准确率 vs token budget。
* **预期**：TriVer 在中等预算最优；极低预算与极高预算优势缩小。
* **可证伪**：若 uniform best-of-N 全程更优，则 adaptive control 价值不成立。

3. **Revision Harm Rate**

* 定义：prefix 原本可导向正确解，但 revise 后最终错误的比例。
* **预期**：TriVer 显著低于 SCoRe/PAG-like revise heuristics。
* **可证伪**：若无下降，则 error/uncertainty 分解没真正帮助控制。

4. **Compute Value Calibration**

* 目标：验证 (EI_t(K)) 是否预测“额外 compute 带来的真实增益”。
* **预期**：TriVer 的 predicted gain 与 empirical gain 高相关、低 calibration error。
* **可证伪**：若相关性不高，controller 的理论接口有问题。

5. **Ablation：去掉其中一半状态**

* `w/o invalidity head`
* `w/o continuation uncertainty`
* `w/o calibration`
* **预期**：

  * 去 invalidity → revise 错误率升高
  * 去 uncertainty → branch 浪费增多
* **可证伪**：若无差异，则双状态分解不必要。

6. **Cross-domain transfer**

* 训练：synthetic + math
* 测试：formal proof / structured puzzles
* **预期**：控制器收益下降但仍保留。
* **可证伪**：若完全不迁移，则机制更像 domain-specific heuristic。

7. **Mechanistic validation**

* 用 SAE / probe 找 error-feature 与 uncertainty-feature；做 activation patching / steering。
* **预期**：

  * 干预 error-feature 更多改变 revise 触发
  * 干预 uncertainty-feature 更多改变 branch 触发
* **可证伪**：若两类 feature 不可分，则“分解对应内部机制”不成立。 ([arXiv][28])

---

### 5.10 核心可证伪预测（falsifiable predictions）

* **P1**：真实“额外 compute 收益”在 **中等 (\mu_t) + 高 (\nu_t)** prefix 上最高，而不是在最低 PRM score 处最高。
* **P2**：相同 PRM score 的两个 prefix，若 (\nu_t) 不同，则最优动作经常不同。
* **P3**：TriVer 主要降低的是 **revision harm** 和 **wasted branching**，不只是提高 raw accuracy。
* **P4**：error-feature 与 uncertainty-feature 的干预效应方向不同。

---

### 5.11 潜在风险与备选路线（Plan B / C）

#### Plan B：如果自然语言 math 的 local validity 标签太噪

* 先退回 **formal theorem proving + synthetic environments**。
* 把主论文焦点改成“在 clean verifiable domains 上，单标量 verifier 不足”。
* 这样机制更强，故事更干净。

#### Plan C：如果性能增益有限，但 action modeling 很强

* 把论文转成 **新评测 + 新分析 + 轻方法**：

  * 发布 oracle-action benchmark
  * 证明 PRM/UQ/self-correction 的 action regret gap
* 这仍可能是顶会强 paper，尤其是 Datasets/Benchmarks 或 spotlight/oral 级。

---

### 5.12 如果结果不如预期，怎么迭代（debug roadmap）

* **Step 1：先看 action quality，不先看最终准确率**

  * 若 `Action Regret` 差：controller 本身没学到
  * 若 regret 好但 accuracy 差：candidate selection / branch implementation 有问题

* **Step 2：拆开两个 head**

  * `invalidity AUROC` 不行 → 标签或 step segmentation 有问题
  * `continuation ECE/Brier` 不行 → rollout target 不稳定，先加 calibration

* **Step 3：检查触发分布**

  * revise 过多 → `τ_bad` 太低或 local verifier 过严
  * branch 过多 → `λ_cost` 太低或 uncertainty 过估
  * branch 太少 → EI 估计保守、prefix rollout target 太尖锐

* **Step 4：做 counterfactual same-score analysis**

  * 若 same PRM score 下不同 (\nu) 没带来不同行为收益 → 你的“二元状态”假设可能不成立

* **Step 5：机制实验失败时**

  * 先换到更小模型做 SAE
  * 再做 resampling-based analysis
  * 最后才做 full-scale intervention

### 6.2 数据与基准选择（含理由）

#### 训练数据

1. **Synthetic verifiable reasoning environments**

* 理由：

  * local/final verifier 最干净
  * 可大量生成 prefix 与 oracle-action labels
  * 最适合验证理论
* 可参考 ReSyn / VAST 的环境化思路。 ([开放评论][58])

2. **数学 reasoning traces**

* 理由：

  * 社区最关心
  * 可用符号检查器做部分 local validity
  * 与 PRM/self-correction 文献直接对齐

3. **Formal reasoning pilot**

* 理由：

  * verifier 最强
  * 是跨域 transfer 的理想落点
  * 很适合机制分析
* 直接对齐 Lean-STaR / DeepSeek-Prover / APOLLO。 ([开放评论][9])

#### 测试基准

1. **Hard2Verify**

* 理由：step-level verification 的最好近端 benchmark。 ([开放评论][17])

2. **MMLU-Pro**

* 理由：比 MMLU 更有 reasoning 压力、prompt sensitivity 更低。 ([开放评论][38])

3. **FrontierMath / IMProofBench（选子集）**

* 理由：极难、低污染风险、适合验证泛化。 ([arXiv][39])

4. **可选：LiveCodeBench reasoning subset**

* 理由：如果想加 code/generality，一定要 contamination-aware。 ([开放评论][5])

### 6.3 模型 / 方法细节

#### Backbone

* 7B-class 开源 reasoning LM
* 主实验：冻结 backbone + dual head
* 增强版：top-layer LoRA distillation

#### Head 结构

* 输入：选定层 hidden states（top-1 或 top-4 concat）
* MLP：`d -> 1024 -> 256 -> {q, α, β}`
* 激活：GELU
* `α, β` 用 `softplus + 1`

#### 推理流程

* 生成 numbered steps
* 每步后运行 dual head
* 若 `q_t > τ_bad`：回退 1 步
* 若 `EI_t(K)` 大：触发 selective self-consistency / branch
* 否则继续 greedy / low-temp continue

#### 可选增强

* 将 TriVer 选出的高价值修正轨迹蒸馏回 policy（SFT 或 DPO 小规模）
* 若 head-only 已经显著，则这部分作为加分项，不作为主 claim

### 6.4 评测协议

#### 主指标

* 最终 accuracy / pass@k
* **Equal-token budget accuracy**
* **Action Regret@Budget**
* **Revision Harm Rate**
* **Compute Value Calibration**

#### 鲁棒性

* step segmentation 扰动
* paraphrase / prompt style 扰动
* 时间切片或来源切片（防 contamination）

#### 统计显著性

* 3–5 seeds
* paired bootstrap 95% CI
* paired permutation / McNemar（分类正确性）

#### 可解释性 / 机制验证

* hidden-state linear probes
* SAE feature discovery
* activation patching / feature steering
* “同 PRM 分数、不同 uncertainty” 的 counterfactual pair analysis

### 6.5 资源估算

#### 本地 GPU 预算（粗上限）

* **head-only 训练**：8 GPU × 18–24h × 3 seeds ≈ 18–24 GPU-days
* **LoRA distillation**：16 GPU × 24–36h × 2 runs ≈ 32–48 GPU-days
* **baseline reproduction**：40–60 GPU-days
* **SAE / probe / intervention**：8–16 GPU-days
* **总本地上限**：**约 110–150 GPU-days**

#### 显存与并行

* 7B head-only：2–4×80GB 即可
* 7B + LoRA + 4k–8k context：8×80GB 较稳
* 并行策略：FSDP / ZeRO-3 + FlashAttention + activation checkpointing

#### API 预算的最佳使用方式

* 不要把主结果建立在 API 模型之上
* 应把 API 预算用在：

  * 生成 diverse hard prefixes
  * 构造 counterfactual rollouts
  * 辅助自然语言 step 解析 / critique
* 最终训练与评测尽量本地可复现

## 7. 写作框架（标题 / 摘要 / 图表 / related work）

### 7.1 标题候选 ×5

1. **Reasoning Fails for Two Reasons: Separating Error from Uncertainty for Adaptive Verification**
2. **Verify, Branch, or Backtrack: Process-State Control for Test-Time Reasoning**
3. **Beyond Scalar Verifiers: Decomposing Process Error and Uncertainty in Reasoning Models**
4. **When Should a Reasoner Think More? Calibrated Process-State Estimation for Adaptive Compute**
5. **From Scoring to Control: Process Error–Uncertainty Decomposition for LLM Reasoning**

### 7.2 摘要骨架

* **背景**：reasoning LM 越来越依赖 verifier、self-correction 和 test-time scaling。
* **问题**：现有方法大多把 process quality 压成单一分数，无法区分“错了”与“不确定”。
* **方法**：提出 TriVer，学习局部错误概率与后续成功分布，并据此做 continue / revise / branch 控制。
* **结果**：在等 token 预算下提升 accuracy，并显著降低 revision harm 与 action regret。
* **机制**：SAE / intervention 表明 error-feature 与 uncertainty-feature 可分离，支持我们的过程状态假设。

### 7.3 贡献点 bullet

* 首次把 reasoning control 明确为 **error-vs-uncertainty decomposition** 问题
* 给出双头 process-state estimator 与 selective compute controller
* 提出 `Action Regret@Budget`、`Revision Harm Rate`、`Compute Value Calibration`
* 在 synthetic / math / formal transfer 上验证
* 提供机制分析，解释为什么 adaptive control 有效

## 最后的建议：为什么我会押注主推 idea

* 它不是“再做一个大模型”。
* 它不是“再调一个 RL loss”。
* 它同时抓住了：

  * **community relevance**：reasoning / verifier / test-time compute
  * **mechanistic depth**：error vs uncertainty
  * **theory space**：单标量不足
  * **new eval**：action regret / revision harm
  * **compute fit**：7B + head + verifiable tasks 足够跑扎实
* 这条线最有机会做出一种 **best-paper 风格的统一叙事**：
  **“额外 compute 不该盲目花；只有当模型知道自己是‘错了’还是‘不确定’时，推理才真正可控。”**
