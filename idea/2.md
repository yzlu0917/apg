## 1. 方向地图（含主题覆盖，Step A：广撒网）

> 时间窗口：我以 **2024Q4–2026Q1（过去约18个月）** 的 arXiv / OpenReview / 顶会论文为主做“热度+影响力”扫描（含 NeurIPS/ICLR/ICML/ACL/EMNLP 及其 D&B track）。
> 你的约束（不靠预训练堆算力、16×80GB A800、API预算充足、强调机制/理论与可证伪实验）会在后续筛选里强约束执行。

### A1) 推理与验证（reasoning / verifier / self-correction）

* 主线：**用 verifiers/PRM/过程监督**把“推理轨迹”变成可优化/可搜索对象；并通过 **test-time search / rerank** 或 RL 训练提升难题表现。([OpenReview][1])
* 痛点：verifier/PRM 容易 **false positive、reward hacking、风格/长度投机**，导致“看似提升、实则脆弱”。([OpenReview][2])
* 重要性：这是“推理可控化”的核心杠杆，且能做出**机制解释 + 可证伪预测**（你会很适配）。

### A2) 可靠性与对齐（robustness / honesty / calibration / uncertainty）

* 主线：从“能答对”走向“**知道自己何时会错**”：不确定性估计、校准、选择性回答/拒答、减少幻觉。([arXiv][3])
* 痛点：高准确率≠可靠不确定性；“校准好”也未必带来更好的错误排序/选择性策略。([arXiv][3])
* 重要性：顶会与产业都缺“能复现的可靠性协议”，容易出 best-paper 级评测/理论。

### A3) 表征与机制解释（mechanistic interpretability / representation learning）

* 主线：从“相关性解释”转向“**可干预的因果机制**”（benchmark + SAE/字典学习 + intervention）。([arXiv][4])
* 痛点：解释常停留在“故事”；缺少**可证伪的机制验证基准**与跨模型可迁移结论。([arXiv][4])
* 重要性：一旦做出“统一原则/可干预机制”，极易冲 best paper。

### A4) 训练与优化新范式（RLHF/RLAIF alternatives / off-policy / preference）

* 主线：以 **DPO/ORPO/SimPO** 等“离线偏好优化”替代高成本 RLHF；并探索 **off-policy RL** 与 reward model 鲁棒性。([arXiv][5])
* 痛点：对偏好噪声/分布外极其敏感；reward over-optimization 与参考模型选择仍是难题。([ICML][6])
* 重要性：可在你算力内做大量 ablation，并能结合因果/理论 lens。

### A5) 数据合成与评测（synthetic data / evaluation / contamination）

* 主线：从静态榜单转向 **动态、可验证、抗污染**基准（LiveBench / LiveCodeBench / EvoCodeBench）。([arXiv][7])
* 痛点：污染/泄漏、LLM-as-judge 偏差、评测分布漂移导致“假进步”。([NeurIPS][8])
* 重要性：best-paper 常见路径：**定义新评测维度 + 解释旧方法为何错**。

### A6) 高效学习（parameter-efficient / inference-aware / sample-efficient）

* 主线：不是堆预训练，而是 **inference-aware fine-tuning、BoN-aware、pass@K 优化**等把训练目标对齐推理预算。([NeurIPS][8])
* 痛点：很多方法“提升来自更大 test-time compute”，训练侧没有原则化对齐。([NeurIPS][8])
* 重要性：适合你：算力够做严谨实验，又不需要数千卡周。

### A7) 多智能体/工具使用（agents / tool use / planning / search）

* 主线：从“单轮工具调用”走向 **长程交互 + 可执行评测**（OSWorld/WebArena/AgentBench/GAIA/BFCL）。([arXiv][9])
* 痛点：环境不可控、评测不可复现、工具链安全与鲁棒性难统一。([arXiv][9])
* 重要性：若能提出**统一评测协议/机制解释**，很可能 best paper。

### A8) 检索与记忆（RAG / long-context / memory systems）

* 主线：不再简单争论“RAG vs LC”，而是用 **benchmark + sufficiency lens + memory agent**系统化比较与路由。([Proceedings of Machine Learning Research][10])
* 痛点：错误来源不清：检索不够、上下文不够、模型不会用、或被 distract。([OpenReview][11])
* 重要性：易做可复现系统实验，也易提出新评测维度。

### A9) 结构化生成/程序归纳（code / formal methods / theorem proving）

* 主线：用 **Lean4/形式验证**把“正确性”变成可执行判定；新基准（VeriBench/VeriBench-FTP）补上“MiniF2F 过饱和”的空白。([OpenReview][12])
* 痛点：搜索空间巨大、形式语言与自然语言对齐难、评测协议差异大。([arXiv][13])
* 重要性：最适合做“可验证、可复现、强机制”。

### A10) 安全与红队（jailbreak / adversarial robustness）

* 主线：从零散攻击到 **标准化评测与对抗训练**（HarmBench/JailbreakBench/MLCommons）。([arXiv][14])
* 痛点：攻击适配性强、评测威胁模型不统一；防御往往牺牲能力。([arXiv][15])
* 重要性：若能给出**统一机制 + 更强反事实基线**，会很硬。

## 3. 深入对比（每方向：现状 → 瓶颈 → 争议 → 可突破点；含代表论文 6–12 篇）

> 我会尽量把“常见设置 / 失败案例 / 争议点”写成可直接变成实验的条目。

### 3.1 推理与验证：PRM / Verifier / Self-correction

**现状（范式）**

* “生成多条推理轨迹 → verifier/PRM 打分 → rerank / search / RL 优化”成为主流。([OpenReview][1])
* PRM 数据获取从“人工过程标注”转向“搜索采样+自动过程验证”。([OpenReview][18])

**常见设置**

* 任务：数学推理/代码/多步决策；用 **oracle（执行器/判题器）**给最终正确性。
* 指标：pass@1 / pass@k、BoN(N) 提升曲线、PRM ROC-AUC、false positive/negative 分解。

**主要瓶颈**

* **False positives 更致命**：一步误判会在长链上累积。([OpenReview][19])
* RL fine-tune 时出现 **reward hacking**：模型学会“骗 PRM”。([OpenReview][2])
* 过程监督的“步级标签”定义不统一、跨域泛化差。([ACL Anthology][20])

**失败案例/争议点**

* “自我纠错”在很多任务上并不自然出现，需要专门训练（如在线RL）。([ICLR 会议录][21])
* 搜索+PRM 有时只是在“更高 compute 下更像 brute force”，机制解释不足。([OpenReview][22])

**可突破点（适合 best-paper 支点）**

* 做出：**“PRM/Verifier 的因果鲁棒性”**（不是更强 PRM，而是解释它为何被骗、如何避免被投机）。
* 给出：**可证伪预测**（例如：降低 FP 比降低 FN 对 BoN 曲线更关键）并在多域验证。([OpenReview][19])

**代表论文（6–12）**

* Scaling Automated Process Verifiers for LLM Reasoning（大规模自动过程验证/PRM）。([OpenReview][1])
* OmegaPRM（用 MCTS 收集高质量过程监督数据）。([OpenReview][18])
* PQM: Process Reward Model with Q-value Rankings（把 PRM 置于 MDP/Q-value 排序框架）。([OpenReview][23])
* Min-Form Credit Assignment…（定位 PRM reward hacking 原因：credit assignment 形式）。([OpenReview][2])
* Precision First Process Rewards…（分析 FP ceiling：FP 导致 BoN 上限）。([OpenReview][19])
* Training LMs to Self-Correct via RL (SCoRe, ICLR 2025)（在线RL让 self-correction 真正发生）。([ICLR 会议录][21])
* AgentPRM（把过程奖励扩展到 agents）。([arXiv][24])
* VersaPRM（多域 PRM）。([OpenReview][25])
* OpenPRM（开放域/指令跟随 PRM）。([OpenReview][26])
* LE-MCTS（用 PRM 引导推理步搜索/集成）。([ACL Anthology][27])

### 3.2 Test-time compute scaling：推理时算法 + 可证明缩放律

**现状（范式）**

* “训练不变，把 compute 挪到推理时”：BoN、tournament/league、MCTS、rerank。([arXiv][16])
* 开始出现“**可证明的 test-time scaling laws**”与更系统的实验验证。([arXiv][4])

**常见设置**

* 固定 base model；控制推理预算（LLM calls / tokens）；比较不同 selection/search 策略的正确率-成本曲线。

**主要瓶颈**

* 选择器（judge/verifier）噪声决定上限；比较策略的假设往往不满足（例如 pairwise 比较不稳定）。([arXiv][4])
* 很多工作难以区分“算法改进”与“采样更聪明/更多”的贡献。

**争议点**

* 是否存在“通用 test-time scaling law”，还是任务/模型依赖强？（论文给出理论条件但现实违反程度未知）。([arXiv][4])

**可突破点**

* 把“选择器噪声/FP”作为中心变量，建立 **选择器质量 → scaling 曲线形状**的机制解释（可连接 3.1）。
* “训练目标对齐推理策略”（inference-aware FT）是很强的切口。([NeurIPS][8])

**代表论文（6–12）**

* Scaling LLM Test-Time Compute Optimally…（推理时 compute 最优分配视角）。([arXiv][16])
* Simple/Provable Scaling Laws for Test-Time Compute（knockout/league）。([arXiv][4])
* Inference-aware Fine-tuning for Best-of-N（BoN-aware 训练）。([OpenReview][28])
* Rethinking Fine-Tuning when Scaling Test-Time Compute（pass@N 视角）。([NeurIPS][8])
* Pass@K Policy Optimization（训练目标从 pass@1 转向集合效用）。([NeurIPS][29])
* TS-LLM / AlphaZero-like tree search（树搜索引导 decoding/训练）。([OpenReview][30])
* DSG-MCTS（动态策略选择的 MCTS）。([ACL Anthology][31])
* SolverLLM（training-free test-time scaling 解优化问题）。([NeurIPS][32])

### 3.3 对齐新范式：DPO 系 + off-policy RL + RM 鲁棒性

**现状（范式）**

* 偏好优化从 RLHF(PPO) 转向 **离线目标（DPO/ORPO/SimPO…）**。([arXiv][5])
* 更关注“数据质量/噪声/参考模型/过优化”。([ACL Anthology][33])

**瓶颈**

* 对偏好噪声、分布外输入的 **reward model 泛化**是关键瓶颈。([ICML][6])
* 参考模型选择与更新（TR-DPO 等）仍缺统一理论。([arXiv][34])

**争议**

* 许多算法差异被超参/数据/评测偏差淹没；需要更强反事实基线与统一评测（可连接 3.5）。([arXiv][35])

**可突破点**

* 用“因果/反事实 lens”定义可复现评测：哪些提升只是 length/style confounders？([arXiv][35])

**代表论文（6–12）**

* Learn Your Reference Model…（TR-DPO/TR-IPO/TR-KTO）。([arXiv][34])
* SimPER（极简/少超参 PO）。([OpenReview][39])
* SACPO（带安全约束的对齐优化）。([OpenReview][40])

### 3.4 可靠性：不确定性/校准/拒答/幻觉

**现状**

* 系统性评估开始出现：跨 80 模型的 uncertainty study。([arXiv][3])
* “拒答/弃权”从安全需求扩展到减少幻觉的通用策略。([ACL Anthology][41])
* 新 hard benchmark 更强调多轮与“必须给引用”的 groundedness。([arXiv][42])

**瓶颈**

* 语言模型不确定性定义多样（TPU/NVU/LVU 等），与实际错误排序不一致。([arXiv][3])
* 多轮对话中的误差累积与“早期幻觉”传播难控制。([arXiv][42])

**争议**

* “让模型更会拒答”会不会损伤能力与用户体验？需要更细粒度评测（正确拒答 vs 过拒答）。([ACL Anthology][41])

**可突破点**

* 把“选择性生成/拒答”与“verifier/PRM 校准”统一成一个可证明框架（可连接 3.1/3.2）。([NeurIPS][43])

**代表论文（6–12）**

* Revisiting Uncertainty Estimation and Calibration of LLMs（80 模型系统研究）。([arXiv][3])
* Know Your Limits: Survey of Abstention（拒答综述与框架）。([ACL Anthology][41])
* HalluHard（多轮高风险域幻觉基准，要求引用）。([arXiv][42])
* On the Entropy Calibration of LMs（熵校准与长生成误差累积）。([NeurIPS][44])
* Efficient semantic uncertainty quantification（语义不确定性与采样效率）。([NeurIPS][45])
* Uncertainty Calibration of Process Reward Models（PRM 校准专门研究）。([NeurIPS][43])
* Revisiting Epistemic Markers in Confidence Estimation（语言标记与不确定性一致性）。([ACL Anthology][46])

### 3.5 数据合成与评测：抗污染、去偏、动态基准

**现状**

* “动态更新”成为抗污染核心：LiveBench、EvoCodeBench。([arXiv][7])
* “可执行判定”比 LLM-as-judge 更可靠：LiveCodeBench 等。([ICLR 会议录][47])
* 自动评测去偏（如长度偏差）开始用因果/反事实视角。([arXiv][35])

**瓶颈**

* 基准更新带来版本化难题：可比性、统计显著性、复现协议。
* 去偏方法容易被“新的 confounder”绕过，需要更系统化的 stress test。

**可突破点**

* 提出“反事实 baseline 协议”：任何方法必须在 **控制长度/风格/检索证据量**后报告提升。([arXiv][35])

**代表论文（6–12）**

* LiveCodeBench（可执行代码评测，cited-by 极高）。([ICLR 会议录][47])
* LiveBench（动态、抗污染、客观评分）。([arXiv][7])
* EvoCodeBench（演化更新+域标签评测）。([NeurIPS 会议录][48])
* Inference-Time Decontamination（利用泄漏基准做评测纠偏）。([ACL Anthology][49])
* Length-Controlled AlpacaEval（去长度偏差的反事实评测）。([arXiv][35])
* Data Contamination Survey（系统梳理污染问题）。([NeurIPS][8])

### 3.6 机制解释：SAE/干预/benchmark/表征工程

**现状**

* 从“解释一个模型”走向“**可比、可证伪的基准**”（InterpBench、MIB）。([arXiv][4])
* SAE 与字典学习用于发现特征；同时出现对“解释可置信度”的反思与基准化。([arXiv][50])
* 表征操控（activation steering / representation engineering）用于能力或对齐控制。([语义学者][51])

**瓶颈**

* 解释是否“真因果”：需要 intervention 与反事实验证，不只是可视化。([ICML][52])
* SAE 特征的“可解释性”可能被 prompt/分布变化破坏。

**可突破点**

* 把“解释”变成“预测”：给出可证伪机制预测（例如某特征被抑制后某类错误必然上升）。

**代表论文（6–12）**

* InterpBench（半合成已知 circuit 基准）。([arXiv][4])
* Mechanistic Interpretability Benchmark (MIB)。([arXiv][53])
* A Theoretical Framework for Mechanistic Interpretability（JMLR）。([ICML][52])
* Instruction-Following via Activation Steering（可控干预）。([语义学者][51])
* RouteSAE（路由到目标 SAE 特征）。([ACM Digital Library][54])
* Scaling Monosemanticity（大模型 SAE 特征发现）。([arXiv][50])
* Revising and Falsifying SAE Feature Explanations（解释可证伪）。([OpenReview][55])
* Automated Interpretability for Sparse Autoencoder Features（自动解释 SAE）。([arXiv][56])

### 3.7 RAG vs Long-Context vs Memory：路由、sufficiency、长期记忆

**现状**

* 争论转向“什么时候用 RAG/LC”与“错误源分解”。([Proceedings of Machine Learning Research][10])
* 长期记忆评测开始细分能力：抽取、多会话推理、时间推理、更新、弃权。([OpenReview][57])

**瓶颈**

* distract/位置敏感导致 LC 不稳；RAG 则受检索召回上限影响。([OpenReview][17])
* “路由器”常被 confounder（长度、证据量、问题类型）影响。

**可突破点**

* “Sufficient Context”提供一个强切口：把错误分为“上下文本身不够” vs “模型不会用”。([OpenReview][11])

## 4. 候选课题（3–5 个）+ Step C：系统化 gap 挖掘工具箱（明确“用哪些策略找到哪些 gap”）

### 4.0 我使用的 gap 工具箱（≥4种；这里用 5 种）

1. **Failure modes map**：把失败模式分解为可测子类（FP/FN、投机特征、长程 credit assignment、工具调用错误类型）。

4. **Unifying principle**：把分散工作统一成一个原理（例如“带噪声验证下的选择与搜索”）。([OpenReview][19])
5. **New eval lens**：提出新评测维度，让旧方法暴露（例如 verifier 的 style-invariance stress test、agent 的可执行“最小失败单元”评测）。([arXiv][14])

## 5. 主推 idea（完整研究蓝图）

> 我选 **候选课题 1：反事实不变性训练的 Verifier/PRM** 作为主推。理由：
>
> * 与当前最热的 PRM/test-time scaling 正交且补关键短板（鲁棒性/可依赖性）。([OpenReview][1])
> * 能给出强机制解释（FP、reward hacking、投机特征）与强可证伪预测。([OpenReview][19])
> * 训练成本主要在 verifier（7B–14B 级即可），非常适配你的算力约束。

---

### 5.1 选择主推 idea 前的“自问自答”（按你要求显式回答）

**Q1：最接近的 3 篇/3 类工作是谁？我们不可替代的东西是什么？**

* 最近工作类 1：**PRM/过程验证规模化**（如 Scaling Automated Process Verifiers、OmegaPRM）。([OpenReview][1])

  * 我们的不可替代：它们提升 PRM “强度”，但**没有系统化解决 verifier 学到 spurious cues 的因果问题**，也缺少反事实压力测试协议。
* 类 2：**指出 PRM 失败机制**（reward hacking/credit assignment、FP ceiling）。([OpenReview][2])

  * 我们的不可替代：从“指出问题”走到“**可复现的反事实生成 + 训练目标 + 可证伪机制实验**”的闭环。
* 类 3：**test-time scaling 的理论/选择算法**（knockout/league 等）。([arXiv][4])

  * 我们的不可替代：把这些理论的关键假设（judge 质量）落到可操作的 **verifier 鲁棒训练与校准**上，解释为何 scaling 有时“看似有效却不可靠”。

**Q2：贡献属于新机制/新理论/新评测/新问题定义哪种？为什么足够强？**

* 组合拳：

  * **新机制/原则**：**Counterfactual invariance** 作为 verifier 学到“因果正确性信号”的原则（抑制风格/长度投机）。
  * **新评测**：Verifier Stress Test（VST）：风格保持但正确性翻转的反事实对，测 FP/投机。
  * **半理论**：给出“FP → BoN 上限/曲线形状”的可检验推导与预测（连接 Precision-First 的分析）。([OpenReview][19])
* 足够强：它不仅追 SOTA，而是解释“为什么 test-time scaling 应该/不应该相信”。

**Q3：算力约束下最小闭环是什么？（2–3周出信号）**

* 只做 **代码/数学**两域（有执行器判定），构建小规模 VST（~10k–50k 反事实对），训练一个 7B verifier LoRA。
* 看 3 个信号即可决定继续：

  1. VST 上 FP 明显下降；
  2. BoN rerank 在固定 N 下稳定提升；
  3. 对长度/格式扰动不敏感（相关性显著下降）。

**Q4：best paper 级叙事一句话**

* **“Verifiers fail because they learn spurious, non-causal signals; counterfactual invariance turns verification into a reliable, measurable, and scalable primitive for test-time reasoning.”**

---

### 5.2 一句话主张（thesis claim）

* **通过“反事实不变性训练 + 校准”的 verifier/PRM，我们把 test-time scaling 从‘可用但脆’变成‘可依赖且可解释’，并用新的 verifier 压力测试基准证明改进来自降低 false positives 与抑制 reward hacking，而非工程堆算力。**

---

### 5.3 创新点列表（novelty checklist，≥5条；含与最接近3类工作逐项对比）

1. **反事实不变性数据生成器（可复现）**

   * 与 PRM 数据采集（OmegaPRM/自动过程验证）不同：我们生成 **style-preserving but correctness-flipping** 的成对样本，直接测/训“因果正确性”。([OpenReview][18])

2. **Verifier Stress Test (VST)：专门测 FP/投机的评测维度**

   * 对比现有 PRM 评测多看 overall AUC/accuracy：VST 把 confounder 控死（长度/格式/语气），逼出 verifier 的投机。

3. **训练目标：pairwise ranking + invariance regularization（而不是 step-wise CE）**

   * 对比 PQM（Q-value 排序）与 PPRM（偏好式 PRM）：我们用“反事实配对 + 不变性约束”让排序更接近 causal signal。([OpenReview][23])

4. **机制验证：把 FP ceiling 变成跨域可证伪预测**

   * 继承并扩展 Precision-First 的“FP 设上限”观点：我们预测并验证“**降低 FP 比降低 FN 更能抬升 BoN 曲线**”，且在数学与代码都成立。([OpenReview][19])

5. **把 reward hacking 诊断与缓解纳入统一协议**

   * 对比只指出 reward hacking 原因（min-form credit assignment）：我们提供可复现实验协议衡量“reward–truth gap”，并展示反事实训练降低 gap。([OpenReview][2])

6. **校准层：把 verifier 分数变成可用的选择性决策**

   * 连接 PRM 校准方向：提供 conformal/校准阈值，支持“拒绝/追加采样/触发检索”的决策。([NeurIPS][43])

### 5.4 核心洞察（为什么不是工程堆砌，而是新视角/机制）

* 现有 PRM/verifier 的核心问题不是“不够大”，而是 **监督信号把‘正确性’与‘风格/长度/熟悉模板’纠缠**。
* 反事实不变性把问题改写为：

  * “在保持所有 spurious 变量不变时，verifier 是否仍能区分正确性？”
* 这让你能做两件 best-paper 级的事：

  1. **定义可复现、可控 confounder 的评测**（VST）
  2. **给出机制化训练目标**（学习在反事实下不变的表示/评分）
* 这本质是“因果与鲁棒学习”在 verifier 上的落地，而不是“更多数据/更多算力”。

### 5.5 方法清晰定义（伪代码级别）

#### 5.5.1 记号

* base solver：生成候选解的模型 (G)（不需要训练或只轻微微调）
* verifier/PRM：(V_\theta(x, y)\rightarrow s\in\mathbb{R})（对问题 x、轨迹/答案 y 打分）
* oracle：可执行判定器 (O(x,y)\in{0,1})（代码执行/单测；数学用判题器或可靠 verifier）

#### 5.5.2 反事实生成（Counterfactual Pair Generator）

目标：构造 ((y^+, y^-))，满足：

* (O(x,y^+)=1)，(O(x,y^-)=0)
* 风格/长度/结构尽量匹配：(\text{style}(y^+)\approx \text{style}(y^-))

可复现的构造方式（优先程序化，减少LLM“玄学”）：

* **代码域**：对 AST 做最小变换（替换运算符、边界条件、常数、排序方向），保证可编译；再用单测筛选。
* **数学域**：对关键数值/符号做最小替换；用外部判题器或等式检查器筛选（若不可用，则只在可判定子集上做）。

#### 5.5.3 训练目标

* **Pairwise ranking loss（核心）**
  [
  \mathcal{L}*{rank} = -\log\sigma\big(V*\theta(x,y^+) - V_\theta(x,y^-)\big)
  ]
* **Invariance regularizer（可选但建议）**
  对 style-preserving 变换 (T)（如格式化、同义改写但不改语义），约束
  [
  \mathcal{L}*{inv}=|h*\theta(x,y)-h_\theta(x,T(y))|^2
  ]
* **Calibration loss（可选）**：Brier/温度缩放在验证集做后校准；并做 conformal 阈值选择。

总损失：
[
\mathcal{L}=\mathcal{L}*{rank}+\lambda \mathcal{L}*{inv}
]

#### 5.5.4 推理时用法（BoN rerank / search）

```
Given x:
  candidates = sample G(x) N times
  scores = Vθ(x, candidates)
  if max(scores) < τ (calibrated):
      abstain or trigger "more compute" (sample more / retrieve / ask clarification)
  else:
      return argmax scores
```

### 5.6 关键超参、训练细节、复杂度估计

**建议默认配置（可在你机器上稳跑）**

* Verifier backbone：7B–14B（LoRA 或全参，优先 LoRA 以便多 ablation）
* 序列长度：代码 2k–4k；数学 1k–2k（优先覆盖完整推理链）
* Batch：以 tokens/step 控制（目标：GPU 利用率>85%）
* (\lambda)（invariance）：从 0.1 起 sweep 到 1.0（看 VST-FP 与主任务 BoN 曲线）
* 训练数据：从 200k pairs 起步 → 2M pairs（逐步扩大，观察收益是否饱和）
* 校准：验证集做温度缩放 + conformal 阈值（对选择性策略很关键）

**复杂度**

* 数据生成（离线）：主要是执行器/单测成本（可并行、可用 API）。
* 推理：BoN rerank 需要 (N) 次 verifier forward（相较生成成本通常可接受）。
* 训练：仅 verifier 训练；不需要大规模预训练。

### 5.7 至少 3 个强基线（含“最难打败”）

1. **标准 PRM/Verifier（step-wise CE / outcome RM）**

   * 代表现有工程主流；最重要对照。([OpenReview][1])

2. **PQM（Q-value ranking PRM）**（强且理论化）([OpenReview][23])

3. **Min-form credit assignment / 相关 reward-hacking 缓解策略**（从 credit assignment 下手）([OpenReview][2])

4. **BoN / tournament / league 的纯 test-time 选择（不改 verifier）**

   * 用于证明“收益来自 verifier 质量提升而不是选择算法”。([arXiv][4])

---

### 5.8 至少 2 个我预期会失败的设置（失败也有价值）

1. **没有可执行 oracle 的开放域事实问答**

* 反事实对的“正确性翻转”难以可靠判定；此失败说明方法边界：需要可验证任务或强外部验证。
* 价值：可把 VST 作为“评测镜头”迁移过去，即使不训练也能揭示 verifier 投机。

2. **高度主观/多答案任务（写作、偏好极分散）**

* “正确性”本身不稳定，反事实训练可能学到噪声。
* 价值：推动你把“可验证性”写成清晰假设，并用 ablation 证明假设必要。

### 5.9 关键实验（Minimum Viable Experiments ≥6，带可证伪预期）

> 每个实验都写成 **falsifiable prediction**：不出现就说明假设错/实现有问题。

1. **VST 基准构建与指标**（核心）

* 预测：标准 PRM 在 VST 上 **FP 显著高**；我们的 CIV-PRM（counterfactual invariance verifier）显著降低 FP。
* 输出：VST-FP、VST-AUC、对长度/格式扰动敏感度。

2. **BoN rerank 主任务提升（固定 N）**（数学+代码至少两域）

* 预测：在同 N 下，我们的 rerank 增益 > 基线 PRM；并且增益主要来自 FP 降低（可分解）。([OpenReview][19])

3. **BoN scaling 曲线形状变化**（N=1,2,4,8,16,32）

* 预测：基线 PRM 曲线早饱和（FP ceiling）；我们的曲线饱和点右移/上移。([OpenReview][19])

4. **反事实控制 confounder 的 ablation**

* 去掉风格匹配（只翻转正确性不控风格）
* 预测：不控风格时，模型可能“看起来更好”但 VST 退化；控风格时更稳，证明收益来自因果信号而非风格。([arXiv][35])

5. **Reward hacking 诊断协议（小规模 RL/偏好优化）**

* 定义 reward–truth gap：被 verifier 高分但 oracle 错误的比例
* 预测：基线 PRM gap 更大；CIV-PRM gap 更小。([OpenReview][2])

6. **跨域泛化：math→code 或 code→math 的 verifier 迁移**

* 预测：仅靠规模化 PRM 很难迁移；反事实不变性训练迁移更好（因为抑制了域特定 spurious cues）。

7. **机制验证：score 与长度/格式的相关性**

* 预测：基线 PRM 的 score 与长度/特定模板强相关；CIV-PRM 显著下降（用相关系数/互信息）。([arXiv][35])

8. **对抗性“合理但错误”的轨迹**（hard negatives）

* 预测：CIV-PRM 对 hard negatives 更不容易给高分；若无差异，说明反事实生成没覆盖关键投机模式。

---

### 5.10 风险与备选路线（Plan B / C）

**主要风险**

* R1：反事实生成质量不够（翻转后太容易被识别/不自然）
* R2：oracle 判定覆盖不足（数学域尤其）
* R3：训练后 verifier 变得过于保守（FN 上升，影响 recall）

**Plan B（优先）**

* 先只做 **代码域（执行器强）**：HumanEval/LiveCodeBench，稳定出机制结论，再扩到数学。([ICLR 会议录][47])

**Plan C**

* 即使训练收益有限，也能产出 best-paper 级贡献：

  * VST 作为 **新评测镜头**（揭示 PRM/选择器普遍 FP/投机）
  * 负结果总结：哪些反事实有效、哪些无效（对社区很有价值）

---

### 5.11 诊断树（debug roadmap：结果不如预期怎么迭代）

1. **先看 VST-FP 是否下降**

* 否 → 检查：

  * 反事实对是否真的 style-preserving？（长度/格式分布）
  * oracle 是否有噪声？（单测不足/数学判定不可靠）
  * 训练是否被 class imbalance/采样偏差带偏

2. **VST 变好但主任务 BoN 没提升**

* 说明：VST 覆盖的 failure mode 未对主任务起主导作用
* 迭代：

  * 扩充反事实算子（针对主任务常见错误类型）
  * 引入“hard negatives mining”：从 BoN 采样里挑高分错解加入训练

3. **提升只在小 N，有大 N 仍饱和**

* 说明：仍有 FP ceiling 或校准阈值不对
* 迭代：

  * 强化 invariance / 做 score calibration
  * 用分桶分析：长链/短链分别画曲线找 FP 来源

4. **FN 上升（过保守）**

* 迭代：

  * 校准阈值用 conformal 控制 risk
  * 在 ranking loss 加入“margin schedule”或引入正确样本的 self-consistency 增强

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

> 以 **12周**为一轮可复现闭环；你可并行两条线（数据/评测线 + 方法/训练线）。
> 资源上限按你给的 **16×80GB A800** 设计，避免“堆预训练”。

### 6.1 数据与基准选择（含理由）

**主基准（可验证、可复现优先）**

* 代码：**LiveCodeBench**（可执行，抗污染倾向）。([ICLR 会议录][47])
* 代码补充：HumanEval/MBPP（通用、便于对比；你可自己下载公开版）。
* 数学：GSM8K/MATH（若用外部判题器可做）；若判定不稳则只用可判定子集。
* 评测镜头：

  * **VST（你构建的新基准）**：反事实对 + 版本化协议
  * 统计报告：FP/FN、BoN 曲线、校准曲线（ECE/Selective risk）

### 6.2 方法/训练细节（算法步骤、损失、推理流程）

* 周 1–2：实现反事实生成器（代码域先行）

  * AST 变换 + 单测筛选 → (y+, y-) pairs
* 周 2–4：训练 verifier（LoRA 起步）

  * loss：ranking + invariance（可先 λ=0，做 ablation）
* 周 4–6：BoN rerank 评测 + 曲线
* 周 6–8：加入 reward hacking 诊断（小规模 RL/偏好优化 或者 simulated reward）
* 周 8–12：跨域泛化 + 机制分析 + 写作图表定稿

### 6.3 评测协议（主指标、鲁棒性、统计显著性、机制验证）

**主指标**

* 主任务：pass@1 / pass@k；在固定推理预算下的准确率-成本曲线
* VST：VST-FP（核心）、AUC、风格扰动敏感度
* 校准：ECE、Selective risk-coverage 曲线（当你引入阈值策略）

**鲁棒性**

* 长度/格式扰动（控制 confounder）
* hard negatives（合理但错）
* 迁移（train-domain vs test-domain）

**统计**

* bootstrap CI（不同题目采样）；显著性用 paired test（同一题目对比）。
* 必须报告：多 seed（≥3）与关键 ablation。

**机制验证（必须做）**

* score 与长度/模板特征的相关性（训练前后对比）
* FP/FN 分解对 BoN 曲线贡献（用分桶或 counterfactual analysis）

### 6.4 资源估算（显存、训练、GPU 天数上限、并行策略）

**Verifier 训练（建议）**

* 7B LoRA：

  * 显存：单卡 80GB 足够（bf16 + gradient checkpointing）
  * 并行：data parallel + ZeRO/FSDP
  * GPU-days 上限建议：**≤ 80 GPU-days**（足够 sweep λ、数据规模、不同 backbone）

**数据生成**

* 单测/执行器并行：CPU 集群 + GPU（可用你 API 预算加速生成/过滤）

**总上限（保守写法）**

* 全项目：**≤ 200 GPU-days**（包含多次 ablation、两域实验），远低于“堆预训练”路线。

### 6.5 复现与工程组织（实验配置、日志、超参搜索）

* 配置：Hydra/JSONnet；所有实验以 config hash 命名
* 日志：W&B/MLflow + 保存每次数据版本（VST v0.1/v0.2…）
* 超参搜索：

  * 第一阶段粗搜：λ、pair 采样策略、训练步数
  * 第二阶段精搜：阈值校准、hard negative mining 比例
* 产出物：

  * 数据：反事实生成脚本 + 版本化数据卡
  * 模型：verifier checkpoints + inference rerank 脚本
  * 评测：一键复现的 leaderboard 脚本（固定随机种子）

## 7. 写作框架（标题×5、摘要骨架、贡献点、图表草图、related work）

### 7.1 标题候选（5个）

1. **Counterfactual Invariance for Verifiers: Making Test-Time Scaling Reliable**
2. **False Positives Are the Bottleneck: A Counterfactual Stress Test and Fix for Process Reward Models**
3. **From Stronger to Safer Verifiers: Counterfactual Training for Robust PRM-Guided Reasoning**
4. **Verifier Stress Testing: Measuring and Reducing Spurious Verification in LLM Reasoning**
5. **Causal Verification Under Style Control: Robust Selection for Best-of-N and Search**

### 7.2 摘要骨架（可直接填空）

* 背景：PRM/verifier 驱动推理时 scaling，但存在 FP/reward hacking/投机导致不可靠。([OpenReview][19])
* 问题：现有评测难以隔离 confounder（风格/长度），导致“看似提升”。
* 方法：提出 VST（反事实压力测试）+ counterfactual invariance training（ranking + invariance + calibration）。
* 理论/机制：给出 FP 对 BoN 曲线的可检验预测，并验证训练降低 FP 与 reward–truth gap。([OpenReview][19])
* 结果：在代码/数学域提升固定预算下性能与鲁棒性；跨域泛化更好。
* 贡献：数据/评测协议开源、模型与脚本可复现。

### 7.3 贡献点 bullet（best-paper 风格）

* **新评测镜头**：Verifier Stress Test（style-controlled correctness flips），揭示现有 PRM 的系统性 FP 与投机。
* **新原则/机制**：counterfactual invariance 让 verifier 学到更接近因果正确性的信号。
* **可证伪机制实验**：FP 降低解释 BoN scaling 曲线改善；reward hacking gap 缩小。
* **跨域验证**：代码（可执行）+ 数学（可判定子集）统一协议。
* **可复现**：版本化数据、训练、评测一键复现。
