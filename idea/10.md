下面给的是“先广撒网，再收缩到最可能冲击 best paper 的主线”的完整版本。口径上，我把 **2024-09 到 2026-03** 作为主窗口；其中 **2026 的 OpenReview 公开投稿**我会当作前沿 signal，但在影响力判断里会对“submitted / withdrawn / under review”降权，不把它们当作已定论。([开放评论][1])

## 1. 方向地图（含主题覆盖）

### 1.1 Step A：广撒网——10 个主题的“主线 / 痛点 / 重要性”

1. **推理与验证（reasoning / verifier / self-correction）**

   * 主线：从 CoT 本身，转向 **solver–verifier**、PRM、solution verifier、test-time scaling、reasoner-verifier 一体化。
   * 痛点：**自验证经常失灵**；PRM/step verification 代价高，且并非总比 Best-of-N 强；验证收益强依赖题目难度、generator 强弱、family 差异。
   * 重要性：这是 2025–2026 “推理热”里最核心的一条，因为它直接决定 test-time compute 到底有没有用。 ([开放评论][2])

2. **可靠性与对齐（robustness / honesty / calibration / uncertainty）**

   * 主线：从“给一个置信分数”升级到 **语言化不确定性、潜在不确定性、过度自信、honesty-style calibration**。
   * 痛点：模型常常 **知道自己不确定，但不会正确表达**； verbal confidence 与真实正确率不一致；instruction-following 场景尤其脆弱。
   * 重要性：如果 uncertainty 不能转化成行动策略（abstain / verify / tool-use），很多“可靠性”工作都只停留在诊断。 ([开放评论][3])

3. **表征与机制解释（mechanistic interpretability / representation learning）**

   * 主线：从 head/neuron 级别转向 **feature circuits、transcoder、SAE、跨训练/跨尺度稳定性**。
   * 痛点：**SAE feature 是否 canonical**、是否因相关特征而“假单义性”仍在争议；可解释单元的选择本身还没稳定。
   * 重要性：这是做“为什么有效”最有潜力的支点，但要小心别落入只做 case study 的坑。 ([开放评论][4])

4. **训练与优化新范式（RLHF/RLAIF alternatives, off-policy RL, direct preference, credit assignment）**

   * 主线：DPO 理论化、off-policy RLHF、reward-model correction、RLVR、selective rollouts。
   * 痛点：**非平稳性、reward misspecification、off-policy 偏差、credit assignment 稀疏**；“RL 到底学到了 reasoning 还是只学了格式”争议仍很大。
   * 重要性：这是 post-training 的核心科学问题，不是单纯 recipe 竞赛。 ([开放评论][5])

5. **数据合成与评测（synthetic data, evaluation, contamination, benchmark design）**

   * 主线：从静态 benchmark 转向 **dynamic evaluation、contamination-aware benchmark、benchmark-free evaluation、synthetic benchmark quality**。
   * 痛点：静态 benchmark 容易被污染；synthetic benchmark 往往 **难度、shortcut、judge leakage** 控不住。
   * 重要性：很多所谓“能力进步”其实先是评测问题；这个方向非常适合做顶会级、可复现、可被社区长期采用的贡献。 ([ACL Anthology][6])

6. **高效学习（parameter-efficient / low-data / modularity / compositionality）**

   * 主线：Self-MoE、结构化专家化、低资源合成数据、compositional generalization 的理论化。
   * 痛点：不少工作还是“效率工程”；真正能给出 **可解释模块化收益** 的很少。
   * 重要性：在你的算力约束下，这一类方向天然友好，但要避免做成普通 PEFT paper。 ([开放评论][7])

7. **多智能体 / 工具使用（agents, tool use, planning, search）**

   * 主线：web agent、deep research agent、tool planning、agent benchmark、multi-agent coordination。
   * 痛点：很多结果对 prompt / observation space 极敏感；**简单 baseline 常常就能追平复杂 agent 架构**。
   * 重要性：社区热度极高，但最好切“可验证子问题”，否则容易变成系统工程。 ([开放评论][8])

8. **检索与记忆（RAG, long-context, memory systems）**

   * 主线：long-context 真实评测、KV-cache 视角、retrieval heads、revisitable memory、multi-turn shared-context。
   * 痛点：needle-in-a-haystack 类测试太理想化；真实多轮共享上下文里模型仍会“lost in conversation”。
   * 重要性：这类方向既能做系统，也能做机制解释，因为 retrieval head / cache reuse 已经有明确结构线索。 ([开放评论][9])

9. **结构化生成 / 程序归纳 / formal methods**

   * 主线：Lean theorem proving、thinking+proving、rewarding formal proof search、proof-progress oracle。
   * 痛点：proof search 稀疏奖励、benchmark overlap、domain transfer、informal thought 与 formal tactic 的接口仍粗糙。
   * 重要性：这是少数可以拿到 **真 oracle** 的 LLM reasoning 场景，非常适合做 theory + mechanism + falsifiable prediction。 ([开放评论][10])

10. **安全与红队（jailbreak, adversarial robustness）**

    * 主线：adaptive jailbreak、agent security benchmark、multi-turn jailbreak、rule-based safety reward。
    * 痛点：攻击普遍强于防御；很多防御只是在特定 benchmark 上“过拟合安全策略”；agent 安全尤其缺统一评测。
    * 重要性：高社会价值，高社区关注；但若做主方向，最好切在 **selective abstention / verifier / formal risk control** 这类可验证子题。 ([开放评论][11])

### 1.2 我为什么把深挖方向收缩为 7 条

* 这 7 条在 **“机制解释潜力 × 评测清晰度 × 16×80GB 可做扎实”** 的乘积上最高。
* 我刻意把“纯堆算力长训练”的路线排除了。
* 我优先选那些 **最近 6–18 个月既有热度，又开始出现反例/负结果/理论争议** 的方向，因为这正是 best-paper 级选题最容易切入的窗口。 ([开放评论][12])

## 3. 深入对比（现状 → 瓶颈 → 争议 → 可突破点）

### 3.1 D1：验证式推理与 test-time scaling

**代表论文（建议必读 10 篇）**

* `Rewarding Progress: Scaling Automated Process Verifiers for LLM Reasoning`：自动化 PRM，强调 “effective reward”，是这条线的主锚点。 ([开放评论][2])
* `On the self-verification limitations of large language models on reasoning and planning tasks`：系统性说明 **self-critique 常失败**，sound external verifier 才稳。 ([开放评论][17])
* `Beyond Solving: A Closer Look at LLMs as Solution Verifiers`：37 models × 9 datasets，提出 **verifier gain**。 ([开放评论][1])
* `Variation in Verification`：把验证效果拆成 **题目难度 / generator 能力 / verifier 能力** 三维。 ([开放评论][18])
* `Reasoning with Confidence`：用 **<10M** 的 uncertainty heads 做 step verification。 ([开放评论][19])
* `Putting the Value Back in RL`：把 verifier/value 重新接回 reasoning RL。 ([开放评论][20])
* `VeriThinker`：训练“学会验证”来压缩 CoT，而不是直接蒸馏短 CoT。 ([开放评论][21])
* `One-Token Verification for Reasoning LLMs`：把 verification 极度轻量化。 ([开放评论][22])
* `Learning to Refine at Test-Time via Intrinsic Self-Verification (ReVISE)`：把自验证变成 refine policy。 ([开放评论][23])
* `Limits of PRM-Guided Tree Search for Mathematical Reasoning`：负结果，PRM-guided tree search **未必优于 Best-of-N**。 ([开放评论][12])

**现状**

* 从“生成更多样本”转向“生成 + 验证 + 选择”。
* 新工作开始明确：验证收益不是常数，而是实例依赖。 ([开放评论][1])

**主要瓶颈**

* 验证是否有效，取决于 **候选中是否有正确答案**，以及 **正确/错误候选是否可分**。
* PRM 训练贵；solution verifier 往往黑盒；跨 family blind spot 机制尚不清楚。 ([开放评论][2])

**争议 / negative results**

* “验证比生成容易”这句经典假设，对 LLM 并不稳。 ([开放评论][17])
* 更复杂的 search 未必比简单 Best-of-N 强。 ([开放评论][12])
* test-time scaling 也不是总有效，甚至会带来更多 hallucination。 ([开放评论][24])

**可突破点**

* 不是再做一个更大 verifier。
* 真正的 gap 是：**在花计算前，就预测“这一题值不值得验证、该用哪种 verifier、该不该 abstain / tool-route”。**

### 3.2 D2：不确定性、诚实性与 calibration

**代表论文（8 篇）**

* `Can Large Language Models Faithfully Express Their Intrinsic Uncertainty in Words?`：提出 verbalized uncertainty 的 **faithfulness** 问题。 ([开放评论][3])
* `Do LLMs estimate uncertainty well in instruction-following?`：专门考察 instruction-following 里的 uncertainty。 ([开放评论][25])
* `Graph-based Confidence Calibration for Large Language Models`：辅助模型做 confidence calibration。 ([开放评论][26])
* `Unifying Latent Uncertainty Signals in Large Language Models for Improved Factual Precision`：从 latent signal 角度统一 UQ。 ([开放评论][27])
* `Learn to be Honest: Mitigate LLMs' Overconfidence ... Self-Hesitation Activation`：从 overconfidence 出发做 honesty-style intervention。 ([开放评论][28])
* `From Calibration to Collaboration`：指出 UQ 应该服务于 **human-AI collaboration**，不是只报分数。 ([开放评论][29])
* `Can Large Language Models Express Uncertainty Like Human?`：把 uncertainty 表达方式和人类沟通习惯对齐。 ([开放评论][30])
* `Epistemic Integrity in Large Language Models`：聚焦 linguistic assertiveness 与真实准确率错位。 ([开放评论][31])

**现状**

* 已经从“logit confidence 能不能用”升级到“语言表达出的 certainty 是否真实反映内部 epistemic state”。
* latent UQ 与 honesty 正在汇合。 ([开放评论][3])

**主要瓶颈**

* 多数工作停在 **诊断**，没有把 uncertainty 接到 **verification / abstention / routing** 的决策上。
* verbal confidence 很容易被 prompt 风格污染。 ([开放评论][3])

**争议**

* 模型到底是“不会估计不确定性”，还是“会估计但不会表达/不会使用”？
* latent signal 能否跨模型族泛化，仍未定。 ([开放评论][27])

**可突破点**

* 把 uncertainty 从“输出属性”升级为“**是否值得花验证计算**”的决策变量。

### 3.3 D3：机制解释与表征稳定性

**代表论文（8 篇）**

* `Transcoders Find Interpretable LLM Feature Circuits`：把 circuit analysis 推到 feature 级。 ([开放评论][4])
* `Talking Heads: Understanding Inter-Layer Communication in Transformer Language Models`：看 inter-layer communication 的可解释结构。 ([开放评论][32])
* `LLM Circuit Analyses Are Consistent Across Training and Scale`：提出 circuits 在训练与尺度上有稳定性。 ([开放评论][33])
* `Sparse Autoencoders Do Not Find Canonical Features`：直接挑战 SAE 单义 feature 假设。 ([开放评论][34])
* `Correlated Features Break Narrow Sparse Autoencoders`：指出相关特征会破坏 monosemanticity。 ([开放评论][35])
* `Correct Features in Sparse Autoencoders`：讨论 SAE 的“正确 L0”不是随便调的。 ([开放评论][36])
* `Automatically Identifying and Interpreting Sparse Circuits in Transformers`：试图自动化 circuit identification。 ([开放评论][37])
* `Toward a Theory of Generalizability in LLM Mechanistic Interpretability`：开始讨论机制解释结论何时可泛化。 ([开放评论][38])

**现状**

* 机制解释正从“找有趣 neuron/head”转到“找 feature-circuit + 可泛化解释”。 ([开放评论][4])

**主要瓶颈**

* 单元选择不稳。
* 解释与任务收益之间仍缺闭环。 ([开放评论][34])

**争议**

* SAE/transcoder 到底找到了“真实 feature”，还是一种工具产物。
* 小模型/末端层结论能否迁移到大模型/后训练模型。 ([开放评论][34])

**可突破点**

* 不把机制解释当独立目标。
* 把它用于解释 **为什么同族 verifier 共享 blind spot、为什么 cross-family 更有效**。

### 3.4 D4：后训练优化——PO / off-policy RLHF / RLVR

**代表论文（9 篇）**

* `What Is Preference Optimization Doing, How and Why?`：直接研究 DPO/PPO 的优化动力学。 ([开放评论][5])
* `Likelihood Displacement in Direct Preference Optimization`：解释 DPO 类方法的分布偏移问题。 ([开放评论][13])
* `Direct Preference Optimization With Unobserved Preference Heterogeneity`：从偏好异质性角度修正 DPO 假设。 ([开放评论][39])
* `Asynchronous RLHF`：off-policy RLHF 的高效路线。 ([开放评论][40])
* `Off-Policy Corrected Reward Modeling for RLHF`：把 RM drift 当 distribution shift。 ([开放评论][41])
* `Tapered Off-Policy REINFORCE`：稳定高效的 off-policy RL。 ([开放评论][42])
* `Reinforcement Learning with Verifiable Rewards`：开始给 RLVR 提理论解释。 ([开放评论][43])
* `Act Only When It Pays: Efficient RL via Selective Rollouts`：只对有信息量的 prompt 花 rollout。 ([开放评论][44])
* `Does Reinforcement Learning Really Incentivize Reasoning Capacity Beyond the Base Model?`：对“RL 学到了什么”提出高影响质疑。 ([开放评论][45])

**现状**

* 社区正在从“谁的 recipe 更强”转向“为什么 DPO / PPO / RLVR 会好或不好”。
* selective rollout / off-policy / reward correction 是强信号。 ([开放评论][40])

**主要瓶颈**

* credit assignment 稀疏；RM 漂移；偏好数据分布非平稳。
* 大量工作仍高度依赖大规模 rollout。 ([开放评论][41])

**争议**

* RL 是否真正提升 reasoning，而非只学会更长、更像 reasoning 的轨迹。 ([开放评论][45])

**可突破点**

* 不直接卷更大 RL。
* 做 **value/verifier/uncertainty 的统一视角**，或者在 verifiable domain 里用更强 credit signal。

### 3.5 D5：动态评测、合成数据与污染控制

**代表论文（10 篇）**

* `DyVal: Dynamic Evaluation of Large Language Models`：动态评测 protocol。 ([开放评论][46])
* `LiveCodeBench`：当前最强 contamination-free code benchmark 之一。 ([开放评论][14])
* `DynaCode`：复杂度感知的动态 code benchmark。 ([ACL Anthology][47])
* `DyCodeEval`：面向 reasoning code tasks 的 dynamic benchmarking。 ([开放评论][48])
* `Evaluating Language Models as Synthetic Data Generators (AGORABENCH)`：标准化比较 LM 生成训练数据的能力。 ([ACL Anthology][49])
* `What Has Been Lost with Synthetic Evaluation?`：指出 synthetic benchmark 与高质量 crowd benchmark 的差距。 ([ACL Anthology][50])
* `DCR: Quantifying Data Contamination in LLMs Evaluation`：更细粒度量化 contamination risk。 ([ACL Anthology][51])
* `Benchmark Contamination Detection: Fragility in Reasoning Evaluation`：指出 reasoning leaderboard 容易被训练导向。 ([开放评论][52])
* `BeyondBench`：往 benchmark-free 评测走。 ([开放评论][53])
* `Benchmarking LLMs Under Data Contamination: A Survey from Static to Dynamic Evaluation`：系统综述。 ([ACL Anthology][6])

**现状**

* 动态 benchmark 与 contamination-aware eval 已从边缘话题变成主流问题。 ([ACL Anthology][6])

**主要瓶颈**

* “动态”不等于“高质量”。
* synthetic benchmark 往往更易被 shortcut 或 judge leakage 攻破。 ([ACL Anthology][50])

**争议**

* moderate contamination 是否会在训练后期“被遗忘”。 ([开放评论][54])

**可突破点**

* 不只是做新 benchmark。
* 做 **能把 solver 能力、verifier 能力、gain calibration、abstention 能力拆开评** 的评测新镜头。

### 3.6 D6：Agents / 工具使用 / 长上下文记忆

**代表论文（10 篇）**

* `A Simple Yet Strong Baseline for LLM-Based Web Agents`：提醒大家很多复杂 agent 可能没赢过强 baseline。 ([开放评论][55])
* `ToolPlanner`：多工具聚类规划。 ([开放评论][56])
* `ToolTree`：分支剪枝的 tool planning。 ([开放评论][57])
* `OrchestrationBench`：多领域 agent orchestration benchmark。 ([开放评论][58])
* `MLGym / MLGym-Bench`：ML research agent benchmark。 ([开放评论][15])
* `DeepResearch Bench`：深度研究 agent 的综合 benchmark。 ([开放评论][59])
* `BrowseComp-Plus`：deep research / retrieval 的更透明评测。 ([开放评论][60])
* `DuoAttention`：retrieval head / KV cache 结构化优化。 ([开放评论][9])
* `SCBench` + `ReMemR1` + `LLMs Get Lost In Multi-Turn Conversation`：说明真实 shared context 和记忆回调仍是大问题。 ([开放评论][61])

**现状**

* agent 线极热，但变量太多。
* memory / context / planning / tool interface 正在逐步解耦。 ([开放评论][15])

**主要瓶颈**

* 环境设置、观察空间、工具接口一改，结果可能重排。
* 长上下文 synthetic test 常过度乐观。 ([开放评论][55])

**争议**

* 是“规划能力”不足，还是 observation/action abstraction 不对。
* 复杂 agent 框架是否真的需要。 ([开放评论][55])

**可突破点**

* 只切 **可验证子任务**：例如 tool-planning 的 selective verification / safe quitting / memory callback。

### 3.7 D7：Formal reasoning / theorem proving

### 4.2 在选主推 idea 前，我先回答 4 个关键问题

**Q1. 最接近的 3 篇 / 3 类工作是谁？我们多了什么？**

* 类 1：`Rewarding Progress` / `Beyond Solving` / `Variation in Verification` 这类 **solver–verifier / verifier gain** 工作。

  * 它们告诉我们 verifier 有时有效。
  * **我们要多做的是**：在 **运行 verifier 之前** 预测 gain，并把 gain 分解成可检验的变量。 ([开放评论][2])
* 类 2：`Reasoning with Confidence` / calibration / honesty 这类 **uncertainty-based** 工作。

  * 它们告诉我们 hidden state 或 verbal signal 含有 uncertainty。
  * **我们要多做的是**：把 uncertainty 变成 **verification routing policy**，而不是只做 confidence estimation。 ([开放评论][19])
* 类 3：`Adaptive Budget Forcing` / `SelfBudgeter` / selective rollout 这类 **adaptive compute** 工作。

  * 它们调“想多久、rollout 多久”。
  * **我们要多做的是**：调“**该不该验证、由谁验证、何时 abstain**”。 ([开放评论][68])

**Q2. 这个贡献属于哪一种？为什么足够强？**

* 不是单一类型。
* 我认为它是 **新机制 + 新理论 + 新评测镜头 + 新问题定义** 的组合。
* 强点在于：它不是“更强模型”，而是提出一个新的 **state variable：verifiability**，并给出可证伪的公式、机制预测、评测协议。

**Q3. 在算力约束下，最小闭环是什么？**

* 2–3 周内最小闭环：

  * 复现 `Best-of-N + verifier` 基线。
  * 在 3 个可验证域（数学 / 代码 / planning puzzle）上构造 2–4 个 generator family 的 candidate 池。
  * 训练一个小 head 预测 **是否有正确候选（coverage）** 和 **验证是否会带来净收益（gain）**。
  * 若能做到 **固定精度下降低 20–40% verify cost**，或 **固定预算提升 2–4 个点**，就继续。
* 这是在 16×80GB 和充足 API 预算下完全现实的。

**Q4. 这个方向的 best-paper 级叙事是什么？**

* **“不是所有问题都应该被验证；真正的进步来自学会何时验证、何时停止、何时承认不可验证。”**

## 5. 主推 idea（完整研究蓝图）

### 5.1 一句话主张（thesis claim）

**VISTA：把 solver–verifier 系统的收益建模为“候选覆盖率 × 错误可检测性 × blind-spot 去相关”的函数，并在花 test-time compute 前预测该收益，从而实现更优的 accuracy–cost–reliability Pareto。**

---

### 5.2 为什么我主推这个 idea

* 它踩中 **D1 + D2 + D5** 三条最热且最缺统一理论的线。
* 它天然适配你的资源：

  * 不需要从零大规模预训练。
  * 需要的是 **candidate generation、轻量 head、LoRA verifier、广泛评测**。
* 它同时有：

  * **强理论锚点**：gain decomposition；
  * **强机制锚点**：blind-spot overlap / latent detectability；
  * **强评测锚点**：新的 verifiability phase diagram 与 counterfactual benchmark。

---

### 5.3 Novelty checklist（含与最接近 3 类工作逐项对比）

**对比类 A：solver–verifier / PRM / verifier gain**

* 现有工作：证明 verifier 有时有用，或提出事后统计量。 ([开放评论][2])
* 我们新增：

  1. **实例级前验 gain 预测**，不是事后解释。
  2. **选择 verifier 类型**，不是固定 verifier。
  3. **把 abstention/tool-route 纳入同一策略空间**。

**对比类 B：uncertainty / honesty / calibration**

* 现有工作：估计或表达 uncertainty。 ([开放评论][27])
* 我们新增：
  4. uncertainty 不再是报告数值，而是变成 **是否验证 / 如何验证** 的控制变量。
  5. 把 verbal / latent / cross-model disagreement 统一成 **detectability features**。

**对比类 C：adaptive compute / adaptive reasoning length**

* 现有工作：调 reasoning length / rollout 预算。 ([开放评论][68])
* 我们新增：
  6. 调的是 **verification budget** 和 **verification route**，不是只调想多久。
  7. 目标函数从 accuracy-only 改成 **accuracy–cost–risk**。

**额外 novelty**

* 8. 引入 **blind-spot overlap**：解释 why cross-family verification often helps。
* 9. 引入 **Counterfactual Verifiability Benchmark（V-MAP/CVBench）**：把 solver 能力与 verifier 能力拆开评。
* 10. 给出 **falsifiable phase-diagram predictions**，不靠玄学故事。

---

### 5.4 核心洞察：为什么这不是工程堆砌

* 许多近期论文都在围着一个隐变量转，但没有把它命名清楚：

  * 题里有没有正确候选？
  * 错误轨迹是否容易被识别？
  * generator 和 verifier 会不会共享同一种盲点？
* 这个隐变量就是 **verifiability**。
* 一旦把它显式化，你会得到三个自然后果：

  1. 验证收益应该在难度上呈 **非单调 / 倒 U 型**；
  2. 同族 verifier 在某些 regime 会更差，因为 blind spot 重叠；
  3. “更长 reasoning”与“更多 verify”都不该均匀加，而应按实例选择。
* 这是一种 **新视角**，不是“把 3 个 trick 拼起来”。

### 5.5 Step C：找 gap 的系统化工具箱（我实际用了哪些策略）

| 策略                      | 我如何用                                                                                            | 找到的 gap                                                                        |
| ----------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |

### 5.6 方法定义：**VISTA**（Verifiability-Informed Selective Test-time Allocation）

#### 5.6.1 记号

* 输入问题：(x)
* 生成器：(g)
* 候选答案/轨迹：(Y={y_i}_{i=1}^K)
* verifier bank：(\mathcal V={v_j}_{j=1}^J)
* correctness：(c_i \in {0,1})
* 轻量特征：(z_i)（logprob、entropy、length、disagreement、hidden-state pooling、verbal confidence 等）

#### 5.6.2 三个核心量

1. **Coverage**
   [
   q(x,Y) \approx P(\exists i: c_i=1 \mid x,Y)
   ]
   表示候选池里是否“有救”。

2. **Detectability**
   [
   \rho_j(x,Y) \approx P(v_j(y^+) > v_j(y^-) \mid x)
   ]
   表示 verifier (v_j) 能否把正确和错误轨迹分开。

3. **Blind-spot overlap**
   [
   o_j(x) \approx P(\text{generator error overlaps verifier error} \mid x,g,v_j)
   ]
   表示 generator 与 verifier 是否会犯同类错。

#### 5.6.3 理论锚点（简洁版）

* 在 **2-candidate pairwise reranking** 的理想化设置下，验证带来的期望增益近似正比于：
  [
  \Delta \propto q \cdot (2\rho - 1)
  ]
* 直观含义：

  * 没有正确候选（(q) 低），验证再强也没用。
  * 错误不可分（(\rho \approx 0.5)），验证再贵也没用。
* 再乘上 overlap 修正和成本项：
  [
  U_j = q \cdot \max(0, 2\rho_j - 1)\cdot (1-o_j) - \lambda , \text{cost}(v_j)
  ]
* 这给出一个 **可证伪** 的政策：只在 (U_j>0) 时调用 verifier (v_j)。

#### 5.6.4 训练目标

* **Coverage head** (E_\phi)：预测 (q)

  * loss：BCE
* **Detectability head** (D_{\psi,j})：预测 (\rho_j)

  * loss：pairwise ranking loss / AUC surrogate
* **Overlap head** (O_{\eta,j})：预测 (o_j)

  * loss：BCE（同题上 generator 与 verifier 是否共同犯错）
* **Calibrator** (C_{\omega,j})：对 verifier score 做校准

  * loss：Brier + ECE surrogate

总损失：
[
\mathcal L = \mathcal L_{\text{cov}} + \alpha \mathcal L_{\text{det}} + \beta \mathcal L_{\text{ovl}} + \gamma \mathcal L_{\text{cal}}
]

#### 5.6.5 推理策略

```text
Input x, generator g, verifier bank V, budget B

1. Draft:
   sample K short candidates Y = {y_i} from g
   extract cheap features Z = {z_i}

2. Estimate:
   q_hat = E_phi(x, Z)
   for each verifier v_j:
       rho_hat[j] = D_psi_j(x, Z)
       ovl_hat[j] = O_eta_j(x, Z)
       U_hat[j] = q_hat * max(0, 2*rho_hat[j]-1) * (1-ovl_hat[j]) - lambda * cost(v_j)

3. Route:
   if max_j U_hat[j] < tau_abstain and q_hat low:
       abstain or route to tool
   elif max_j U_hat[j] < tau_verify:
       return best draft candidate
   else:
       j* = argmax_j U_hat[j]
       run verifier v_j* on top-M candidates
       select candidate with best calibrated verifier score

4. Optional:
   if q_hat is low but "resample utility" > 0:
       generate extra candidates before verification
```

#### 5.6.6 关键超参（建议起点）

* draft 候选数 (K=4) 或 (8)
* top-M verified finalists：(M=2) 或 (4)
* 温度混合：0.2 / 0.8 各一半，制造 candidate diversity
* feature layers：取 4 个中高层 pooled hidden states
* 轻量 head：5M–20M 参数
* overlap feature：family ID、logit margin、representation similarity、candidate disagreement
* (\lambda)：按验证 token 成本归一化，在开发集做 Pareto sweep

#### 5.6.7 复杂度

* 固定 always-verify：
  [
  O(KL_g + ML_v)
  ]
* VISTA：
  [
  O(KL_g + p \cdot ML_v + \text{small-head})
  ]
  其中 (p) 是被判定“值得验证”的实例比例。
* 若 (p \in [0.3, 0.5])，常见会得到 **显著 cost 节约**。

---

### 5.7 至少 3 个强基线（含最难打败的）

1. **最难 baseline**：固定预算的 **Best-of-N + strongest cross-family verifier**

   * 这是最实在也最难打的线。
2. **轻量 verifier baseline**：`Reasoning with Confidence` / one-token verification 风格

   * 测你是否真的超过“便宜 verifier”。 ([开放评论][19])
3. **adaptive compute baseline**：ABF / SelfBudgeter / adaptive reasoning length

   * 测“是不是只要学会少想一点就够了”。 ([开放评论][68])
4. 备选补充 baseline：

   * self-consistency
   * PRM-guided reranking / tree search
   * verifier gain static threshold
   * abstention by raw confidence only

---

### 5.8 我预期会失败的 2 个设置（失败也有价值）

1. **极难题、低 coverage 区域**

   * 现象：generator 几乎不给出正确候选。
   * 结果：再聪明的 verifier 也救不了。
   * 价值：验证了 “coverage 是首要瓶颈” 这一理论分解。

2. **adversarially plausible wrong traces**

   * 现象：错误轨迹在表面风格上高度像正确轨迹，甚至共享正确中间结论。
   * 结果：detectability 会塌。
   * 价值：验证 blind-spot overlap / counterfactual hard negatives 是否真的必要。

---

### 5.9 Minimum Viable Experiments（至少 6 个，含可证伪预期）

1. **Accuracy–cost Pareto 主实验**

   * 比较 VISTA vs always-verify vs never-verify vs UHead vs adaptive-length。
   * 预期：VISTA 在相同准确率下降低 20–40% verify cost，或相同成本下提 2–4 点。
   * 若没有，则主张失败。

2. **Phase diagram 验证**

   * 横轴 coverage (q)，纵轴 detectability (\rho)。
   * 预期：gain 在低 (q) 和低 (\rho) 区域接近 0，在中间区域最高。
   * 若不出现这类结构，理论锚点要重写。

3. **same-family vs cross-family verifier**

   * 预期：cross-family 在高 overlap regime 显著更好。
   * 若没有，则 blind-spot 假设被削弱。

4. **hidden-state / black-box / verbal confidence 三类特征 ablation**

   * 预期：latent + disagreement > verbal confidence only。
   * 若 verbal confidence 已足够，方法 moat 会变弱。

5. **Counterfactual wrong-trace benchmark**

   * 构造“正确答案-错推理”“错答案-高似然风格”“局部符号错误”等类型。
   * 预期：旧 verifier 在某些错型上系统性失效，VISTA 的 detectability head 能更早识别。
   * 若旧 verifier 全部稳，则 benchmark 不够难。

6. **ablation：去掉 coverage / overlap / calibration**

   * 预期：

     * 去 coverage：在 hard region 过度验证；
     * 去 overlap：same-family 误判增多；
     * 去 calibration：risk-coverage 曲线变差。

7. **mechanistic probe**

   * 用 linear probe / CKA / clustering 看正确/错误轨迹在不同 regime 的 separability。
   * 预期：verification 有效的实例，其中高层表示分离度更高。

8. **OOD 泛化**

   * 在未见 generator family、未见 task 上测 gain prediction。
   * 预期：预测会掉，但仍优于 raw confidence；否则就是过拟合 verifier bank。

### 5.10 风险与备选路线（Plan B / C）

**主要风险**

* gain predictor 只在训练 domain 有效，跨域掉太多。
* 黑盒 API 模型拿不到 hidden states，导致 method 受限。
* counterfactual benchmark 构造不好，会被 reviewer 质疑“人造”。

**Plan B**

* 把主张收缩为：
  **“Verifiability phase diagram + selective verification policy”**
* 即便机制稍弱，依然可成一篇很强的 empirical+method paper。

**Plan C**

* 若路由策略效果一般，但 blind-spot 机制证据强：

  * 转成 **cross-family verification mechanism** paper。
* 若 benchmark 资产最强：

  * 转成 **CVBench/V-MAP benchmark paper + analysis**。

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

### 6.1 数据与基准选择（含理由）

**A. 可验证主战场**

* **数学**：MATH500 + 近期竞赛风格题目

  * 原因：有明确答案，适合 coverage / detectability 分解。
* **代码**：LiveCodeBench + DyCodeEval

  * 原因：可执行验证，且 contamination 处理更好。 ([开放评论][14])
* **规划/符号推理**：Game of 24 / Graph Coloring / STRIPS-like tasks / Reasoning Gym

  * 原因：难度可控、oracle 清晰、适合做 phase diagram。 ([开放评论][17])

**B. 支撑性扩展**

* **事实性 / 证据型 QA**（可选）

  * 用 retrieval-grounded exact evidence，测试“半可验证”场景。
  * 不作为第一主表，避免 judge 噪声冲淡主叙事。

**C. 自建 benchmark：V-MAP / CVBench**

* 每个问题配：

  * 1 个正确轨迹
  * 3–5 类 counterfactual wrong trace
  * 多 generator family 输出
  * exact / tool / judge label
* 目标：单独评估 verifier 的 detectability，而不是 solver accuracy。

**训练细节**

* Stage 2：提取 pooled hidden states + textual stats
* Stage 3：训练 coverage/detectability/overlap/calibration heads

### 6.3 评测协议

**主指标**

* Accuracy @ fixed cost
* Cost @ fixed accuracy
* Area under Accuracy–Cost Curve
* Risk–Coverage Curve（abstention）
* Gain prediction AUROC / Brier / calibration error

**鲁棒性**

* same-family vs cross-family
* easy / medium / hard difficulty bins
* with/without correct candidate
* adversarial wrong-trace type
* in-domain vs out-of-domain

**统计显著性**

* paired bootstrap CI
* paired permutation test
* difficulty-stratified reporting
* 每个表至少 3 seeds（head training）

**可解释性 / 机制验证**

* representation separability
* blind-spot overlap correlation
* layerwise probe
* counterfactual edit sensitivity

### 6.4 资源估算

**保守预算**

* 候选生成与 verifier 扫描：40–70 GPU-days
* hidden-state 抽取与缓存：10–20 GPU-days
* 轻量 heads：<5 GPU-days
* 可选 7B LoRA verifier：8–16 GPU-days
* 机制 probes + 全量 ablation：20–30 GPU-days

**总计**

* **约 80–120 GPU-days 上限**
* 在 **16×80GB A800** 下完全可控

**并行策略**

* 8 GPU 机器 A：vLLM / candidate generation / API orchestration
* 8 GPU 机器 B：LoRA、head 训练、mechanistic probes、ablation
* 存储：

  * 若只存 pooled features：几十 GB 到百 GB
  * 若存 step-level trace features：建议控制在 1–2 TB 内

### 6.7 如果结果不如预期，怎么迭代（debug roadmap）

**症状 A：VISTA 比 always-verify 还差**

* 看 coverage head 是否把“无正确候选”识别错了
* 若 coverage 差：加 candidate diversity / 多温度 / 多 generator

**症状 B：gain 预测相关性低**

* 看 detectability label 是否太噪
* 加 counterfactual negatives
* 用 pairwise rank label 替代 hard binary gain label

**症状 C：same vs cross-family 没差别**

* 看 family 选得是否太接近
* 加更异质的 generator / verifier
* 或转向 tool verifier vs model verifier 的对比

**症状 D：主表提升小，但 risk-coverage 很好**

* 把叙事从“更准”切到“**更可靠地知道何时不该答**”

**症状 E：机制 probe 没有清晰图**

* 不死磕 SAE
* 退回更稳的 linear separability / disagreement / overlap statistics

## 7. 写作框架（标题 / 摘要 / 图表 / related work）

### 7.1 标题候选 ×5

1. **VISTA: Verifiability-Informed Selective Test-Time Allocation for LLM Reasoning**
2. **Not Every Problem Should Be Verified: Learning When Verification Helps in LLM Reasoning**
3. **From Reasoning Scaling to Verifiability Scaling**
4. **Predicting Verifier Gain Before Spending Compute**
5. **Blind Spots, Not Just Confidence: Selective Verification for Reliable LLM Reasoning**

### 7.2 摘要骨架

* 句 1：最近 reasoning progress 强依赖 test-time compute 与 verifier，但收益高度不稳定。
* 句 2：我们提出一个统一视角：verification gain 由 coverage、detectability、blind-spot overlap 决定。
* 句 3：基于此，我们提出 VISTA，在花验证计算前预测收益并选择 verifier / abstain / tool-route。
* 句 4：我们同时引入 V-MAP/CVBench，把 solver 能力、verifier 能力、gain calibration 拆开评。
* 句 5：跨数学、代码、规划等可验证域，VISTA 在 accuracy–cost–risk Pareto 上优于强基线，并验证了若干关于 cross-family verification 与难度效应的可证伪预测。

### 7.3 贡献点 bullet

* 提出 **verifiability** 作为 solver–verifier 系统的核心状态变量
* 给出 **coverage × detectability × overlap** 的 gain 分解
* 设计 **VISTA**：实例级 selective verification policy
* 引入 **V-MAP/CVBench**：counterfactual verifiability benchmark
* 实证揭示 **cross-family blind spots** 与 verification usefulness 的关系
* 给出 **accuracy–cost–risk** 的统一评测协议

## 最后给一个明确结论

**我最推荐的主线是：`VISTA = verifiability-aware selective verification`。**

原因很简单：

* 它不是堆算力；
* 它有理论、有机制、有新评测；
* 它能在你的算力和 API 预算内做扎实；
* 它直接踩中 2025–2026 reasoning 研究里最关键、但还没有被统一解决的问题。

一句话总结这篇 paper 的最佳叙事：

**“下一代推理系统不该只是更会想，而该更会判断：什么时候值得验证，什么时候应该停下，什么时候根本不可验证。”**
