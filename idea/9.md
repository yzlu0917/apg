## 1. 方向地图（含主题覆盖）

### 1.1 我先给结论

* 在你给的算力约束下，**最值得冲 NeurIPS best-paper 叙事**的，不是“再训一个更大 reasoning model”，而是：

  * **把 test-time compute 从“盲目多想”变成“有价值地想”**；
  * **把可靠性从“答对率”推进到“何时该继续想 / 该检索 / 该验证 / 该 abstain”**；
  * **把评测从单一 final score 推进到 failure attribution + contamination-resistant evaluation**。
    这条线同时连接了 verifier reasoning、uncertainty/abstention、RAG sufficiency、agent/tool use、benchmark validity，且最近 12–18 个月文献已经出现明显“碎片化但可统一”的信号。([开放评论][1])

### 1.2 主题覆盖（10/10）

1. **推理与验证（reasoning / verifier / self-correction）**
   主线：PRM/PAV、生成式 verifier、self-correction RL、verifier-guided search 都在迅速推进。
   痛点：step-level supervision 很贵，verifier 常把“局部合理”误当“全局有益”，而且“process 是否天然优于 outcome”本身已被理论工作质疑。
   重要性：这是 reasoning 赛道的中心战场，也是最容易做出**可验证、可解释、可程序化评测**的地方。 ([开放评论][1])

2. **可靠性与对齐（honesty / calibration / uncertainty / abstention）**
   主线：不确定性估计、honesty alignment、abstention benchmark、early-warning probes。
   痛点：很多方法只能预测“现在像不像错”，却不能判断“再想一步会不会好”；更糟的是，CoT 有时反而会降低 abstention。
   重要性：这条线直接连接部署安全与人机协作可信度。 ([开放评论][2])

3. **表征与机制解释（mechanistic interpretability / representation）**
   主线：SAE 从玩具分析走向可扩展工具；“knowledge awareness”“concept atlas”“feature falsification”开始出现。
   痛点：feature explanation 容易自说自话，跨模型迁移与因果验证仍薄弱。
   重要性：这是少数能给 LLM 研究带来**机制叙事**、而不是只报分数的赛道。 ([开放评论][3])

4. **训练与优化新范式（RLHF/RLAIF alternatives / credit assignment）**
   主线：VinePPO、SPO、GRPO-λ、online preference optimization 都在解决 delayed reward / coarse credit 的问题。
   痛点：很多提升依赖更大 test-time compute 或更强 rollouts；且 offline vs online、process vs outcome 的差距未必像社区想象得那么本质。
   重要性：如果想做“非堆算力”的方法论文，这里最容易做出**原理 + ablation**。 ([开放评论][4])

5. **数据合成与评测（synthetic data / contamination / benchmark design）**
   主线：synthetic alignment data、contamination-limited benchmark、judge bias / preference leakage、construct validity 讨论正在升温。
   痛点：benchmark 越热越容易污染；LLM-as-judge 与 synthetic data generator 的耦合会系统性抬高结果。
   重要性：这是最适合做 **“重新定义问题 + 重塑评测标准”** 的方向，且 moat 很强。 ([开放评论][5])

6. **高效学习（parameter-efficient / modularity / compositionality）**
   主线：LoRA modularity、Self-MoE、composable adapters、组合泛化理论都在冒头。
   痛点：近期多数工作偏“有趣但不决定社区主线”；要冲 best paper，必须把它和 reasoning / reliability 绑定。
   重要性：它适合做系统性补强，但单独成线现在不如上面几条强。 ([开放评论][6])

7. **多智能体 / 工具使用（agents / tool use / planning）**
   主线：BFCL、ToolComp、OrchestrationBench、research agent / AutoML agent benchmark 都在上升。
   痛点：规划错误、执行错误、验证错误经常混在一起；多智能体系统增益经常不稳。
   重要性：工具环境天然有**可执行反馈**，非常适合做 failure attribution 和 exact evaluation。 ([开放评论][7])

8. **检索与记忆（RAG / long context / memory systems）**
   主线：Sufficient Context、memory agents、revisitable memory、RAG inference scaling。
   痛点：系统常常分不清“检索不够”还是“模型不会用”；长上下文更长不等于更会推理。
   重要性：这条线与 abstention、adaptive compute、agents 天然相连。 ([开放评论][8])

9. **结构化生成 / 程序归纳 / theorem proving**
   主线：formal verification、Lean-based theorem proving、verified code generation 正快速升级。
   痛点：formal gap 依然大，且很多强结果依赖专门 prover 或专有 reasoner。
   重要性：如果要做“强可验证 + 强机制 + 少 judge 偏差”的工作，这是黄金试验场。 ([开放评论][9])

10. **安全与红队（jailbreak / adversarial robustness）**
    主线：AgentHarm、JailbreakBench 系工作、RobustKV、refusal feature AT、TamperBench。
    痛点：adaptive attack 下许多 defense 会塌；安全分数经常依赖 prompt set。
    重要性：影响力大，但如果没有新机制或新评测，容易落入“又一个 defense”。 ([开放评论][10])

### 1.3 我建议重点盯住的 8 个方向

* 验证式推理 / self-correction
* 不确定性 / abstention / honesty
* 机制解释 + 可控干预
* 信用分配 RL / online preference optimization
* 评测设计 / 污染 / judge 偏差
* agents / tool use / planning
* RAG / 长上下文 / 记忆
* formal-verifiable reasoning / code

### 2.2 证据快照（支撑上表）

* **A. 验证式推理**：PAV 是 ICLR 2025 Spotlight；OmegaPRM 在检索快照里已显示高 citation velocity；SCoRe 也已进入高跟进密度状态，且 UHeads / ReVISE / backtracking verifier 等 follow-up 很密。 ([开放评论][1])
* **B. 不确定性 / abstention**：Alignment for Honesty 已有较高社区引用；AbstentionBench 发表仅数月已有明显关注；同时出现了 “CoT 反而伤 abstention” 与 “hidden-state probes 可提前预测失误” 两类反直觉结果。 ([开放评论][2])
* **C. 机制解释**：SAE scaling 论文 citation 很高；Entity-awareness 论文把 hallucination 与“我知不知道这个实体”机制连接起来；随后 Atlas-Alignment、feature falsification、CAFT 都在补“迁移性、可控性、可证伪性”。 ([开放评论][3])
* **D. RL / credit assignment**：VinePPO、SPO、GRPO-λ 都在集中打 credit assignment；但 ICML 2025 的理论工作指出 process 不一定比 outcome 更“统计上容易”，ICLR 2026 工作又发现 online/semi-online DPO 与 GRPO 差距可能没想象大。 ([开放评论][11])
* **E. 评测 / 污染**：LiveBench 强调 contamination-limited；Preference Leakage 直接指出 synthetic generator 与 judge 的 relatedness 会污染评测；NeurIPS 2025 的 construct validity 论文系统审查了 445 个 benchmark。 ([开放评论][12])
* **F. agents / tool use**：BFCL 已被作者描述为事实上的 function-calling 标准；ToolComp 开始引入 process supervision；MAST 则给出多智能体 failure taxonomy，说明社区已从“刷分”转向“拆失败原因”。 ([开放评论][7])
* **G. RAG / 记忆**：Sufficient Context 明确区分“context 不够”和“模型不会用”；ReMemR1 与 MemoryAgentBench 都在 ICLR 2026 把记忆从工程模块变成可分析对象。 ([开放评论][8])
* **H. formal-verifiable**：CLEVER 把 verified code generation 做成高质量 benchmark；HILBERT 展示 informal reasoning + formal verification 的强系统范式；Lean process-oracle 工作说明 formal verifier 可直接变成 dense reward。 ([开放评论][9])

## 3. 深入对比（每个方向：现状 → 瓶颈 → 争议 → 可突破点）

### 3.1 A. 验证式推理 / PRM / self-correction

**代表论文（6）**

* **Rewarding Progress / PAV**：把好 step 定义成“未来正确概率的增量”，而不是局部对错。
* **OmegaPRM**：用 MCTS 自动化采大规模 process labels。
* **THINKPRM**：让 verifier 自己生成验证轨迹。
* **SCoRe**：多轮 online RL 学 self-correction。
* **ReVISE**：test-time intrinsic self-verification/refinement。
* **Backtracking against In-Context Value Verifiers**：把 verifier 直接接进 search/backtracking。 ([开放评论][1])

**现状**

* 社区已经从“只看 final answer”转到“评估 reasoning process”。
* 但最强结果往往同时增加了 verifier、search、rollout 数，导致**方法改进**和**计算增加**纠缠在一起。 ([开放评论][1])

**主要瓶颈**

* process label 质量与规模很难兼得。
* verifier 常常只会判“这一步像不像合理”，不等于“这一步对最终成功是否真有用”。
* 跨任务泛化弱：math verifier 对 RAG / tool-use 不一定有效。 ([开放评论][1])

**争议 / negative results**

* 理论上，outcome supervision 在标准 coverage 假设下**不一定比 process supervision 更难**；说明当前经验优势可能来自算法设计，而非 supervision 形态本身。 ([开放评论][13])

**可突破点**

* 不再问“这一步对不对”，而问“**再走一步值不值**”。
* 这会把 verifier 从“trace grader”升级成“action-value estimator”。

---

### 3.2 B. 不确定性 / abstention / honesty

**代表论文（6）**

* **Uncertainty-based Abstention in LLMs**：经典不确定性→abstain。
* **Alignment for Honesty**：把 honesty 做成 benchmark + metric + training。
* **AbstentionBench**：系统性 benchmark。
* **Chain-of-Thought Degrades Abstention Unless Inverted**：指出 CoT 可伤 abstention。
* **Language Models Can Predict Their Own Behavior**：内部表示可提前预警 jailbreak / should-abstain 失误。
* **UHeads / Reasoning with Confidence**：<10M 轻量 uncertainty heads 可匹敌超大 PRM。 ([开放评论][14])

**现状**

* “会不会答错”已经不够，社区更关心“该不该答”“该不该拒绝”“该不该多想”。
* white-box hidden-state 信号开始显示出比 verbal confidence 更强的潜力。 ([开放评论][15])

**主要瓶颈**

* 现有方法大多只估计**当前正确性**，却不估计**额外计算的收益**。
* verbal confidence 在 adversarial / jailbreak 场景下很脆弱。 ([开放评论][16])

**争议 / negative results**

* CoT 不是总能提升可靠性；在 abstention 场景下甚至可能系统性变差。
* 这直接挑战了“多想总更好”的默认假设。 

**可突破点**

* 把 uncertainty 从“是否现在答”升级为“**继续想 / 检索 / 验证 / abstain 哪个动作最值**”。

---

### 3.3 C. 机制解释 + reasoning control

**代表论文（6）**

* **Scaling and Evaluating Sparse Autoencoders**：SAE 走向可扩展。
* **Do I Know This Entity?**：发现 hallucination 与 entity-recognition / self-knowledge feature 强相关。
* **Atlas-Alignment**：概念空间跨模型对齐。
* **CAFT**：用 concept ablation 控制 OOD generalization。
* **Revising and Falsifying SAE Feature Explanations**：把 feature 解释做成可证伪流程。
* **IPG**：用因果/梯度思路去定位 reasoning-critical components。 ([开放评论][3])

**现状**

* mech interp 不再只是“找到 feature 并命名”，而是在走向**迁移、控制、证伪**。
* 其中最有价值的不是“解释一个 attention head”，而是解释**可靠性边界**。 ([开放评论][17])

**主要瓶颈**

* feature 容易过拟合 prompt / dataset。
* 许多解释没有强 counterfactual intervention。
* 跨模型迁移仍不稳。 ([开放评论][18])

**争议**

* “feature 真的是单语义原子吗？”仍有分歧。
* 因此最近开始出现“falsify explanation”而不是“just describe it”。 ([开放评论][18])

**可突破点**

* 找到**recoverability / marginal-value**相关特征，并证明它们可因果控制推理策略，而不只是控制 surface style。

---

### 3.4 D. 信用分配 RL / preference optimization

**代表论文（6）**

* **VinePPO**：MC-based credit assignment，绕过 value net 弱点。
* **SPO**：segment-level credit assignment。
* **GRPO-λ**：把 eligibility traces 搬到 reasoning RL。
* **COPO**：count-based online preference optimization。
* **Bridging Offline and Online RL for LLMs**：online/semi-online DPO 与 GRPO 的系统比较。
* **Min-Form Credit Assignment Is All PRM Needs**：试图统一 process reward 与优化目标。 ([开放评论][11])

**现状**

* 社区共识是：**credit assignment 是 reasoning RL 的核心瓶颈**。
* 但当前大多在 sequence / segment 层次做 credit，而不是在“动作值”层次。 ([开放评论][4])

**主要瓶颈**

* 真正有效的方法通常仍需很多 rollouts。
* 强 online RL 与强 teacher rollout 很容易把 paper 变成资源竞赛。 ([开放评论][11])

**争议**

* process vs outcome 的优势到底来自 supervision 形态，还是来自更好的 optimization / rollout？
* offline vs online 的差距也未必绝对。 ([开放评论][13])

**可突破点**

* 如果 test-time 动作价值能被直接估计，很多 full-RL 的难点可以先被“轻量 controller”绕开。

---

### 3.5 E. 评测设计 / contamination / judge bias

**代表论文（7）**

* **LiveBench**：contamination-limited、持续更新。
* **Preference Leakage**：judge 与 synthetic generator 的 relatedness 会污染评测。
* **Construct Validity in LLM Benchmarks**：系统审查 445 benchmark。
* **FuncBenchGen**：controllable、contamination-free hidden-DAG function-calling eval。
* **ESDAE**：评估 synthetic agent benchmark 是否保留真实排名。
* **Magpie**：alignment data synthesis from scratch。
* **CoDeC**：用 in-context learning 方式检测 contamination。 ([开放评论][12])

**现状**

* 社区已经意识到：benchmark 不只是“题库”，而是**研究方向调度器**。
* 现在最稀缺的不是更多题，而是更强的**construct validity、freshness、controllability、failure attribution**。 ([开放评论][19])

**主要瓶颈**

* judge bias、数据污染、prompt overfitting。
* synthetic benchmark 若不保 rank-preservation，很容易“看上去可控，实际没信号”。 ([开放评论][20])

**争议**

* contamination 有时会“被训练后遗忘”，这又让污染检测更难。 ([开放评论][21])

**可突破点**

* 做一个**可恢复性 / sufficiency / compute-value**都可标注的可执行 benchmark，而不是只给 final EM/F1。

---

### 3.6 F. agents / tool use / planning

**代表论文（6）**

* **BFCL**：function calling 事实标准。
* **ToolComp**：多工具 reasoning + process supervision。
* **OrchestrationBench**：把 planning 与 execution 拆开评。
* **ML-Tool-Bench**：更复杂的 ML task tool-use。
* **BrowseComp-Plus**：更公平透明的深度检索 agent 评测。
* **Why Do Multi-Agent Systems Fail? / MAST**：多智能体 failure taxonomy。 ([开放评论][7])

**现状**

* agent paper 正在从“展示案例”转向“拆解 planning / memory / verification / coordination”。
* 这是好现象，说明方向开始成熟。 ([开放评论][7])

**主要瓶颈**

* final success 不能告诉你究竟是 plan 错、call 错、memory 丢、还是 verifier 没抓住。
* 多智能体常常只是把错误复制几遍。 ([开放评论][22])

**争议**

* MAS 是否真带来稳定增益？很多结果并不稳。 ([开放评论][22])

**可突破点**

* 把 tool-use 看作“离散动作空间上的边际价值选择”，再配可执行 hidden-DAG oracle。

---

### 3.7 G. RAG / long context / memory

**代表论文（7）**

* **Sufficient Context**：把“信息够不够”与“模型会不会用”分开。
* **Sparse RAG**：并行编码/稀疏化降成本。
* **RetrievalAttention**：长上下文推理时的训练无关加速。
* **HawkRAG**：全局 memory + retrieval。
* **ReMemR1**：revisitable memory。
* **MemoryAgentBench**：四类 memory competence 系统评测。
* **Inference Scaling for Long-Context RAG**：研究 test-time scaling 对 RAG 的作用。 ([开放评论][8])

**现状**

* 社区已经不满足于“更长 context window”，而是在讨论**什么时候该检索、什么时候该记忆、什么时候该回看**。 ([开放评论][23])

**主要瓶颈**

* 现有系统很少明确建模“再检索一次是否值得”。
* memory 与 RAG 的收益常混在一起。 ([开放评论][8])

**争议**

* 大模型在 sufficient context 上通常能答好，但在 insufficient context 上常常不 abstain；小模型则会在 sufficient context 下也乱答/乱拒。 ([开放评论][8])

**可突破点**

* 用 recoverability / marginal-value 信号来统一 `answer vs retrieve vs abstain`。

---

### 3.8 H. formal-verifiable code / theorem proving

**代表论文（7）**

* **CLEVER**：curated verified code generation benchmark。
* **DeepSeek-Prover-V1.5**：proof assistant feedback + RL。
* **HILBERT**：informal reasoner + prover + verifier + retriever。
* **Process-Verified RL via Lean**：Lean 直接给 tactic-level dense reward。
* **Formal Theorem Proving by Rewarding LLMs**：formal reward + lemma generation。
* **LeanAgent**：lifelong theorem proving。
* **InvBench / Vericoding**：把 verifier-based program verification 做成评测任务。 ([开放评论][9])

**现状**

* 这是目前最干净的“reasoning with exact feedback”试验场之一。
* symbolic verifier 让很多 judge bias 问题直接消失。 ([开放评论][24])

**主要瓶颈**

* general LLM 的 informal reasoning 很强，但 formal completion 仍弱。
* 很强结果常依赖专门 prover 或强 API reasoner。 ([开放评论][25])

**争议**

* 领域窄，是否能外推到 general reasoning？
* 但从研究设计角度，它反而提供了更可靠的 falsifiable setting。 ([开放评论][25])

**可突破点**

* 把 theorem proving 里的“symbolic process oracle”迁移到 tool-use / RAG 这类更泛化的可执行环境。

## 4. 候选课题（含 gap 工具箱）

### 4.0 我显式用了 6 种 gap 挖掘策略

1. **Failure modes map**

   * 找到的核心 failure：

     * `F1` 低置信但**可恢复**；
     * `F2` 低置信且**不可恢复**；
     * `F3` context/tool state **本身不足**；
     * `F4` verifier / agent 只是“多花算力”并没增加成功率。
   * 缺口：现有工作大多只区分“对/错”，很少区分“**现在错但再做一步值不值**”。
   * 这直接导出主推 idea。 ([开放评论][8])

2. **Assumption stress test**

   * 被击穿的隐含假设：

     * “process supervision 天然优于 outcome”；
     * “更多 CoT 总更好”；
     * “confidence 足以决定 abstain”；
     * “LLM-as-judge 与 synthetic generator 可独立看待”。
   * 缺口：需要一个比 correctness/confidence 更贴近决策的量。 ([开放评论][13])

3. **Counterfactual baseline**

   * 发现很多 paper 没有做**matched-cost**比较：方法增益可能只是来自更多 token / 更多 samples / 更多 retrieve。
   * 缺口：需要“同预算下，哪个动作最有收益”的评测。 ([开放评论][26])

4. **Unifying principle**

   * PRM、abstention、RAG sufficiency、agent planning，其实都可统一成：
     **估计额外动作的边际价值（marginal value of extra computation/evidence）**。
   * 这是我认为最强的统一支点。 ([开放评论][1])

5. **New eval lens**

   * 新评测维度不该只有 accuracy，而要有：

     * recoverable / irrecoverable split
     * missed opportunity rate
     * wasted compute rate
     * utility-under-budget frontier
   * 现有 benchmark 很少系统提供这些标签。 ([开放评论][19])

6. **Scaling / complexity lens**

   * UHeads 这类 <10M head 已能匹敌乃至超过大 PRM；说明 under our compute，**轻量 introspection**比“训更大 verifier”更有性价比与 moat。 ([开放评论][27])

### 4.2 每个候选课题的 best-paper 潜力判断

#### 课题 1：**MATE：边际价值驱动的自适应推理/检索/验证控制器**

* **核心问题**：现在的 LLM controller 多在问“我现在对吗？”，但真正该问的是“**再做一步值吗？**”
* **关键 gap**：没人把 PRM、abstention、RAG sufficiency、agent action selection 统一成**动作边际收益估计**。
* **best-paper 支点**：

  * 新机制：recoverability / marginal-value
  * 新评测：recoverable-vs-irrecoverable
  * 轻理论：confidence-only policy 的不足
* **为什么现在做合适**：近期已经同时出现 PAV、UHeads、Sufficient Context、CoT hurts abstention、Preference Leakage 这些“拼图”。([开放评论][1])

#### 课题 2：**RecoverabilityBench**

* **核心问题**：现有 benchmark 只看 final score，无法评估是否“白花算力”或“错失可恢复机会”。
* **关键 gap**：没有一个 contamination-resistant、可执行、带 sufficiency/recoverability 标签的跨域 benchmark。
* **best-paper 支点**：新的评测范式本身就可能改写赛道。
* **为什么现在做合适**：construct validity 与 judge leakage 已成为社区显式焦虑。 ([开放评论][19])

#### 课题 3：**Recoverability Subspace**

* **核心问题**：模型内部是否编码了“再想一步会不会变好”的特征？
* **关键 gap**：已有工作解释 confidence / knowledge boundary，但几乎没人解释 recoverability。
* **best-paper 支点**：机制解释 + 因果干预。
* **为什么现在做合适**：SAE 工具链成熟，且社区开始重视 falsification。 ([开放评论][3])

#### 课题 4：**Symbolic Process Oracles for Tool Agents**

* **核心问题**：tool-use agent 能否像 theorem prover 一样拿到 exact step-level reward？
* **关键 gap**：tool benchmarks 仍大量依赖 final state 或 LLM judge。
* **best-paper 支点**：把 symbolic verification 从 formal domain 推到 agents。
* **为什么现在做合适**：BFCL / ToolComp / FuncBenchGen / MAST 已把问题暴露得很清楚。 ([开放评论][7])

#### 课题 5：**Counterfactual Segment RL**

* **核心问题**：segment-level RL 能否用 counterfactual minimal edit 来做更真因果的 credit assignment？
* **关键 gap**：现有 segment 仍更多是工程切段。
* **best-paper 支点**：算法创新。
* **为什么现在做合适**：SPO / VinePPO / GRPO-λ 热，但也意味着更拥挤。 ([开放评论][4])

## 5. 主推 idea（完整研究蓝图）

### 5.0 选择前，我先回答 4 个关键问题

**Q1. 最接近的 3 篇 / 3 类工作是谁？我们到底多了什么？**

* **类 1：PAV / PRM / verifier work**
  他们估计“step 是否好”，我们估计“**额外动作的收益**”；他们主要服务 rerank / search，我们做统一 action selection。 ([开放评论][1])
* **类 2：uncertainty / abstention / UHeads / LM behavior probes**
  他们估计“现在是否该答/拒”，我们估计“**继续想 / 检索 / 验证 / abstain 哪个最优**”；他们通常不建模 recoverability。 ([开放评论][27])
* **类 3：Sufficient Context / RAG selective generation**
  他们讨论“context 是否足够”，但主要是 RAG 静态情形；我们把它扩展到 reasoning、tool-use、verification，并引入 cost-aware controller。 ([开放评论][8])

**Q2. 这个贡献属于什么类型？为什么够强？**

* **组合型贡献**：`新机制 + 新评测 + 新算法 + 轻理论`
* 强点在于：不是“又一个更强 prompt loop”，而是提出一个更合理的**决策变量**：`marginal action value`。

**Q3. 在我们算力约束下，2–3 周最小闭环是什么？**

* 第 1 周：做 synthetic tool-DAG + synthetic sufficiency-QA 两个环境；跑 fixed-budget baseline。
* 第 2 周：收集 2–5 万个 state-action counterfactual 标签；训练 5–10M 的 benefit head。
* 第 3 周：验证一个关键命题：
  **在相同当前 confidence 桶内，benefit head 能否区分 recoverable vs irrecoverable？**
* 如果这个命题不成立，就不继续扩展到多域。

**Q4. best paper 级叙事是什么？**

* **“LLM 的 test-time scaling 不该再是盲目多想；真正缺失的量，是额外一步推理/检索/验证的边际价值。”**

---

### 5.1 一句话主张（thesis claim）

**MATE**（**M**arginal **A**ction-value of **T**hinking and **E**vidence）主张：

> 对可靠的 LLM 推理，关键不是估计“我现在对不对”，而是估计“再做一步是否值得”；这个边际收益信号可由隐藏状态与少量 counterfactual rollouts 学出，并统一驱动 thinking / retrieval / verification / abstention。

### 5.2 Novelty checklist（至少 5 条）

1. **新决策变量**：从 `correctness/confidence` 改成 `marginal action value`。
2. **统一 4 类动作**：`answer / think / retrieve / verify / abstain` 在同一 action-value 框架下比较。
3. **新监督方式**：用 counterfactual rollout 标注“动作后成功率”，而不是只标 answer correctness。
4. **新评测透镜**：recoverable-vs-irrecoverable、wasted-compute、missed-opportunity、utility frontier。
5. **强机制验证**：检验 recoverability 是否存在低维 hidden-state subspace，并做因果干预。
6. **强对比于最近 3 类工作**

   * vs **PRM/PAV**：我们预测动作收益，不只是 step 好坏。
   * vs **UHeads / abstention**：我们解决“该不该继续算”而不只是“该不该答”。
   * vs **Sufficient Context**：我们做动态 action policy，而不只是静态 sufficiency 诊断。 ([开放评论][1])

### 5.3 核心洞察：为什么这不是工程堆砌

* 很多失败其实有 3 种状态：

  * **S1**：现在已经能答对
  * **S2**：现在答不对，但**再做一步有救**
  * **S3**：现在答不对，而且**再做一步也没救**
* 现有 confidence / abstention 方法常把 `S2` 和 `S3` 混成“低置信”。
* 现有 PRM 常把 `局部看起来对` 和 `真的值得继续` 混为一谈。
* 我们的视角把这些分开：
  **边际收益**才是 inference-time action selection 的充分统计量；confidence 不是。
* 这也自然解释了近期的矛盾结果：

  * 为什么 verifier 有时有效；
  * 为什么 CoT 有时伤 abstention；
  * 为什么 test-time scaling 有时只是在增加 abstain 或浪费 token。 ([开放评论][1])

### 5.4 方法定义（伪代码级）

#### 5.4.1 状态、动作、效用

* 状态 `s_t = (x, y_<=t, c_t, z_t, b_t)`

  * `x`: 原始问题
  * `y_<=t`: 当前 reasoning / draft / tool trace
  * `c_t`: 当前上下文/检索结果
  * `z_t`: 工具或环境状态
  * `b_t`: 剩余预算（tokens / tool calls / latency）

* 动作集合 `A(s_t)`

  * `ANSWER`
  * `THINK(Δtok)`
  * `RETRIEVE(k)`
  * `VERIFY(1-step critique+revise)`
  * `ABSTAIN`

* 目标效用
  [
  U = \mathbb{1}[\text{correct}] - \lambda_{tok} \cdot \text{tokens} - \lambda_{tool} \cdot \text{toolcalls} - \lambda_{hall}\cdot\mathbb{1}[\text{wrong and answered}] - \lambda_{abs}\cdot\mathbb{1}[\text{abstain on answerable}]
  ]

#### 5.4.2 训练数据构造（counterfactual labels）

对每个状态 `s_t`：

1. 记录 backbone hidden state `h_t`
2. 对每个动作 `a ∈ A(s_t)`：

   * 固定预算与解码设定
   * 跑 `K` 个 counterfactual continuation
   * 用 exact checker / gold / executable verifier 判是否成功
3. 得到

   * `p_a = P(success | s_t, a)`
   * `u_a = p_a - cost(a)`
4. 定义 recoverability
   [
   r(s_t)=\max_{a\neq ANSWER,ABSTAIN} u_a-u_{ANSWER}
   ]

#### 5.4.3 轻量控制头（Benefit Head）

* 输入：`h_t` + meta features

  * 当前 answer entropy / logprob
  * 剩余预算
  * 当前检索深度 / tool depth
  * prefix 长度
* 输出：

  * `\hat p_a` for each action
  * optional `\hat σ_now`: current sufficiency
  * optional `\hat r`: recoverability score

#### 5.4.4 损失

[
L = \sum_a \text{BCE}(\hat p_a, p_a)

* \alpha \cdot L_{\text{rank}}
* \beta \cdot L_{\text{calib}}
* \gamma \cdot L_{\text{suff}}
  ]

- `L_rank`：更优动作排在更前
- `L_calib`：Brier / ECE regularization
- `L_suff`：如果环境有 exact sufficiency label，就做辅助任务

#### 5.4.5 推理策略

```text
while budget remains:
    compute hidden state h_t
    score all valid actions with Benefit Head
    utility_hat(a) = p_hat(a) - cost(a)
    if max_a utility_hat(a) is ANSWER:
        return answer
    if max_a utility_hat(a) < tau and p_hat(ANSWER) < tau_ans:
        return ABSTAIN
    execute argmax_a utility_hat(a)
```

### 5.5 关键超参、训练细节、复杂度

**默认实现**

* Backbone：两个 7B–8B 开源 backbone

  * 一个通用 instruct
  * 一个 reasoning-tuned
* Benefit Head：2-layer MLP 或小 Transformer head，参数量 **5M–15M**
* 采样点：

  * reasoning domain：每 32 或 64 token 一个 decision point
  * tool / retrieval domain：每次 tool call / retrieval batch 后一个 decision point
* Counterfactual rollout：

  * `K=2~4`
  * `THINK`: +64 / +128 token
  * `RETRIEVE`: top-4 docs or 1 DAG expansion
  * `VERIFY`: critique + revise 一轮
* 损失权重：`α=0.5, β=0.1, γ=0.25` 起步
* 优化器：AdamW, lr `2e-4`, batch `2k~8k states`
* 校准：留 10% calibration split 做 conformal/risk calibration

**复杂度**

* 数据收集：`O(N · |A| · K · L)`
* 头训练：近似 `O(N·d·|A|)`，可忽略
* 推理开销：头部额外成本几乎忽略不计；主要成本来自 controller 选择去执行的动作

**工程上限**

* 存储 pooled hidden state，而不是整段激活；200k states 量级的多层 pooled features 存储可控。
* 不需要 full-model RL 才能出论文主结果；controller 本身就能形成闭环。

### 5.6 至少 3 个强基线（含最难打的）

1. **Confidence-only controller**

   * entropy / semantic entropy / verbal confidence / selective prediction
   * 动作规则：低置信→abstain 或 low置信→多想一次
2. **Verifier-only controller**

   * PAV / PRM / UHead-style step verifier
   * 动作规则：低 verifier score→revise / rerank
3. **Fixed-budget scaling baseline**

   * self-consistency / ReVISE / retrieve-more / verify-once
   * 关键：**严格 matched compute**
4. **最难打的 baseline**

   * `matched-cost self-consistency + verifier-triggered revise`
   * 原因：它几乎集合了当前主流最强实用手段

### 5.7 Minimum viable experiments（至少 6 个，且可证伪）

1. **E1. 动作收益预测是否优于 correctness/confidence 预测**

   * 指标：AUROC / AUPRC for “extra action profitable?”
   * 预期：Benefit Head 显著优于 entropy / verbal confidence
   * 可证伪：若两者无差异，主张基本失败

2. **E2. 相同 compute 预算下，controller 是否支配 fixed scaling**

   * 任务：math、synthetic RAG、tool-DAG
   * 指标：utility frontier、accuracy、hallucination、latency
   * 预期：MATE 曲线更优
   * 可证伪：若只在高预算赢、低预算不赢，则叙事变弱

3. **E3. 在相同当前 confidence 桶内，是否仍能分出 recoverable vs irrecoverable**

   * 这是最关键实验
   * 预期：Benefit Head 在 matched-confidence bucket 内仍有明显区分力
   * 可证伪：若不能，说明 marginal-value 不可从 hidden state 可靠恢复

4. **E4. sufficiency/recoverability 标签是否解释现有矛盾结果**

   * 比较：

     * context sufficient vs insufficient
     * CoT / no-CoT
     * verify / no-verify
   * 预期：

     * sufficient但低当前正确率 → think/verify 可能有效
     * insufficient → retrieve 或 abstain 才有效

5. **E5. 机制实验：recoverability 信号位于哪些层？**

   * 做 layer-wise probe；再对 top features 做 causal ablation/steering
   * 预期：后中层到末层出现稳定 recoverability signal；其子空间与 knowledge-awareness 有交集但不重合
   * 可证伪：若信号到处都一样弱，则机制故事不足

6. **E6. 失败归因指标**

   * 指标：wasted-compute、missed-opportunity、wrong-answer-when-irrecovable
   * 预期：MATE 明显降低 wasted-compute 与 missed-opportunity
   * 可证伪：若只提升 accuracy，不改善 failure attribution，就不够 best-paper

7. **E7. 跨域迁移**

   * 只在 synthetic tool + QA 训头，迁移到 math 或另一 backbone
   * 预期：至少保留一部分收益
   * 可证伪：若完全不迁移，需收窄主张

8. **E8. 评测污染鲁棒性**

   * 主结果只用 executable / exact metrics，不用 LLM judge
   * 外部 benchmark 仅做补充
   * 预期：和 judge-based ranking 不完全一致，反而证明新评测有价值

### 5.8 我预期会失败的 2 种设置

1. **纯开放式创作 / 非可验证 instruction-following**

   * 因为 counterfactual label 太噪
   * 失败价值：说明本方法依赖可验证或高质量 reward 的环境

2. **完全黑盒 API-only 模型**

   * 拿不到 hidden state，Benefit Head 版本受限
   * 失败价值：说明白盒 introspection 是关键资源；可发展 verbal self-estimated variant 作为弱版

### 5.9 潜在风险与 Plan B / C

**风险 1：recoverability 信号太弱**

* **Plan B**：加 cheap lookahead feature

  * 例如让模型先做 1 个 16-token scout thought，再估动作价值
* **Plan C**：把工作收窄为 tool/RAG 可执行域，主打 benchmark + controller

**风险 2：标签太噪**

* **Plan B**：做 pairwise ranking 而非绝对回归
* **Plan C**：只保留 exact-verifiable 任务作为主结果

**风险 3：只在 synthetic 数据好**

* **Plan B**：加入 LiveBench/ToolComp/BFCL/MDBench 风格外测
* **Plan C**：把论文定位成“principled evaluation + adaptive controller”，而不是“universal LLM policy”

### 5.10 诊断树（debug roadmap）

* **Step 1：先看 oracle upper bound**

  * 若 oracle-best-action 相比 fixed policy 也没提升
    → 任务本身不适合“adaptive compute”
* **Step 2：若 upper bound 有，但模型没学到**

  * 看 action-profit AUROC
  * 低 → hidden features 不够 / 标签噪声大
* **Step 3：若 AUROC 好，但主结果差**

  * 看 calibration 与 threshold
  * 调 conformal / utility weights
* **Step 4：若某域有效某域无效**

  * 看是 action space mismatch，还是 sufficiency labeling 有问题
* **Step 5：若只靠更多 compute 才赢**

  * 强化 matched-cost protocol；砍掉会偷预算的组件

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

### 6.1 数据与基准选择

**主训练/主评测：3 个域**

1. **Synthetic Tool-DAG**

   * hidden-DAG function dependency
   * exact answer / exact process validity
   * 作用：最干净地测 `answer vs tool/retrieve vs abstain`

2. **Synthetic Sufficiency-QA**

   * 从程序生成知识图谱 / 文档集
   * 可精确控制：hop 数、distractor 数、是否缺关键事实
   * 作用：测 `answer vs retrieve vs abstain`

3. **Math / symbolic reasoning**

   * 精确答案检查
   * 作用：测 `answer vs think vs verify`

**外部有效性评测**

* BFCL / ToolComp 风格子集
* 一个 contamination-limited QA / RAG 集
* 一个 long-context / memory 子集
  这些选择的理由是：都能较少依赖 LLM-as-judge。 ([开放评论][7])

### 6.2 周计划（8 周）

**Week 1**

* 搭 synthetic Tool-DAG generator
* 搭 synthetic Sufficiency-QA generator
* 跑 fixed-budget baselines
* 定义 utility / failure attribution 指标

**Week 2**

* 采第一批 20k–50k states
* 训练 Benefit Head v0
* 做 E1 / E3（go/no-go）

**Week 3**

* 加入 math 域
* 做 matched-cost self-consistency / verifier baselines
* 若 E3 不成立，立即 pivot 到 benchmark 方向

**Week 4**

* 扩大到 100k–200k states
* 做 E2 / E4 / E6
* 固化 action space 与 threshold

**Week 5**

* 跨 backbone 迁移
* 外测到 tool/RAG 公开 benchmark 子集
* 跑 oracle upper bound

**Week 6**

* 机制分析：layer-wise probes
* 若信号明确，再做单层 SAE / sparse feature 分析

**Week 7**

* 理论部分：

  * confidence-only suboptimality 命题
  * greedy marginal-value regret bound
* 全部 ablation 与显著性检验

**Week 8**

* 写作
* 补实验
* 整理 release：generator、config、pooled states schema、eval harness

### 6.3 资源估算

**MVP**

* 数据收集：约 **20–35 GPU-days**
* Head 训练：**<2 GPU-days**
* 基线与评测：**15–25 GPU-days**
* 合计：**40–60 GPU-days**

**完整论文版**

* 加跨域、跨 backbone、机制解释与更多 seeds：
  **80–120 GPU-days 上限**

**显存与并行**

* 7B/8B backbone inference + hidden-state extraction：单卡 80GB 足够
* rollout generation：8 卡并行最好
* 另一台机器同时跑 baseline / evaluation
* 如果做 SAE，只在 1–2 层做，不做全模型全层

### 6.4 评测协议

**主指标**

* utility under budget
* exact accuracy / EM
* hallucination / wrong-answer rate
* abstention precision / recall
* avg tokens / tool calls / latency

**鲁棒性**

* prompt paraphrase
* distractor 增加
* context insufficiency
* harder-hops / longer dependency chains
* budget 缩放曲线

## 7. 写作框架（标题 / 摘要 / 图表 / related work）

### 7.1 标题候选 ×5

1. **Beyond Confidence: Estimating the Marginal Value of Thinking, Retrieval, and Verification in Language Models**
2. **Reason, Retrieve, or Refrain: Counterfactual Action-Value Estimation for Reliable LLM Inference**
3. **When Does More Compute Help? Hidden-State Benefit Models for Adaptive LLM Reasoning**
4. **Compute Is a Choice: A Unified Controller for Thinking, Retrieval, and Abstention**
5. **From Correctness to Recoverability: Action-Value Estimation for Reliable Test-Time Scaling**

### 7.2 摘要骨架

* **问题**：现有 verifier、abstention、RAG controller 都在用错误的决策变量，只判断“现在对不对/够不够自信”。
* **核心想法**：定义并学习额外动作的边际收益；用 hidden states + counterfactual rollouts 估动作值。
* **方法**：MATE，统一比较 answer / think / retrieve / verify / abstain。
* **结果**：在 math、synthetic RAG、tool-DAG 上，以相同预算支配 fixed-budget 与 confidence/verifier baselines。
* **分析**：recoverability 信号可从隐藏状态读取，并解释近期关于 CoT、abstention、test-time scaling 的矛盾现象。
* **意义**：把 test-time scaling 从“多想一点”变成“有价值地想一点”。

### 7.3 贡献点 bullet

* 提出 **marginal action value** 作为 test-time inference 的核心决策变量
* 用 counterfactual rollouts 学轻量 Benefit Head
* 提出 recoverability-aware 的新评测协议
* 在 matched-cost 下统一比较 think / retrieve / verify / abstain
* 给出机制分析与轻理论，解释为什么 confidence-only 不够
