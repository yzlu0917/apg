## 1. 方向地图

> 覆盖主题空间：1–10 全覆盖（推理验证、可靠性对齐、机制解释、训练优化、数据与评测、高效学习、Agents、检索记忆、结构化生成/形式化、安全红队）。

### 1.1 推理与验证 Reasoning / Verifier / Self-correction

* 主线：**RLVR / Verifier(PRM/ORM) 驱动的训练** + **test-time scaling（采样/搜索/重排序）** + **自我修正/自检**。SCoRe、RL Tango 等把“生成器—验证器—强化学习”串成闭环。 ([OpenReview][1])
* 痛点：验证器**可被 reward hacking**、**校准差**、以及“看起来像推理”的**伪迹**误导；同时有强负面结果指出 RLVR 更多是“采样效率提升”，未必扩展真正推理能力边界。 ([arXiv][2])
* 重要性：这是当下 LLM 迈向“可验证智能”的核心栈，且具备**强可证伪**（准确率/校准/鲁棒性/攻击成功率）与可复现（可验证任务）优势。

### 1.2 可靠性与对齐 Robustness / Honesty / Calibration / Uncertainty

* 主线：从“会答”走向“知道自己何时不该答/如何表达不确定性/如何保持诚实”。包括不确定性估计、校准、幻觉评测、诚实性基准与分解（honesty vs accuracy）。 ([NeurIPS][3])
* 痛点：开放域事实性难有确定标签；评测依赖 judge model 时**评估偏差**与可复现性挑战；CoT 甚至可能遮蔽幻觉线索。 ([ACL Anthology][4])
* 重要性：对齐与可靠性是 NeurIPS/ICLR 长期主航道，且非常适合做**机制解释或理论保证**（如 conformal 风格有限样本保证）。

### 1.3 表征与机制解释 Mechanistic Interpretability / Representation

* 主线：以 **Sparse Autoencoder(SAE)**、电路发现、特征可迁移/可证伪为核心，推动“可解释单元/特征”的系统化评测与理论。 ([ICLR 会议录][5])
* 痛点：SAE 特征**是否“规范/唯一”**存在争议；“解释是否等于有用”也有系统性质疑；电路发现存在复杂度瓶颈。 ([ICLR 会议录][6])
* 重要性：能产出 **best-paper 级机制洞察**（why it works/why it fails），但评测与结论边界要非常严谨。

### 1.4 训练与优化新范式 RLHF/RLAIF Alternatives / Credit Assignment

* 主线：DPO 系列、偏好学习的分布鲁棒与漂移、生成器—奖励模型的协同训练、以及“让 reward model 会推理”。 ([ICLR][7])
* 痛点：偏好数据质量与漂移、奖励模型误设（reward misspecification）、以及“训练稳定性—泛化—可控性”三者难兼。 ([NeurIPS][8])
* 重要性：不需要堆预训练算力即可做出可验证贡献（算法/理论/评测），但要防“工程调参论文”。

### 1.5 数据合成与评测 Synthetic Data / Benchmark / Contamination

* 主线：动态/无污染评测（LiveCodeBench）、数据污染因果量化、数据归因评测（DATE-LM）、合成数据有效性与“collapse”理论。 ([OpenReview][9])
* 痛点：污染检测与防护仍不统一；benchmark construct validity（测到的到底是什么）越来越关键；合成数据可能导致分布退化。 ([OpenReview][10])
* 重要性：最适合做 **NeurIPS best-paper 风格**：新基准/新评测视角 + 深分析，且可复现壁垒强。

### 1.6 高效学习 PEFT / Modularity / Compression

* 主线：LoRA 模块化组合（LoRA-LEGO）、更细粒度低秩（GraLoRA）、初始化与压缩校准数据选择等。 ([OpenReview][11])
* 痛点：模块组合的“语义可组合性”缺乏理论；压缩/量化的校准协议不统一。 ([ACL Anthology][12])
* 重要性：算力友好，容易做系统性 ablation，但要避免“又一个 LoRA 变体”的弱叙事。

### 1.7 多智能体/工具使用 Agents / Tool Use / Planning / Security

* 主线：SWE-bench 类软件工程闭环、Web/GUI agents、multi-agent 协作与不确定性协同、以及 prompt injection 安全基准（WASP/WebInject）。 ([SWE-bench][13])
* 痛点：评测难（环境随机性、不可复现）、安全面临“间接提示注入”；工具链越强攻击面越大。 ([NeurIPS][14])
* 重要性：强社区热度 + 强应用牵引，适合做“新评测 + 新机制防御/验证”。

### 1.8 检索与记忆 RAG / Long-Context / Memory

* 主线：更好的 context selection（影响函数视角）、多粒度检索聚合、参数化记忆解码器（Memory Decoder）、长上下文评测与 token 压缩。 ([OpenReview][15])
* 痛点：RAG 的瓶颈往往不在 retriever，而在“证据是否足够/是否冲突/是否被正确使用”；长上下文成本与鲁棒性。 ([OpenReview][16])
* 重要性：能做可复现 benchmark，但要避免“拼组件”而缺少原则。

### 1.9 结构化生成/程序归纳 Code / Formal Methods / Theorem Proving

* 主线：Lean 等形式系统中的多轮证明、统一评测框架（ProofGym）、更强的形式化基准（VeriBench-FTP）、以及 RLVR 扩展到更广领域。 ([NeurIPS][17])
* 痛点：形式化任务的可扩展数据与交互协议；跨系统/跨任务泛化仍弱。 ([NeurIPS][18])
* 重要性：**强可验证、强可复现**，非常适合做“机制+理论+严格实验”。

### 1.10 安全与红队 Jailbreak / Adversarial Robustness

* 主线：开放基准（JailbreakBench）、更强攻击（weak-to-strong、诗歌 jailbreak）、以及防御与安全评测规范化。 ([OpenReview][19])
* 痛点：攻击/防御军备竞赛；评测协议不一致导致“看起来提升”的假象。JailbreakBench 明确推动标准化。 ([arXiv][20])
* 重要性：安全方向需要更强的“新评测视角/可复现协议”，是 best paper 常见落点。

## 3. 深入对比

### 3.1 推理与验证方向

**现状**

* “训练侧”：RLVR +（可学习/可推理的）verifier，出现 generator–verifier 共同训练（RL Tango）。 ([OpenReview][22])
* “推理侧”：best-of-N、树搜索、用 PRM 做过程打分与剪枝；也有人专门做 PRM 校准和 instance-adaptive scaling。 ([arXiv][2])

**常见设置**

* 基准：MATH/竞赛数学、HumanEval/LiveCodeBench（代码）、以及 OOD reasoning 集。 ([arXiv][23])
* 评估：pass@1 / pass@k、推理预算-性能曲线、verifier F1/AUROC、校准误差（ECE/Brier）。 ([arXiv][2])

**主要瓶颈**

* verifier 可能学习“表面线索”（格式、长度、措辞）而非逻辑有效性；PRM 还存在系统性过度自信。 ([arXiv][2])
* RLVR 被指出可能**不扩展**超出 base model 的推理边界（更像分布重加权/采样效率提升）。 ([OpenReview][24])

**争议点 / negative results**

* “RL 是否真的带来新推理能力？”：NeurIPS runner-up 明确给出强负面发现与度量（大 k 的 pass@k）。 ([OpenReview][24])
* “自我验证是否有效？”：存在系统性限制分析。 ([ICML][25])

**可突破点**

* 需要把 verifier 从“黑盒打分器”提升为：**可校准 + 抗 reward hacking + 对反事实推理轨迹敏感**的组件（否则训练和 test-time scaling 都在错误信号上放大）。 ([arXiv][2])

**代表性论文 6–12**

* RL Tango：generator 与生成式 verifier 共同 RL 训练、互相增强，且强调对 reward hacking 的鲁棒性。 ([OpenReview][22])
* SCoRe：多轮在线 RL 训练自我修正，指出离线 SFT 容易分布错配/行为坍塌。 ([arXiv][23])
* Does RLVR really incentivize reasoning beyond base：用大 k pass@k 探测“能力边界”，给出负面结论。 ([arXiv][26])
* Know What You Don’t Know：PRM 校准 + instance-adaptive scaling 降成本保精度。 ([arXiv][2])
* RISE：自我验证强化推理（NeurIPS 2025）。 ([NeurIPS][27])
* Incentivizing LLMs to self-verify：用训练/奖励鼓励自检（NeurIPS 2025）。 ([NeurIPS][28])
* On Self-Verification Limitations：系统性分析自检局限（ICLR 2025）。 ([ICML][25])
* Do We Need Step-by-step Verification：反思过程监督/验证目标设定（OpenReview 2025）。 ([OpenReview][29])
* On Learning Verifiers：把 verifier 学习形式化为 PAC 框架（arXiv/OpenReview 2025）。 ([arXiv][30])
* Reward Reasoning Models：让 reward model 自己“推理后再打分”（NeurIPS 2025）。 ([NeurIPS][31])

**影响力证据（示例）**

* SCoRe（OpenReview cited-by 373）。 ([OpenReview][1])
* RLVR 边界负面结果（OpenReview cited-by 451；NeurIPS runner-up）。 ([OpenReview][24])
* RL Tango 为 NeurIPS 2025 best paper（且 OpenReview cited-by 22）。 ([NeurIPS Blog][21])

### 3.2 可靠性与对齐方向

**现状**

* 校准与不确定性：从“温度缩放/后验”走向“conformal/集合预测/语言校准”。 ([NeurIPS][32])
* 诚实性：明确区分 honesty 与 accuracy，并提供基准与诊断。 ([OpenReview][33])

**瓶颈**

* 标签与 judge 依赖：开放域事实性难标注；LLM-as-judge 的偏差会污染结论。
* CoT 对可靠性评测的副作用：CoT 可能遮蔽幻觉线索。 ([ACL Anthology][4])

**争议点**

* “语言校准”是否能被下游可靠利用？CLC 给出把语言校准解释为集合预测的统一视角。 ([NeurIPS][32])

**可突破点**

* 把可靠性从“输出端”推进到“推理过程/工具调用/agent 行动”的全过程风险控制（risk-controllable decision making）。 ([arXiv][2])

### 3.3 机制解释与表征方向

**现状**

* SAE 成为“通用特征字典学习”路线的主力，并且出现了规模化评测与跨模态扩展。 ([ICLR 会议录][5])
* 同时出现对 SAE 的反思：特征是否规范、解释能否被证伪、解释是否带来效用。 ([ICLR 会议录][6])

**瓶颈**

* 评测缺乏统一“ground truth feature”的场景；跨模型/跨层的对齐问题难。 ([ICLR 会议录][6])
* 电路发现存在搜索空间/复杂度瓶颈。 ([ICLR 会议录][39])

**争议点**

* “SAE 找到的是否是 canonical features？”：ICLR 2025 给出否定证据。 ([ICLR 会议录][6])
* “更可解释是否更有用？”：NeurIPS 2025 有专门研究。 ([NeurIPS][40])

**可突破点**

* 把解释从“描述性”推进为“可预测/可操控”：给出 **falsifiable prediction**（操控某组特征→行为如何变化），并把反例系统化。 ([NeurIPS][41])

### 3.4 训练与优化新范式方向

### 3.5 数据合成与评测方向

**现状**

* 动态评测与污染控制成为“评测科学”：LiveCodeBench 成为代码评测事实标准；污染效应开始用因果方法刻画。 ([OpenReview][9])

* 关注合成数据的有效性与崩塌机制。 ([NeurIPS][55])

**瓶颈**

**可突破点**

* 用“新评测维度”暴露旧方法：例如对 open-ended 多解任务测“多样性与人类偏好校准”（Infinity-Chat/Artificial Hivemind）。 ([OpenReview][56])

### 3.6 高效学习方向

### 3.7 Agents/工具使用方向

**现状**

* SWE-bench 成为软件工程 agent 的硬标准；Web/GUI agent 安全（prompt injection）被独立成 benchmark。 ([SWE-bench][13])

## 5. 主推 idea

> 选择：**候选课题 A**（反事实鲁棒 Verifier）。原因：它同时踩中（i）推理/verifier 热点，（ii）RLVR 负面结果后的关键缺口，（iii）可验证任务带来的强可复现与强证伪。

### 5.1 一句话主张 thesis claim

* **我们提出一种“反事实一致性驱动的 verifier 训练与评测框架”，使过程验证器对逻辑有效性而非表面线索敏感，从而在推理重排序与 RLVR 训练中显著提升鲁棒性、校准性与抗 reward hacking 能力。**

### 5.2 先回答我必须回答的关键问题

**Q1：最接近的 3 类工作是谁？我们多了什么不可替代？**

1. **PRM 校准与 instance-adaptive scaling**：解决“分数=概率”的问题，但主要在 iid 轨迹上做校准，未系统解决“分数依赖伪迹/反事实鲁棒性”。 ([arXiv][2])
2. **生成器—验证器协同训练（RL Tango）**：显著提升训练效率与泛化，但仍可能在分布漂移或对抗轨迹下出现“看起来合理但逻辑破坏”的漏洞；缺少我们提供的 counterfactual 评测与训练约束。 ([OpenReview][22])
3. **Verifier 学习理论化/目标设定（On Learning Verifiers）**：给出形式化学习目标，但没有我们“可复现的反事实数据生成器 + 训练协议 + 端到端实证”。 ([arXiv][30])

**Q2：贡献类型？为什么足够强？**

* 组合拳：**新评测维度（New eval lens）+ 新训练目标（mechanism）+ 可选理论保证（risk control / invariance）**。
* 强点：把 verifier 从“经验打分器”变成“可证伪的科学对象”（你可以系统构造反例，逼它暴露机制缺陷）。

**Q3：算力约束下最小闭环是什么？（2–3 周信号）**

* 2–3 周内完成：

  * 构建一个小型 **CF-TraceBench-mini（数学+代码）**：~5k 题，每题 1 个正确轨迹 + 若干最小反事实轨迹（操作符/关键行替换）。
  * 评测 2–3 个现成 verifier/PRM（含开源或自训小模型）在“标准正确率 vs 反事实鲁棒性”上的差距。
  * 若观察到“标准不错但反事实崩溃”，立即证明 gap 存在并可写成强动机。

**Q4：best paper 级叙事一句话**

* **“当前推理系统失败不在模型不会推理，而在 verifier 奖励信号被伪迹/对抗轨迹欺骗；我们用反事实一致性把 verifier 校准成真正的逻辑判别器，并证明它能系统性提升推理与训练鲁棒性。”**

### 5.3 创新点列表 novelty checklist

> 至少 5 条，并与“最接近的 3 类工作”逐项对比。

1. **Counterfactual Trace Robustness（CTR）评测维度**

* 不再只看 pass@k / verifier F1，而是测：在**语义破坏但表面相似**的反事实轨迹上 verifier 是否能正确翻转。
* 相比 PRM 校准：校准关注概率一致性；CTR 关注语义敏感性。 ([arXiv][2])

2. **CF-TraceBench 数据生成协议（可复现）**

* 统一生成：数学（操作符/关键一步）、代码（关键行 mutation + 单测）、逻辑（量词/条件翻转）三类可验证反事实。
* 相比 RL Tango：RL Tango 更关注协同训练效果；我们提供“可复现反例生成器”作为长期资产。 ([OpenReview][22])

3. **Counterfactual Consistency Verifier Training（C3V）目标函数**

* 用“正确轨迹 vs 最小反事实错误轨迹”的**对比学习/排序损失**，强制 verifier 去掉表面线索依赖。
* 相比 On Learning Verifiers：把抽象目标落为可跑的训练协议与数据。 ([arXiv][30])

4. **Style-invariance（语义保持变换不变）正则**

* 对同一逻辑内容的多种表述（paraphrase/格式变化）做一致性约束，避免 verifier 走捷径。
* 相比现有 PRM/ORM：很多只在单一格式 CoT 上训练。 ([arXiv][2])

5. **Reward-hacking resistance 的系统性测量与训练闭环**

* 用“对抗性轨迹生成器”主动寻找能骗过 verifier 的轨迹，形成 adversarial training/评测协议。
* 与 RL Tango 的“随机性降低 hacking”互补：我们给出可控、可复现的攻击面定义。 ([OpenReview][22])

6. 可选增强：**风险可控推理（risk–coverage）**

* 将 C3V verifier 的分数通过 conformal/分组校准转成“可控错误率”的选择策略（在 test-time scaling 或 agent action 中都能用）。
* 与 CLC/PRM 校准的差别：对象是“轨迹选择/推理正确性”，且重点是反事实鲁棒的 verifier。 ([NeurIPS][32])

### 5.4 核心洞察

* 现有 verifier/PRM 的训练数据通常把“正确性”与“风格/长度/格式/常见模板”强相关，导致它学到 spurious cues。
* **反事实轨迹**（语义破坏但表面相似）会切断这种相关性：

  * 如果 verifier 仍给高分 → 说明它在看伪迹；
  * 如果它能稳定翻转 → 说明它更接近逻辑判别器。
* 这不是工程堆砌：这是一个**可证伪的机制假设**（spurious-cue hypothesis），并配套了能系统构造反例的实验设计。

---

### 5.5 方法清晰定义

#### 5.5.1 数据：CF-TraceBench（自动生成）

对每个可验证问题 (x)：

1. 生成一条（或多条）候选推理轨迹 (T) 与最终答案 (a)。
2. 用确定性 verifier (V_{\text{det}})（数学答案检查/单测/Lean）得到标签 (y \in {0,1})。
3. 构造最小反事实编辑 (E) 得到 (T^{cf} = E(T))，保证：

   * 表面相似（长度/格式基本不变）
   * 正确性翻转（尽可能使 (V_{\text{det}}(x,T^{cf}) = 0)）
4. 可选：构造语义保持的变体 (T^{inv})（paraphrase/格式改写）用于不变性正则。

#### 5.5.2 训练：Counterfactual Consistency Verifier（C3V）

Verifier 模型 (s_\phi(x,T)\in\mathbb{R}) 输出“轨迹正确性分数”（可做 step-level 或 trajectory-level）。

**损失函数**

* 监督分类：
  [
  \mathcal{L}*{cls} = \text{BCE}(\sigma(s*\phi(x,T)), y)
  ]
* 反事实排序（核心）：对每个正例轨迹 (T^+) 和其反事实 (T^-)：
  [
  \mathcal{L}*{cf-rank} = \max(0, m - (s*\phi(x,T^+) - s_\phi(x,T^-)))
  ]
* 语义保持不变性：对等价变体 (T) 与 (T^{inv})：
  [
  \mathcal{L}*{inv} = | \sigma(s*\phi(x,T)) - \sigma(s_\phi(x,T^{inv})) |_2^2
  ]
* 总损失：
  [
  \mathcal{L}=\mathcal{L}*{cls}+\lambda*{cf}\mathcal{L}*{cf-rank}+\lambda*{inv}\mathcal{L}_{inv}
  ]

**伪代码级描述**

* Input：训练集 (\mathcal{D}={(x,T,y,T^{cf},T^{inv})})
* For batch:

  * compute (s^+=s_\phi(x,T)), (s^-=s_\phi(x,T^{cf})), (s^{inv}=s_\phi(x,T^{inv}))
  * loss = BCE(sigmoid(s^+), y) + λ_cf * hinge(m - (s^+-s^-)) + λ_inv * (sigmoid(s^+)-sigmoid(s^{inv}))^2
  * update φ

#### 5.5.3 推理：用 C3V 做可控重排序 / 预算分配

给定问题 (x)，生成 N 条候选轨迹 ({T_i})（来自 base LLM sampling / tree search）。

* 计算分数 (p_i=\sigma(s_\phi(x,T_i)))
* 选择 (\arg\max_i p_i) 或做 risk–coverage（阈值 (\tau)，低于阈值则增加采样/拒答/调用工具）。

### 5.6 训练/推理关键超参与复杂度

**建议默认超参（可作为起点）**

* Verifier backbone：7B 级（如 Qwen2.5-7B）或 3B 级（更便于多实验）。RL Tango 里也用 7B verifier 作参考。 ([OpenReview][22])
* 序列长度：2k–4k（取决于 CoT 长度；代码轨迹可更长）。
* (\lambda_{cf})：1.0 起（0.5/1/2 网格）；(\lambda_{inv})：0.1 起；margin (m)=0.5。
* 反事实样本比例：每个正例配 1–4 个 (T^{cf})（多样化编辑类型）。
* 优化：AdamW，lr 1e-5~5e-5（LoRA）/ 5e-6~2e-5（全参），warmup 3%。

**复杂度**

* 训练：与普通序列分类相同量级，额外只在 batch 内多一次 forward（对 (T^{cf}) 与 (T^{inv})）。
* 数据生成：主要成本在“生成轨迹 + 确定性验证 + 反事实编辑”，可大量用 API 并行；生成一次、训练多次。

### 5.7 至少 3 个强基线

1. **现成 PRM/ORM + 标准校准/IAS**（例如 PRM 校准论文路线）。 ([arXiv][2])
2. **协同训练 verifier 的方法**（RL Tango）或其简化版本（冻结 generator 只训 verifier/反之）。 ([OpenReview][22])
3. **仅用错误样本/负样本但不做“最小反事实”**（随机错误 CoT、或“Learning from Mistakes”类设置）。 ([OpenReview][95])
   4)（额外）**只做语义保持不变性，不做反事实排序**（检验“反事实约束”的必要性）。

### 5.8 预期会失败的设置（至少 2 个）与失败价值

1. **开放域不可验证任务**（创意写作/主观问答）：缺少确定性 verifier，反事实“正确性翻转”不可自动保证。

   * 失败价值：清晰界定方法适用边界，推动“可验证代理任务”构造。
2. **反事实编辑空间极难定义的自然语言推理**（例如多跳事实混杂、语义歧义强）：最小编辑可能改变问题含义而非逻辑。

   * 失败价值：促使我们提出更严格的“编辑有效性判据”或转向形式化子任务。

### 5.9 Minimum Viable Experiments（≥6）与可证伪预期

1. **E1：基线 verifier 的反事实崩溃测量（核心动机实验）**

* 做：在 CF-TraceBench-mini 上测现成 PRM/verifier：标准 AUROC/F1 vs CTR（反事实翻转率）。
* 预期：标准指标不错，但 CTR 显著低（表面线索依赖）。
* 证伪：若基线 CTR 已接近满分，则“反事实鲁棒性 gap”不成立，需要换更难编辑或换任务。

2. **E2：C3V 训练后 CTR 显著提升**

* 做：同 backbone、同训练预算，比较“无 cf-rank” vs “有 cf-rank”。
* 预期：CTR 大幅上升，且不显著牺牲标准正确性。

3. **E3：跨风格泛化（style shift）**

* 做：训练时只见某种 CoT 格式；测试用不同模板/不同语言/不同长度。
* 预期：C3V 的性能掉得更少（因为学的是语义敏感性而非格式）。
* 证伪：若掉幅相同，说明 inv 正则与 cf 生成不够“打断伪迹”。

4. **E4：作为 reranker 提升推理性能-预算曲线**

* 做：对 MATH/HumanEval/LiveCodeBench：固定采样预算 N，比较“无 verifier / 旧 verifier / C3V”。
* 预期：在相同 N 下，C3V 更高 pass@1；或在相同性能下需要更小 N（更省算）。
* 证伪：若 rerank 无收益，则 verifier 信号仍不够区分。

5. **E5：Reward hacking / 对抗轨迹攻击**

* 做：训练一个“攻击者”生成高 verifier 分但最终错误的轨迹（不公开危险内容；只在数学/代码可验证域）。
* 预期：C3V 的攻击成功率显著低于基线（更难骗）。
* 证伪：若同样容易骗，说明反事实覆盖不足或目标函数不对。

6. **E6：把 C3V 接入 RLVR / 协同训练（小规模）**

* 做：在 7B 级模型上做小规模 RLVR（或仿 RL Tango 的设置），对比“基线 verifier vs C3V verifier”。
* 预期：训练更稳定、最终准确率更高，且不出现明显的“能力边界收缩”。（用大 k pass@k 检查边界） ([OpenReview][24])
* 证伪：若仍边界收缩，则支持“RLVR 结构性限制”，我们把贡献收敛到“verifier 评测+鲁棒性”上也仍可成文（Plan B）。

---

### 5.10 风险与 Plan B/C

**主要风险**

* 反事实编辑太“模板化”，模型学会识别编辑痕迹而不是真正理解。
* 跨域（数学→代码/逻辑）泛化不如预期。

**Plan B**

* 把论文定位为 **新评测基准 + 负面发现 + 诊断工具**（类似 JailbreakBench/LiveCodeBench 的叙事），即使训练收益一般也能冲高分。 ([OpenReview][19])

**Plan C**

* 若反事实编辑难做：先在“形式化可编辑域”做深（算术表达式/代码 mutation/Lean tactic 替换），把机制讲透，再逐步扩展。

---

### 5.11 Debug roadmap（诊断树）

1. **若 E1 没看到基线崩溃**

* 检查反事实是否真的“语义破坏”：用确定性 verifier 确认翻转率；提高编辑强度/多样性。
* 增加“表面相似约束”（长度/格式保持），避免被轻易识别。

2. **若 C3V 训练后标准指标掉很多**

* 降低 λ_cf；改用 pairwise logistic loss；或只对高置信正例做 cf-rank。
* 检查反事实是否过噪，导致学习困难。

3. **若 CTR 提升但 rerank 不提升**

* 可能是“分数校准/排序温度”问题：做后处理校准（temperature/isotonic/conformal）。
* 或采样生成多样性不足：提高候选多样性（top-p/temperature）再测。

4. **若仍可被对抗轨迹骗过**

* 引入 adversarial data mining：把攻击样本加入训练；同时强化 inv 正则。

> 下面是“上限友好”的估计，留足反复 ablation 的空间。

* 数据生成（API/离线）：主要是 CPU+API 并行；GPU 不成为瓶颈。
* Verifier 训练（7B LoRA）：

  * 200k–1M 样本、seq 2k：预计 0.5–2 GPU-days（8 卡级）
* 对抗 mining：

  * 主要是推理生成；可用 16 卡并行，1–2 天内完成多轮
* RLVR 小规模（可选）：

  * 7B、短周期：预计 2–6 GPU-days（8–16 卡），严格控制不走“堆算力”。

### 6.5 评测协议（主指标 + 鲁棒性 + 显著性 + 机制验证）

**主指标**

* Verifier：AUROC/F1、ECE/Brier、CTR（反事实翻转率）、Reward-hacking ASR（攻击成功率）
* 推理：pass@1/pass@k、预算-性能曲线、risk–coverage（阈值下错误率）

**鲁棒性**

* 风格 shift：不同 CoT 模板、不同语言、不同长度
* 任务 shift：数学→代码→形式化

**统计显著性**

* paired bootstrap（题目级别）给 95% CI
* ablation 全部报告均值±CI，不只报最好点

**机制验证**

* 相关性诊断：verifier 分数与长度/格式特征的相关性在训练前后应显著下降
* 反事实敏感性：同表面相似度下，语义破坏样本分布应整体左移（可画 density）

## 7. 写作框架

### 7.1 标题候选 ×5

1. **Counterfactual Consistency for Verifiers: Making Process Reward Models Sensitive to Logic, Not Style**
2. **CF-TraceBench: A Counterfactual Benchmark for Robust Verifier Learning in LLM Reasoning**
3. **When Verifiers Fail: A Counterfactual Lens on Reward Hacking and Process Supervision**
4. **Training Verifiers with Minimal Counterfactual Edits for Robust Test-Time Scaling**
5. **From Calibration to Causality: Counterfactual-Robust Reward Models for Verifiable Reasoning**

### 7.2 摘要骨架

* 背景：verifier/PRM 是推理与 RLVR 的核心，但在校准、漂移、reward hacking 上不可靠。
* 观察：现有 verifier 在标准指标上表现好，但在“语义破坏且表面相似”的反事实轨迹上系统性失败。
* 方法：提出 CF-TraceBench（生成协议）与 C3V（训练目标：cf-rank + inv + 校准）。
* 结果：显著提升 CTR、降低对抗成功率，并在 rerank / test-time scaling /（可选）RLVR 训练中带来稳健收益。
* 机制：提供证据说明改进来自“去除表面线索依赖、增强语义敏感性”。
* 资产：开源数据生成器、基准与训练代码，支持复现与扩展。

### 7.3 贡献点 bullet（best-paper 风格）

* 新问题定义/新评测视角：Counterfactual Trace Robustness
* 新基准：CF-TraceBench（可复现生成协议）
* 新方法：C3V 训练目标 + 校准/风险控制接口
* 强实证：跨任务、跨风格、对抗评测、显著性
* 机制解释：spurious cue 诊断 → counterfactual 约束如何修复

### 7.4 主要图表草图说明

* Fig1：反事实评测示意（同一轨迹最小编辑 → 正确性翻转）
* Fig2：标准 AUROC vs CTR 的散点图（展示“标准好但反事实崩溃”的 gap）
* Fig3：加入 cf-rank 后 CTR/ASR 改善曲线（含 ablation）
* Fig4：预算-性能曲线（rerank/test-time scaling）
* Fig5：机制证据（分数与长度/格式相关性下降；反事实 density shift）

* Verifier/PRM 学习与协同训练（RL Tango、verifier learning 框架等） ([OpenReview][22])
