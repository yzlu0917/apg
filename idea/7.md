## 1. 方向地图（含主题覆盖）

> 时间窗口：以 **2024-09 ~ 2026-03（最近 6–18 个月）** 为主，必要时少量补充更早的“奠基性工作”（会明确标注）。

### 1.1 Step A：广撒网（覆盖 9/10 个主题空间）

1. **推理与验证（reasoning / verifier / self-correction）**

* 主线：从“让模型多想/多采样”走向“**用可验证信号 + 过程监督**把推理变成可学习、可控的序列决策”，典型是 RLVR、PRM/过程验证器、生成-批判-修正闭环。([OpenReview][1])
* 痛点：很多“自我反思”是表面修辞；RLVR 是否真的学到“新推理模式”存在争议。([NeurIPS][2])
* 重要性：这是把 LLM 从“好用”推向“可依赖”的核心路径，且天然具备可证伪实验（verifier 失败/误差→性能崩）。

2. **可靠性与对齐（robustness / honesty / calibration / uncertainty）**

* 主线：把“我不知道”变成可训练/可控的行为：不只是拒答，而是**校准置信度、可选择性生成、风险-效用权衡**。([NeurIPS][3])
* 痛点：校准≠排序错误能力；CoT/后训练可能破坏语义校准；很多 UQ 指标与真实风险脱节。([NeurIPS][3])
* 重要性：高风险应用与 Agent 工具调用都离不开“可控不确定性”。

3. **表征与机制解释（mechanistic interpretability / representation）**

* 主线：SAE/稀疏表征 + 因果干预，试图把“特征-电路-行为”闭环做成可评测、可复现实验科学。([OpenReview][4])
* 痛点：解释生成容易空泛；可迁移解释与可重复发现仍缺标准；“解释是否真的因果”需基准化。([OpenReview][4])
* 重要性：Best-paper 级“为什么有效”常需要机制证据，而不是仅 SOTA。

4. **训练与优化新范式（RLHF/RLAIF alternatives, off-policy/online RL, credit assignment）**

* 主线：DPO/IPO/ORPO 等“偏好优化谱系”与 RLVR/过程监督并行发展；同时出现更强的 reward/RM 评测，暴露 reward hacking 与 style bias。([OpenReview][5])
* 痛点：偏好数据噪声、分布漂移、reference 依赖与可复制性；“好看的对齐曲线”可能不对应真实能力提升。([ICLR 会议录][6])
* 重要性：Agent/工具学习常需要更细粒度 credit assignment。

5. **数据合成与评测（synthetic data / evaluation / contamination / benchmark design）**

* 主线：从静态一次性 benchmark 走向“**动态、可更新、可溯源的污染控制评测**”，并开始严肃讨论 construct validity（测到的到底是不是能力）。([OpenReview][7])
* 痛点：污染检测难、评测逻辑 bug、多指标不可比；LLM-as-judge 也需要“无金标准”的评测方法学。([NeurIPS][8])
* 重要性：Best paper 级贡献常来自“把整个方向的测量尺子校准”。

6. **多智能体/工具使用（agents / tool use / planning / search）**

* 主线：Agent 从 prompt 工程走向“**可搜索（MCTS/树搜索）+ 可训练（Gym 化环境）+ 可自动设计（meta-agent search）**”。([ICLR 会议录][9])
* 痛点：评测不稳定、环境漂移、长链错误累积；工具注入/越权成为真实世界主要 failure mode。([GitHub][10])
* 重要性：你偏好 Agent；且该方向非常“NeurIPS 叙事友好”（search/policy/verification 可统一）。

7. **检索与记忆（RAG / long-context / memory systems）**

* 主线：从“塞更长上下文”走向“长期记忆能力分解 + 可诊断 benchmark + 可编辑/可更新记忆”。([OpenReview][11])
* 痛点：记忆正确性、时序更新、避免幻觉与冲突污染；长上下文切换导致性能掉。([ACL Anthology][12])
* 重要性：Agent 一旦进入多轮长期互动，记忆就是可靠性的核心瓶颈。

8. **安全与红队（jailbreak / adversarial robustness / prompt injection）**

* 主线：从单轮 jailbreak 到**多轮、多模态、工具链 prompt injection**；开始出现系统化 benchmark 与自动化红队。([OpenReview][13])
* 痛点：防御常靠启发式；攻击适应性强；工具/记忆引入新的攻击面（数据外流、工具劫持）。([arXiv][14])
* 重要性：Agent 的真实落地风险主要来自这里；很适合做“可验证、可复现”的系统化研究。

9. **结构化生成/程序归纳（code / formal methods / theorem proving）**

* 主线：用可验证环境（unit tests / proof checker）把生成变成“可度量可学习”；证明系统与验证代码 benchmark 同时成长。([NeurIPS][15])
* 痛点：可验证信号稀疏、泛化差、对 lemma/检索依赖；真实世界任务更开放。([arXiv][16])
* 重要性：这里是“理论 + 实证”天然共存的沃土，也能反哺 Agent（例如 SWE/Gym）。

## 3. 深入对比（每个方向：现状→瓶颈→争议→可突破点）

### 3.A RLVR + 过程验证器 + 自纠错（推理与验证）

**现状（范式）**

* Outcome verifier（最终对错）→ Process verifier/PRM（逐步反馈）→ 在线 RL/搜索学习。([OpenReview][19])
* “自纠错”从 prompt 反思走向**训练出内化的 generate→critique→revise**策略。([OpenReview][1])

**代表论文 10 篇（6–12 要求满足）**

* Training Language Models to Self-Correct via Reinforcement Learning (SCoRe, ICLR 2025 Oral)：单模型多轮在线 RL 训练自纠错。([OpenReview][1])
* Rewarding Progress: Scaling Automated Process Verifiers for LLM Reasoning (ICLR 2025 Spotlight)：提出“progress”式过程奖励与理论条件。([OpenReview][19])
* S^2R: Teaching LLMs to Self-Verify and Self-Correct via RL：把 self-verify 明确化并训练。([arXiv][22])
* SPOC: Self-Prompting for Open-ended Tasks：面向开放任务的自我修正/自提示范式。([OpenReview][23])
* PAG: Policy as Generative Verifier：统一 policy 与 verifier 角色，选择性修正。([OpenReview][24])
* Does Reinforcement Learning Really Incentivize Reasoning? (NeurIPS 2025)：系统性指出 RLVR 可能主要提升“采样效率”而非新推理。([NeurIPS][2])
* Beyond Outcome Verification: Verifiable Process Reward Models for Structured Reasoning (2026)：强调过程可验证与结构推理。([arXiv][25])
* Reinforcement Learning for Reasoning… (1-shot RLVR, NeurIPS 2025)：极少样本也能用 verifiable reward 推动能力。([NeurIPS][26])
* RISE: Reinforcing Reasoning with Self-Verification (NeurIPS 2025)：针对“superficial self-reflection”提出在线 RL 框架。([NeurIPS][27])
* Verification Engineering for RLVR (EMNLP 2025)：工程化验证器设计与数据/配方经验总结。([ACL Anthology][28])

**主要瓶颈 / 失败模式（Failure modes map）**

* **表面反思**：模型生成“看似检查”但不改变行动/答案。([NeurIPS][27])
* **verifier 偏差与 reward hacking**：过程奖励若与真实成功不一致，会诱导走偏。([ICLR 会议录][6])
* **可验证领域局限**：很多开放任务缺少客观 verifier，难以 RLVR。([arXiv][29])

**争议点（negative results / limitations）**

* RLVR 是否产生“新推理策略”？有工作给出否定/保守结论。([NeurIPS][2])
* 过程监督是“更好 credit assignment”还是“更强正则 + 更高计算”？需要拆解。([OpenReview][19])

**可突破点（冲 best-paper 的支点）**

* 把“过程验证器”从推理任务扩展到 **Agent 工具链**：每一步 tool call 都是可检查的“过程”。（与第 5 节主推 idea 强相关）

### 3.B Agent（工具使用/规划/搜索/自动设计）

**现状（范式）**

* 从单 Agent prompt 流程 → **搜索（MCTS/树）+ 多 Agent 结构 + 自动设计/编排**。([ICLR 会议录][9])
* 基准从 toy 环境 → 更接近真实工作流（SWE、MCP 工具生态、用户-工具-代理互动）。([OpenReview][30])

**代表论文 10 篇**

* Automated Design of Agentic Systems (ICLR 2025)：把 agent 设计空间系统化并自动搜索。([OpenReview][17])
* AgentSquare (NeurIPS 2025)：系统性评测 agent 在现实任务空间的表现。([OpenReview][31])
* SWE-Search (ICLR 2025)：MCTS + 迭代改进，提升 SWE-bench agent。([ICLR 会议录][9])
* SWE-Gym：用于训练 SWE agents/verifiers 的真实可执行环境。([OpenReview][30])
* τ-bench (ICLR 2025)：工具-代理-用户交互与规则遵循 benchmark。([OpenReview][32])
* MCP-Bench (NeurIPS 2025)：面向 MCP 工具生态的工具使用 benchmark。([OpenReview][20])
* WebArena-Verified（NeurIPS 2025 workshop 发布）：修复/验证 WebArena 评测稳定性问题。([GitHub][10])
* SWE-Bench Pro：长时程 SWE agent benchmark。([OpenReview][33])
* SE-Agent (NeurIPS 2025)：自进化轨迹优化，减少冗余推理。([NeurIPS][34])
* MLE-bench / AI Research Agents for ML (NeurIPS 2025)：把 agent 视作搜索策略做 ML 研究任务。([NeurIPS][35])

**主要瓶颈 / 失败模式（Failure modes map）**

* **长链误差累积**：一次 tool call 错了，后面“合理化”把错越滚越大。
* **评测不稳定/不可复现**：环境变动、评测逻辑 bug、隐含捷径。([GitHub][10])
* **工具注入/越权**：工具文档/输出本身可成为攻击面（tool hijacking、log leak）。([arXiv][14])

**争议点**

* “多智能体一定更强吗？”不少场景单 agent + 好策略仍更稳（需要严谨对照与反事实基线）。([OpenReview][36])

**可突破点**

* 把 Agent 统一到“**风险受控的序列决策**”：目标不只是成功率，而是成功率-风险的 Pareto 前沿（见主推 idea）。

### 3.C 长期记忆与 RAG（含 benchmark）

**现状**

* 记忆能力被拆成：抽取、跨会话推理、时间推理、知识更新、该拒答时拒答。([OpenReview][11])
* RAG 不再只比 EM/F1，而开始强调冲突污染、稳健性与模块协同优化。([NeurIPS][37])

**代表论文 9 篇**

* LongMemEval：长期记忆能力分解 benchmark。([OpenReview][11])
* Evaluating Memory in LLM Agents via Incremental Multi-session…：面向 agent 记忆能力的系统评测。([OpenReview][38])
* Revisitable Memory for Long-Context LLM Agents：可回访记忆机制与评测。([arXiv][39])
* HMT (NAACL 2025)：分层记忆 transformer，应对长上下文与频繁切换。([ACL Anthology][12])
* Beyond a Million Tokens (ICLR 2026)：长程记忆评测与增强。([arXiv][40])
* SeCon-RAG (NeurIPS 2025)：冲突检测+过滤，提升 RAG 稳健性。([NeurIPS][37])
* RAG-DDR：端到端训练 RAG 的可微数据奖励。([OpenReview][41])
* D-RAG：提升 RAG 分布鲁棒性。([ACL Anthology][42])
* FreshStack (NeurIPS 2025)：技术文档检索/RAG 的现实可更新 benchmark。([NeurIPS][43])

**瓶颈**

* **记忆正确性 vs 新鲜性**：更新/遗忘/冲突处理仍缺统一原则。([NeurIPS][37])
* **可复现评测**：长上下文实验对实现细节非常敏感（cache、chunk、reranker）。([NeurIPS][43])

**可突破点**

* 结合不确定性/风险控制：记忆写入/读取是 Agent 的“动作”，可以被风险预算约束（见主推 idea 可作为扩展实验）。

### 3.D 评测科学：污染/构念效度/动态 benchmark

**现状**

* “污染自由 + 动态更新”成为共识：例如持续收集新题。([OpenReview][7])
* construct validity（评测是否测到目标能力）被摆到台面。([OpenReview][21])

**代表论文 9 篇**

* LiveCodeBench (ICLR 2025)：动态收集竞赛题，强调污染控制与多维 code 能力。([OpenReview][7])
* Construct Validity in LLM Benchmarks (NeurIPS 2025)：系统讨论“评测是否真测到能力”。([OpenReview][21])
* CoDeC (NeurIPS 2025)：用 in-context 影响区分 memorization vs OOD，检测污染。([NeurIPS][8])
* Causally Quantifying Test Set Contamination (NeurIPS 2025)：因果方式量化污染效应。([NeurIPS][44])
* FreshStack (NeurIPS 2025)：现实技术文档 IR/RAG benchmark，强调未污染与可扩展。([NeurIPS][43])
* Benchmarking LLMs’ Judgments with No Gold / GRE-bench：无金标准下评测 LLM judgment。([OpenReview][45])
* 数据污染综述（arXiv 2025）：系统总结污染定义、影响与对策。([arXiv][46])
* EMNLP 2025 contamination survey：从静态到动态评测的污染视角。([ACL Anthology][47])
* Controllable contamination-free eval framework（OpenReview）：可控、可复现实验框架。([OpenReview][48])

**瓶颈/争议**

* “更真实”的基准往往更不稳定（环境、API、网站变化）。([GitHub][10])
* LLM-as-judge 的可操纵性与构念问题仍大。([OpenReview][45])

**可突破点**

* 把 Agent 风险/安全也纳入“评测构念”：不仅 pass/fail，而是“风险受控能力”（主推 idea 的评测支点）。

### 3.E 不确定性/校准/Conformal（面向生成与决策）

**现状**

* NeurIPS 2025 出现系统性校准研究：精度≠可靠不确定性，且校准不等于错误排序能力。([NeurIPS][3])
* Conformal/选择性生成开始从分类走向生成，试图给出统计保证。([ACL Anthology][49])

**代表论文 11 篇**

* Revisiting Uncertainty Estimation and Calibration of LLMs (NeurIPS 2025)：大规模不确定性评估与结论。([OpenReview][50])
* Benchmarking Uncertainty Calibration in LLMs (2026)：更系统的校准 benchmark 讨论。([arXiv][51])
* SConU (ACL 2025)：选择性 conformal 不确定性估计，用于过滤/拒答。([ACL Anthology][49])
* COPU：基于 conformal 的 prompt uncertainty。([arXiv][52])
* Conformal Prediction Sets for Next-Token (2025)：词表巨大时的覆盖-效率权衡与改进。([arXiv][53])
* Annotation-Efficient Universal Honesty Alignment (2025)：低标注成本的“诚实/边界感”对齐。([arXiv][54])
* On the Entropy Calibration of LMs (NeurIPS 2025)：研究熵与 log loss 的累积偏差。([NeurIPS][55])
* Scalable Best-of-N via Self-Certainty：用自确定性做 BoN 选择。([OpenReview][56])
* SAFER (2026)：风险约束 sample-then-filter 的框架。([arXiv][57])
* Conformal Information Pursuit (NeurIPS 2025)：交互式任务里用 conformal set 作为不确定性度量。([NeurIPS][58])
* Conformal Language Modeling (ICLR 2024，奠基)：生成任务上的 conformal set 构造与保证。([OpenReview][59])

**瓶颈（Assumption stress test）**

* 经典 conformal 依赖 exchangeability/i.i.d；Agent 场景天然非 i.i.d（策略改变数据分布）。
* 生成任务“正确性指标”难定义，导致“有保证但没意义”的风险。([OpenReview][60])

**可突破点**

* 把正确性定义迁移到 **Agent 的可验证事件**（schema 正确、规则遵循、无泄露、无越权、工具调用成功）→ 让 conformal guarantee 变得“可落地”。

### 3.F 偏好优化/奖励模型评测与鲁棒对齐

**现状**

* Reward Model 本身成为对象：RM-Bench 专门测“微妙内容差异”和“风格偏置”。([OpenReview][5])
* DPO 继续扩展到更鲁棒的目标与数据噪声场景。([NeurIPS][61])

**代表论文 9 篇**

## 5. 主推 idea（完整研究蓝图）：**Conformal Risk Control for Tool-Using Agents**

### 5.0 选择前的关键自问自答（必须可见）

**Q1：最接近的 3 类工作是谁？我们“不可替代”的增量是什么？**

1. **Conformal / selective generation for LLM 输出不确定性**（如 SConU、Conformal LM）：

   * 它们给“回答/token 集合”做保证，但通常不是针对 **Agent 的动作（tool call）风险事件**。([ACL Anthology][49])
   * 我们增量：把 conformal 的对象从“文本输出正确”转为“**工具调用风险（schema/权限/泄露/规则）**”，并给出序列决策下的评测与（尽可能）保证。

2. **工具型 Agent 基准与系统（τ-bench、MCP-Bench、SWE-Gym 等）**：

   * 它们测成功率/规则遵循，但缺少“风险保证”这一层的理论/方法。([OpenReview][32])
   * 我们增量：提出“成功率-风险 Pareto + 校准曲线”的评测体系，并提供通用算法。

3. **Verifier/自纠错/RLVR（SCoRe、Rewarding Progress 等）**：

   * 它们用 verifier 改善推理，但通常不提供统计意义的风险控制，也少专注“工具越权/注入”。([OpenReview][1])
   * 我们增量：把 verifier 分数 **conformalize** 成可控风险阈值；并面向 prompt injection/tool hijacking 做系统评测。([arXiv][14])

**Q2：贡献类型是什么？为什么足够强？**

* 组合型：**新原则（risk-bounded decision making）+ 新评测（risk-success frontier & coverage）+ 新方法（online conformal gating for actions）**。
* 足够强的原因：Agent 的风险事件是可验证且高价值目标，conformal 提供少见的“统计保证”叙事，容易形成 NeurIPS best-paper 风格的清晰主张。

**Q3：算力约束下最小闭环（2–3 周信号）是什么？**

* 不训练大模型：用现成开源/闭源 LLM 作 policy；实现 **Conformal Action Gating**；在 **τ-bench + MCP-Bench + prompt injection 子集**上画出第一条“成功率-风险”曲线：

  * 若在同等或更低风险下成功率不降（或小降），就值得继续。([OpenReview][32])

**Q4：best-paper 级叙事一句话**

* “我们把 Agent 的工具调用从启发式安全，升级为**可校准、可复现、可（近似）保证的风险受控序列决策**。”

---

### 5.1 一句话主张（thesis claim）

> **对工具型 LLM Agent，我们提出一种在线 conformal 风险控制框架，使得在给定风险预算 α 下，Agent 的可验证风险事件（越权/泄露/违规工具调用）发生率在统计意义上受控，并系统性提升成功率-风险 Pareto 前沿。**

---

### 5.2 创新点列表（novelty checklist，≥5 条；含与最接近 3 类工作逐项对比）

1. **把 conformal 的对象从“答案正确性”迁移到“Agent 动作风险事件”**（schema/权限/敏感信息外泄/规则违反），并定义一套通用可验证接口。
2. **在线/分域（Mondrian-style）conformal** 处理 Agent 的非平稳分布：跨 domain / 跨 tool library 自适应校准阈值（显式做 assumption stress test）。
3. **Risk–Success Frontier 评测协议**：不仅报 success rate，而是画 Pareto 曲线 + 覆盖/校准曲线，附显著性与置信区间。
4. **把 verifier gating / BoN / 自纠错统一成“风险受控的 sample-then-filter”理论视角**（从 heuristic 变 principle）。([OpenReview][56])
5. **面向 prompt injection/tool hijacking 的系统化验证**：在 ToolHijacker、Log-To-Leak、Securing AI Agents… 等攻击基准上，验证风险控制的真实有效性。([arXiv][14])
6. **反事实基线（Counterfactual baseline）**：在相同 test-time compute 下对比“固定阈值 gating”“无 conformal 的 verifier gating”“自确定性 BoN”，保证结论不是计算堆出来的。([OpenReview][56])

---

### 5.3 核心洞察（为什么不是工程堆砌，而是新视角/机制）

* Agent 的工具调用引入了“**可验证事件空间**”：JSON schema 合法、权限策略合规、输出是否包含敏感字段、是否遵循规则等（比开放式文本正确性更硬）。([OpenReview][32])
* 一旦有了可验证事件，就可以把“安全/可靠”从主观偏好变成**统计风险控制问题**：

  * 不是“我觉得更安全”，而是“在 α=0.05 下，违规率被控制（或清楚地失败在哪里）”。
* 机制层面：风险控制抑制了 Agent 最致命的失败模式——**早期一步越权/泄露导致后续链式合理化**（长链误差放大）。这能产生清晰可证伪预测（见 5.6）。

---

### 5.4 方法定义（伪代码级别）+ 超参/损失/复杂度

#### 5.4.1 定义：风险事件与打分

* 环境：每个任务是序列决策，状态 (s_t) 包含对话历史、工具返回、记忆等。
* 动作：(a_t) 是结构化动作（如 `call_tool(name, args)` 或 `respond(text)`）。
* 风险事件：(R(s_t,a_t)\in{0,1}) 可由规则/策略/检测器/执行结果判定（例如：调用了黑名单工具、参数越权、泄露敏感字段、违反 τ-bench 规则）。([OpenReview][32])
* 评分函数（nonconformity score）：(\phi(s_t,a_t)\in\mathbb{R}) 越大越“危险”。可组合：

  * verifier 分数（小模型或 LLM-judge）、
  * self-certainty（熵/一致性等）、
  * schema/权限静态检查得分。([OpenReview][56])

#### 5.4.2 训练/校准：Online Conformal Thresholding（核心）

* 数据：用若干任务 rollout 采样动作候选并执行判定 (R)，得到 ((s_t,a_t,R))。
* Split conformal（初始）：在校准集上计算 (\phi) 的分位数阈值 (q_\alpha)。
* Online 更新：滑动窗口或分域（按 tool/domain）维护多个阈值 (q_{\alpha}^{(d)})，缓解分布漂移。

#### 5.4.3 推理：Conformal Action Gating（CAG）

```text
Input: state s_t, policy π, scorer φ, threshold qα, candidate count K
1) Sample candidates A = {a_t^1..a_t^K} ~ π(·|s_t)   (temperature/top-p)
2) Compute scores u_i = φ(s_t, a_t^i)
3) Feasible set F = {a_t^i : u_i ≤ qα}
4) If F non-empty: choose a* = argmin_{a in F} u(a)  (or best utility under constraint)
   Else: abstain / ask clarification / choose safest fallback action
5) Execute a*, observe next state s_{t+1}
6) (Optional) online update qα with observed R(s_t,a*)
```

#### 5.4.4 关键超参（建议范围）

* 候选数 (K)：8 / 16 / 32（主要 test-time compute 旋钮）
* 风险预算 (\alpha)：0.01 / 0.05 / 0.10（画 Pareto）
* Online 窗口 (W)：256 / 1024 个最近动作样本（按 domain 分桶）
* scorer 组合权重：用最小验证实验做网格（不超过 20 组）

#### 5.4.5 复杂度估计

* 每步需要 (K) 次采样（可并行），外加 (K) 次轻量打分。整体是 **O(K)** 的 test-time 扩展；无需大规模训练。
* 在 16×80GB A800 上：可以把不同任务/seed 并行跑，主要瓶颈转到环境执行与 API 调用（你预算充足）。

---

### 5.5 至少 3 个强基线（含最难打败 baseline）

1. **Vanilla Agent**：单样本 tool call（ReAct/标准 function calling 风格），无 gating。
2. **Best-of-N with Self-Certainty**：只用自确定性/熵做 BoN 选择（不做 conformal 校准）。([OpenReview][56])  ← *我认为最难打败：便宜、强、通用*
3. **Verifier-gated (fixed threshold)**：有 verifier，但阈值手调或在开发集上调（无 coverage 保证）。
4. **针对 prompt injection 的现有防御**：例如 NeurIPS 2025 的轻量防御方法（作为安全子任务对照）。([NeurIPS][81])

---

### 5.6 关键实验（minimum viable experiments，≥6 个）+ 预期现象（falsifiable）

> 目标：每个实验都能“失败得很清楚”，能定位假设/机制哪里不成立。

1. **Coverage/Calibration 实验（核心）**

* 设定 α∈{0.01,0.05,0.1}，测工具违规事件率是否 ≤ α（带置信区间）。
* 预期：CAG 的违规率随 α 单调变化；fixed-threshold 与 self-certainty 不能稳定匹配目标 α。

2. **Risk–Success Pareto Frontier**

* 在 τ-bench / MCP-Bench 上画 success vs risk 曲线（不同 K、α）。([OpenReview][32])
* 预期：在同等 risk 下，CAG 的 success ≥ baselines；或在同等 success 下 risk 更低。

3. **分布漂移（Assumption stress test）**

* 训练/校准在 domain A（某些工具/行业），测试在 domain B（新工具/新规则）。
* 预期：纯 split conformal 覆盖会劣化；online / 分域 conformal 能显著恢复覆盖。

4. **Prompt Injection / Tool Hijacking 防线有效性**

* 在 ToolHijacker、Log-To-Leak、Securing AI Agents… 子集上测攻击成功率（ASR）与任务成功率。([arXiv][14])
* 预期：CAG 显著降低 ASR，且主要通过“拒绝高风险 tool action”实现；若失败，能定位 scorer 被注入/污染。

5. **反事实基线（Counterfactual baseline）**

* 固定 K 与总调用次数：比较“CAG vs 非校准 verifier gating vs self-certainty BoN”。
* 预期：若 CAG 只是 compute 堆砌，则在等 compute 下优势消失；我们预期仍保留优势（来自校准/阈值合理性）。

6. **消融：score 组成与 online 更新**

* 只用 verifier / 只用 self-certainty / 组合；去掉 online 更新。
* 预期：组合最好；online 更新主要改善 OOD 覆盖与风险控制稳定性。

7. **机制验证：长链误差放大抑制**（可选但很强）

* 统计“早期一步违规 → 后续失败概率”的因果关联；看 CAG 是否减少这类路径。
* 预期：CAG 显著降低“早期高风险动作”占比，从而改善长链成功率。

---

### 5.7 预期会失败的设置（≥2）+ 失败价值

1. **强非交换性环境**（策略行为改变任务分布很剧烈、或对抗者自适应）：

* 可能出现覆盖失真（违规率 > α）。
* 价值：明确指出 conformal 的假设边界，并推动我们做“更强 online/对抗校准”或“保守保证”。

2. **风险事件不可验证/标注噪声极大**（例如主观安全标准、或检测器误报严重）：

* 可能导致 CAG 过度拒绝、success 大幅下降。
* 价值：逼迫我们提出“可验证 proxy”的设计原则（也是论文贡献的一部分）。

---

### 5.8 潜在风险与 Plan B/C

**主要风险 1：覆盖保证不稳（分布漂移太强）**

* Plan B：分域/Mondrian conformal（按工具/域/任务类型分桶）；引入 online 校准窗口。
* Plan C：转为“保守上界 + 经验风险控制”，强调可复现评测与失败边界，而非强保证（仍可 best-paper：把假设边界讲清）。

**主要风险 2：success 掉太多（过度保守）**

* Plan B：加入“utility under constraint”选择：在满足 u≤qα 的集合里最大化任务价值（例如用轻量价值函数）。
* Plan C：把 abstain 变成“请求澄清/补充信息”的行动，提高可用性（在 τ-bench 这类交互基准上尤其自然）。([OpenReview][32])

**主要风险 3：scorer 被 prompt injection 污染**

* Plan B：scorer 多样化与隔离（规则检查 + 小模型 + LLM judge 多投票），并对工具输出做信任域标记。([OpenReview][79])

---

### 5.9 诊断树（debug roadmap）

1. **风险率 > α（覆盖失败）**

* 先查：风险事件定义/检测器是否稳定（误检/漏检）

  * 若检测器不稳 → 修检测器/改为更可验证 proxy
* 再查：校准集是否代表测试分布

  * 若 OOD → 分域 conformal / online window / 提升校准覆盖
* 再查：score 是否“单调地反映风险”

  * 若不单调 → 重新设计 φ（加入更强规则特征、或改 verifier 提示/训练）

2. **success 明显下降**

* 查：abstain 触发频率是否过高

  * 降低保守度（增大 α）或增大 K
* 查：在可行集合里是否选错（argmin u 不等于最大 utility）

  * 引入受约束效用最大化

3. **对 prompt injection 没效果**

* 查：攻击是否绕过风险事件定义（没被算作 R=1）

  * 扩充风险事件 taxonomy（泄露、越权、工具劫持、跨上下文污染）
* 查：scorer 是否可被注入

  * 做 scorer 隔离/多路验证/工具输出净化

## 6. 执行计划（里程碑 + 资源估算 + 实验清单）

> 假设总周期 10–12 周；**2–3 周内必须拿到“是否值得继续”的信号**（见第 6.2）。

### 6.1 数据与基准选择（含理由）

* **τ-bench**：规则遵循 + 工具交互，天然适合定义“违规风险事件”。([OpenReview][32])
* **MCP-Bench**：真实工具生态（MCP）场景，覆盖工具库/工具文档的攻击面。([OpenReview][20])
* **Prompt injection / tool hijacking 基准**：

  * ToolHijacker（tool selection 攻击）([arXiv][14])
  * Log-To-Leak（MCP 工具外流）([OpenReview][79])
  * Securing AI Agents Against Prompt Injection Attacks（847 用例）([arXiv][80])
* （可选增强）WebArena-Verified：若要做网页 agent，优先用 verified/patch 版本提高复现性。([GitHub][10])

### 6.2 里程碑（按周）

**Week 1：基线复现 + 风险事件定义（可验证）**

* 跑通 τ-bench / MCP-Bench / 注入基准最小子集。([OpenReview][32])
* 定义风险事件 taxonomy（至少 4 类）：schema 违规、规则违规、敏感信息外流、危险工具调用。
* 建立日志格式：每步保存 (state, candidates, chosen, scores, R, success)。

**Week 2：实现 CAG（训练-free 版本）+ 第一条 Pareto 曲线**

* scorer v0：规则检查 + self-certainty（熵/一致性）([NeurIPS][55])
* split conformal 校准阈值 qα。
* 输出：success-risk 曲线 + 覆盖曲线（是否能控制风险）。

**Week 3：最小闭环决策（Go/No-Go 信号）**

* 加入注入攻击评测（ToolHijacker/Log-To-Leak 小规模）。([arXiv][14])
* 若看到：

  * 风险可控（接近 α）且 success 仅小幅下降/甚至提升 → **继续**
  * 否则进入 debug roadmap（5.9）

**Week 4–5：Online/分域 conformal + 反事实基线**

* 实现 online window / Mondrian 分桶（按工具/域/任务类型）。
* 做 counterfactual baseline（等 compute 比较）。
* 输出：OOD 覆盖恢复曲线 + 等 compute 优势。

**Week 6–7：更强 scorer（轻量训练，不堆大模型）**

* 训练一个小型风险预测器（例如 1–3B 或更小 encoder）做 φ 的一部分：

  * 数据来自你已有 rollout 日志（无需人工标注）。
* 加入多 scorer 投票（规则 + 小模型 + LLM judge）。

**Week 8–9：机制验证与泛化**

* 机制实验：早期高风险动作是否显著降低；风险控制是否减少长链崩溃。
* 泛化：换模型（开源/闭源）、换工具库/域，验证方法可迁移。

**Week 10–12：收尾（全量评测 + 消融 + 写作）**

* 全量跑齐所有 α、K、ablation、attack suites。
* 做统计显著性（paired bootstrap）与复现脚本封装。

### 6.3 模型/方法细节（工程落地）

* **policy LLM**：优先用 7B–14B 指令模型做可复现实验；再用闭源强模型做“上限对照”。
* **推理流程**：vLLM/并行采样；每 GPU 跑不同任务/seed；工具执行异步。
* **复杂度控制**：K=8/16 作为默认；K=32 仅用于曲线端点。

### 6.4 评测协议（主指标 + 鲁棒性 + 显著性 + 机制验证）

* 主指标：

  * **Task Success**（基准自带）
  * **Risk Rate**（你定义的可验证违规率）
  * **Coverage gap**：risk_rate − α
* 鲁棒性：跨 domain、跨工具、跨模型；加入 adversarial prompts。([OpenReview][76])
* 统计：paired bootstrap（按任务配对），报告 95% CI。
* 机制验证：风险分数与失败的相关/校准；早期风险动作对最终失败的影响。

### 6.5 资源估算（显存/时间/GPU 天数上限/并行策略）

* 训练需求：可做到“几乎不训练大模型”；若训练小 scorer：

  * 1–4×80GB，1–2 天足够（数据量取决于 rollout）。
* 主要成本：并行跑 benchmark + tool 执行 + API 调用。
* 并行策略：16 GPU 分成 4 组：

  * 2 组跑 τ-bench/MCP-Bench，1 组跑注入攻击套件，1 组做 scorer 训练/消融。

### 6.6 复现与工程组织

* 配置：Hydra/JSON 配置化（α、K、scorer 组合、基准版本）。
* 日志：每步 action-level trace；可回放；保存 tool I/O（脱敏）。
* 版本：锁定 benchmark commit；对 Web 环境用 verified/patch 版本。([GitHub][10])
* 报告：每个实验自动生成 Pareto 曲线与 coverage 图。

## 7. 写作框架（标题×5、摘要骨架、贡献点、图表草图、related work）

### 7.1 标题候选（5 个）

1. **Risk-Bounded Tool-Use Agents via Online Conformal Verification**
2. **Conformal Action Gating: Statistical Risk Control for LLM Agents with Tools**
3. **From Heuristics to Guarantees: Calibrating and Controlling Agentic Tool Risks**
4. **Measuring What Matters: Risk–Success Frontiers for Tool-Using Agents**
5. **Safe by Construction? Conformal Risk Control against Prompt Injection in Tool Agents**

### 7.2 摘要骨架（按 best-paper 叙事）

* 背景：工具型 Agent 进入真实系统，但风险事件（越权/泄露/注入）成为主要障碍。
* 缺口：现有方法缺少“可校准、可复现、可（近似）保证”的风险控制。
* 方法：提出 CAG（online conformal gating）+ 通用风险事件接口 + 分域/在线校准。
* 理论/机制：在交换性假设下给出风险控制；并系统评估假设破坏时的失效边界与修复。
* 实验：τ-bench、MCP-Bench、prompt injection 套件；展示 Pareto 前沿提升与风险率受控。
* 结论：把 Agent 安全从 heuristic 提升到 principle，并给出标准评测协议与可复现实现。

### 7.3 贡献点 bullets（论文开头直接列）

* **C1（Problem/Metric）**：定义工具型 Agent 的可验证风险事件与 risk–success frontier 评测协议。
* **C2（Method）**：提出 online conformal action gating，把任意 verifier/score 校准成风险预算 α 的可执行决策规则。
* **C3（Theory/Mechanism）**：给出在假设成立时的风险控制结论，并系统刻画分布漂移下的失败与在线修复。
* **C4（Empirics）**：在 τ-bench、MCP-Bench 与 tool/prompt injection 套件上验证通用性与鲁棒性。([OpenReview][32])
