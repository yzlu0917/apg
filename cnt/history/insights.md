# Insights

## Patterns

- Tight continuation budgets are required for synthetic necessity to be nontrivial. If the remaining budget freely absorbs a dropped step, every candidate collapses to recoverable and the benchmark stops measuring necessity.
- Paraphrase edits should stay inside the same surface family whenever possible. Switching the whole template family makes detectability too easy and contaminates the naturalness audit.
- Real-domain trace collection should truncate generation at the first `Final answer:` line. Otherwise the model can spill into the next imagined problem and contaminate saved reasoning text.
- A second local repair-style prompt is enough to recover some correct GSM8K traces from `Qwen3-1.7B`; for bootstrapping success traces, this is worth keeping before heavier tooling is added.
- For Stage A continuation prompts, never include a literal placeholder like `Final answer: <number>`. Small local models may copy the placeholder verbatim and destroy the verifier signal.
- In early-prefix math pilots, original-prefix solvability can already be low. Without conditioning candidate selection or stronger continuators, `N_t` gets dominated by continuator weakness instead of step necessity.
- Locked-prefix continuator prompts matter. If the continuator is allowed to reinterpret or repair earlier steps too freely, late-step `N_t` collapses even when the edited arithmetic is clearly wrong.
- For arithmetic reasoning traces, heuristic wrong edits can be stronger than free-form model-generated swaps. Small local models often rewrite too softly; direct arithmetic perturbations produce more consequential local counterfactuals.
- Long Stage A audits need incremental write-out and resume. Once the continuator gets larger than the 1.7B bootstrap model, all-or-nothing execution wastes too much compute and makes held-out audits operationally brittle.
- A stronger held-out continuator does not necessarily kill the signal globally; it may selectively repair only a few prefixes. Per-problem concentration matters more than overall averages when deciding what to keep.
- Conservative paraphrase can help more on held-out than on train-side. Reducing paraphrase freedom fixed the broad held-out drift, but it also exposed a separate local train-side template failure.
- A second paraphrase pass can still matter even after the first held-out fix. Once broad drift is gone, small template choices (`giving` vs sentence split, article insertion vs no article) are enough to move the keep-set.
- Once paraphrase is under control, larger-scale audits expose a different regime. Keep-rate can fall even when held-out weighted `N_t` rises, because new trace families contribute more swap-repairable prefixes than paraphrase-drifty ones.
- For scaled cross-continuator filtering, read `keep_fraction` and kept-subset weighted `N_t` together. A cleaner kept subset with lower global keep rate usually means the pool got broader, not necessarily that the object collapsed.
- Operation-family edits need plausibility bounds. Extreme operator flips like `550 / 110 -> 550 * 110` invite explicit held-out repair (`Wait, that's not right...`) and can dominate the audit for the wrong reason.
- When a plausible operator change is unavailable, falling back to a strong local quantity error is better than forcing an obviously absurd operation swap. On the current math pool this cut held-out `mean_swap_solve` from `0.2049` to `0.1146`.
- A final train-side-only filter can clean the training core without much attrition. Requiring perfect train original solvability plus zero train-side paraphrase gap reduced the post-Stage-A pool from `35` kept to `33`, while raising within-pool keep fraction from `0.7292` to `0.8250`.
- Stage B cannot be gated on local pair metrics alone. In the first matched-training pilot, `pref_accuracy` was already saturated at `1.0` before training, while held-out rollout `mean_n_t_weighted` still fell from `0.2133` to `0.1152`.
- Continuator-generated positive suffixes can be verifier-correct but locally dirty. A suffix that ends with the right `Final answer` can still contain an inconsistent intermediate statement and poison offline training if used as the positive completion.
- For rollout-style matched training, use verified success-trace suffixes on the chosen side and counterfactual continuator suffixes only on the edited side. On the current math pilot, this removed the `gsm8k-00164` collapse and restored held-out rollout metrics to the base-model level.
- Once dirty positives are removed, matched SFT-only and clean-positive CNT can still be indistinguishable at smoke scale. That means the next bottleneck is not safety but separation power: either the training signal or the held-out gate must get stronger before CNT-specific gains can appear.
- Apparent Week 3 gains on tiny eval splits should be rechecked with a looser continuation budget. In the current split-compare run, the only positive seed disappeared when `continuation_max_new_tokens` increased from `120` to `160`, because the control model had already produced the right reasoning but left `Final answer:` blank under the tighter budget.
- Final-answer parsing must be slightly more permissive than a bare numeric regex. Blank `Final answer:` stubs and explicit forms like `Final answer: $15` can otherwise create fake rollout deltas even when the reasoning and answer are already recoverable under the same verifier.
- After hardening the rollout gate, re-evaluate earlier Stage B claims before carrying them forward. In the current project, both the `seed23` apparent win and the `smoke01` apparent harm disappeared once the verdict logic was fixed.
- Once the hardened gate is in place, replace random split smoke with disjoint full-coverage folds. In the current 16-example Stage B core, the 4-fold comparison stayed exactly tied across all folds, which rules out split luck as the main explanation for the missing CNT gain.
- An `original-prefix` rollout anchor is not enough by itself. In the current Stage B core, adding one extra success-trace SFT anchor per kept pair left the hard fold00 unchanged: matched SFT-only and CNT still co-degraded from `0.9 / 0.8` to `0.8 / 0.7`.
- If Stage B pair supervision is meant to affect held-out rollout behavior, align the pair completion to rollout rather than just the next step. In the current project, `anchor + rollout pair` repaired the hard fold00 back to the base-model rollout, while `anchor + step pair` stayed tied with matched SFT-only.
- Weighted `N_t` can move in the wrong direction during Stage B if a model degrades original and edited prefixes more uniformly. On the current hard fold00, matched SFT-only raised `mean_n_t_weighted` while lowering both `mean_original_solve` and `mean_drop_solve`, so Stage B decisions must read utility and weighted necessity together.
- A hard-slice repair is not enough; budget-stress it immediately. In the current project, the only post-hardening positive signal (`anchor + rollout pair` on fold00 at `lc120`) disappeared when `continuation_max_new_tokens` increased to `160`, so it was not safe to promote that recipe as the new default.
- When the conservative Stage B core is too easy, build a reproducible non-ceiling slice instead of abandoning the domain. In the current GSM8K setup, selecting examples with either non-ceiling base rollout solve or sub-perfect Stage A held-out solve exposed a budget-robust CNT-vs-control separation that the easy-heavy aggregate had hidden.
- Stage A held-out hardness and Stage B discriminative hardness are different objects. In the expanded `30`-trace GSM8K core, an `audit_only` hard slice (`gsm8k-00007 / gsm8k-00132`) is genuinely hard under the Stage A audit but still yields an exact `base = control = cnt` tie at Stage B under both `lc120` and `lc160`.
- When growing the Week 3 gate after a Stage A pool expansion, do not rely on held-out audit difficulty alone. Use fresh Stage B base-rollout artifacts to find non-ceiling examples; otherwise the selected slice can shrink to a tiny but non-discriminative set.
- A broader mixed-source hard gate can expose recipe tradeoffs that the smaller regression probe hides. On the rebuilt `7`-example gate over `audit07`, `signal03` improves weighted `N_t` relative to control at `lc120` but improves `drop_solve` relative to control at `lc160`; the sign of the apparent win depends on which metric is read first.
- Removing explicit drop-prefix SFT is not enough to restore faithfulness. On the rebuilt `7`-example gate at `lc160`, the `anchor_only` variant (`include_drop_sft = false`) makes control and CNT tie on the main rollout metrics, so the broader conflict is not primarily caused by those positive SFT rows.
- Once bootstrap trace collection is working, seed diversity is a faster lever than squeezing harder on a small pool. On GSM8K, one 8-GPU batch of `8` seed-shuffled runs with `max_examples = 128` and `target_successes = 16` expanded the merged pool from `30` to `112` unique verified traces, which is enough to rebase Week 2/3 on a larger evidence set instead of over-optimizing tiny hard slices.
- For resumable Stage A shards, the presence of `stage_a_math_summary.json` does not mean the shard is complete. The summary is rewritten after every finished record; use session exit or the theoretical candidate-count total to decide when it is safe to merge.
- On the `112`-trace GSM8K pool, larger-pool scaling preserved the same qualitative Week 2 diagnosis: conservative filtering stayed cheap (`298 / 331` kept) and the final joint audit stayed strong (`230 / 298` kept), while the dominant failure family remained held-out swap repair rather than paraphrase drift.
- Rebuilding a larger Week 3 gate is much more expensive than the old tiny probes. On the `audit08` core, fresh Stage B base-rollout folds over `110` unique examples are required before `base_and_audit` selection can run, and the unchanged strict rule still only reaches `29` examples. This means getting from the old `4/7`-example probes to a `30+` main gate is not automatic under the old selection rule; if we want more than `29`, that will have to be an explicit gate-definition change.
- The first larger-pool Week 3 compare can be stricter than the old probes in a useful way. On the strict `29`-example `audit08` mixed gate, matched SFT-only control and CNT tie on `mean_drop_solve`, but CNT loses `mean_original_solve` slightly. This is stronger evidence than the old tiny-gate ambiguity: once the gate is large enough, weak CNT recipes can stop looking like “unclear ties” and start looking mildly worse than control.
- On a fixed Week 3 gate, `weight_source` sweeps can be lower-leverage than they look if the pathological high-weight examples are eval-only. In the current strict `29`-example gate, switching `heldout -> min/train` removed the old `gsm8k-00155` original-prefix regression, but the real effect came from changing which drop-repairable training families were favored, not from directly down-weighting that eval-only row.
- Once the larger fixed gate is in place, Stage B recipe changes can become extremely sparse. Here the main rollout deltas after the first sweep are only `1-2` record flips (`gsm8k-00134`, `gsm8k-00198`, `gsm8k-00118`), yet those flips are enough to reverse the sign of utility or weighted `N_t`. Read the per-record frontier, not just the mean.
- Same-gate Stage B sweeps should reuse base rollout artifacts. For a fixed eval-id set and unchanged base model, repeated `base_rollout` passes are pure duplicated cost; `scripts/run_math_stage_b_fold_compare.py` now supports `--reuse-base-rollout-from` to avoid paying that cost again.
- A hard-negative-only pref objective is not automatically the fix for the strict-gate Stage B frontier. On the current `29`-example gate, `step-pair + pref_margin_target = 1.0` simply flips the sign of the `step-pair` tradeoff: the decisive record moves from `gsm8k-00134` to `gsm8k-00198`, utility drops back below control, and weighted `N_t` rises again.
- Original-anchor truncation negatives can be active without improving the frontier. On the current strict `29`-example gate, `step-pair + anchor_pair_mode = original_truncated_pref` increases anchor-side `pref` rows from `232 -> 343` in train and `78 -> 119` in eval, and it changes which examples flip under rollout, but the aggregate `delta(cnt-control)` remains exactly the same as plain `step-pair`.
- On the current `audit08` keep-set, literal utility sign-consistency is already implied for Stage B `pref` rows. All `230 / 230` kept candidates satisfy `min(train_delta, heldout_delta) >= 0` for `drop`, `swap_quantity`, and `swap_operation`, so applying that rule directly to swap-pair eligibility is a no-op.
- The real utility-aware lever is `drop-prefix` SFT, not swap-pair eligibility. In the same `audit08` keep-set, `227 / 230` candidates still have `min(train_drop_delta, heldout_drop_delta) = 0`, so generic drop-reconstruction supervision can dominate unless `drop-prefix` SFT rows are filtered or reweighted.
- Filtering zero-margin `drop-prefix` SFT is not enough by itself to create a CNT win. On the strict `29`-example gate, `step-pair + drop_sft_filter = one_side_positive` removes the old `step-pair` separation entirely: control and CNT end up exactly tied on rollout utility and weighted `N_t`.
- Adding original-anchor truncation negatives on top of that filtered-drop base does not dominate the tie; it recreates the old frontier. On the same strict `29`-example gate, `step-pair + drop_sft_filter + original_truncated_pref` moves from exact tie to `drop_solve -0.0161 / weighted N_t +0.0158`, so extra anchor-side pressure alone is not the missing positive signal.
- On the cleaned `dropfilter` base, adding rollout-pair rows alongside step-pair rows is still not enough to separate CNT from control. `step_and_rollout + drop_sft_filter = one_side_positive` improves offline pair margins, but the strict `29`-example rollout gate stays an exact tie, so the remaining Week 3 bottleneck is not just “missing rollout modality”.
- Once a Week 3 gate has absorbed several exploratory recipe sweeps, freeze it explicitly and move development to a complementary dev pool instead of continuing to search on the same examples. On `audit08`, freezing the strict `29`-example mixed gate and splitting out the remaining `81` examples gives a cleaner final-test vs recipe-dev separation without changing the scientific gate definition.
- A recipe-dev pool should not be fed directly into `--eval-example-ids-path` for compare if the complement is the frozen final-test gate. That would make the final-test examples become the training split. The correct pattern is to emit a dev-only `audit_kept` file and split train/eval *within* the dev pool.
- Making `equiv` rows uniformly weighted is not enough by itself to unlock Stage B. On the current `81`-example dev pool, `step_and_rollout + drop_sft_filter + equiv_weight_mode = uniform` improves offline pair margins in every fold, but the rollout-side CNT-minus-control delta stays exactly zero across all four folds.
- On the current `81`-example dev pool, naive utility-aware weighting is likely close to a no-op. The kept dev rows have `drop` utility margin identically `0`, while `swap_quantity` / `swap_operation` margins are almost uniformly `1.0`, so a next recipe should target the objective structure more directly rather than expecting simple utility reweighting to reshape training.
- Suppressing `step`-pref while keeping `rollout`-pref and anchor-pref active does change Stage B behavior, but it recreates the same sparse frontier instead of unlocking a broad win. On the current `81`-example dev pool, `pref_step_multiplier = 0`, `pref_rollout_multiplier = pref_anchor_multiplier = 1` lifts `drop_solve` slightly above control on two folds while lowering `mean_n_t_weighted`; this is strong evidence that pref-subtype weighting is an objective lever, but not yet the right one.
- Stronger anchor-side semantic negatives can neutralize a sparse dev frontier without creating a positive separation. On the current `81`-example dev pool, adding `anchor_pair_mode = original_counterfactual_pref` on top of the `rollout-pref-only` recipe collapses the `signal11` frontier back to an exact 4-fold tie. That suggests the next useful Stage B move is a genuinely cross-prompt protection loss, not more anchor-only preference shaping.
- Even a genuine cross-prompt protect loss can still be rollout-no-op. On the current `81`-example dev pool, `protect_mode = original_over_drop_hinge` adds explicit `original prompt -> gold suffix` vs `drop prompt -> repaired gold completion` supervision, yet the 4-fold rollout aggregate stays exactly tied and the protect offline margin becomes slightly more negative for CNT than control. After this point, continuing to tweak the same Stage B family is much less justified than either changing objective family outright or writing Week 3 up as a clean negative/partial-result branch.
- Zero-weight Stage B rows should short-circuit before any logprob work. Once dev recipes zero out whole loss families (`lambda_n = lambda_inv = lambda_protect = 0`), leaving those rows active in `_row_loss` turns the wall-clock into a fake engineering bottleneck and makes objective-family comparisons look slower than they really are.
- Even a genuinely different bundle-level family can still reproduce the same Week 3 frontier. On the current `81`-example dev pool, bundle-level cross-prefix ranking improves offline bundle margins in all four folds, but rollout stays tied in three folds and becomes `utility - / weighted N_t +` in the fourth. So offline movement alone is no longer evidence that a new family is closer to a real CNT-over-control win.

## Pitfalls

- Wasteful successful traces are not enough by themselves. If the evaluation budget is tied to the full trace length plus slack, deletion-based necessity can degenerate to zero even when the underlying task has real dependencies.
- Detectability numbers must be read together with `corr(CNT, detectability)`. A family can be classifiable yet still not dominate the necessity score; both numbers matter.
- Real-domain swap edits that look wrong to us may still be repaired away by the continuator. If the continuator can cheaply self-correct, late-step `N_t` will collapse toward zero even when the edited step was semantically important.
- In the refined math setup, paraphrase is now the editor family that needs the most scrutiny. Swaps can already be made maximally consequential, but paraphrase still sometimes changes solve probability more than a finished invariance audit should allow.
- Small-sample keep-sets can overestimate readiness. `audit03` looked clean at `9 / 12`, but the first 16-trace rerun dropped to `26 / 48` because held-out `swap` repair generalized less well than the toy pool suggested.
- For cross-continuator filtering, `positive weighted N + adequate original solvability + small paraphrase gap + bounded swap solve` is a workable first keep rule. On the current 12-pair audit it removes the obviously repairable prefixes without collapsing the pool.
- The template `X, so Y` is fragile. Splitting it into two sentences (`X. Therefore, Y`) can preserve semantics but still perturb continuation behavior enough to fail the invariance audit. Keep this family as a dedicated editor test case.
- For `X, so Y`, same-sentence rewrites are safer than sentence splitting. On the current math traces, `X, giving Y` materially improved train-side paraphrase stability and increased the cross-continuator keep-set from `8 / 12` to `9 / 12`.
- Fully dropped failure families are worth tracking by problem id, not just aggregate counts. In the current 16-trace audit, `gsm8k-00076` and `gsm8k-00096` account for a disproportionate share of held-out-swap drops and should be used as regression probes for the next swap-editor pass.
- If a larger audit recovers after an editor fix while paraphrase metrics stay flat, that is strong evidence the failure was in the editor rather than the object. Here `audit05` moved from `26 / 48` to `35 / 48` without changing held-out paraphrase behavior, confirming that diagnosis.
- Conservative pre-training filters should be train-side-only. Using train-side solvability and paraphrase invariance to define the final core keeps the evidence chain clean and avoids hand-tuning on held-out failures.
