# Insights

## Patterns

- Judge-style leakage and trained-verifier leakage must be evaluated separately. Prompt judges and external closed judges can show very high `ASS_total` while trained same-backbone heads remain much less answer-sensitive on the same benchmark slice.
- The same-backbone answer-masked baseline is a gating baseline, not an appendix baseline. If it stays strong, algorithmic repair claims should be downgraded immediately.
- Hardening task difficulty is more informative than adding paraphrase diversity once the masked baseline survives held-out verbalizer evaluation.
- Planning-style generated traces may need a stronger scaffold than route-style traces. Prompt-only tightening was enough to recover graph coverage, but blocksworld required legality-aware stepwise generation.
- On the current full tri-domain generated OOD slice, pure `cond-swap` repair generalizes better than the more complex joint repair. The joint objective can look strongest on synthetic validation and still transfer worse than a simpler invariance term.
- On a naturalized version of the same full-hybrid slice, the ranking can flip again: `cond-swap` keeps very low `ASS_total`, but its `AMCD` win can disappear. That is a signal for a faithfulness-utility frontier, not necessarily a failed implementation.
- Self-mined `hard-neg` replay can push the model back toward utility while reintroducing answer sensitivity. On the current naturalized full-hybrid slice, it helps recover `AMCD` only by giving back the low-`ASS_total` advantage from `cond-swap`.
- Replay-pool balance matters, but it is not a magic fix. On the current naturalized full-hybrid slice, domain-balanced replay preserves low `ASS_total` and fails to recover `AMCD`, while blocksworld-focused replay still reintroduces high answer sensitivity.
- On the current full-hybrid studies, `AMCD` is a much better utility proxy than ordinary AUROC. Head-level ordinary AUROC can look respectable while exploitability barely changes, but `AMCD` tracks both selection gain and exploitability.
- Minimal explicit factorization does not automatically help. On the current frozen-backbone pilot, a dual-head `G_total = G_proc + alpha * G_cons` can legitimately select `alpha = 0.0`, meaning the consistency channel is too weak or too noisy to improve the frontier.
- Even a more faithful step-scan `G_proc` does not automatically rescue dual-head disentanglement. If validation still chooses `alpha = 0`, the issue is likely signal quality or scale, not just the earlier full-trace process-head shortcut.
- A stronger reranker does not automatically fix answer leakage. On the current full-hybrid OOD slice, `Qwen3-Reranker-4B/8B` both work substantially better in the masked condition than in the visible condition.
- Utility and calibration can separate cleanly. The current `Qwen3-Reranker-8B` masked setup improves `AMCD`, selection gain, and exploitability, while `ECE / Brier` remain weak enough that calibration should stay an appendix-style enhancement rather than a core mechanism claim.
- If a stronger masked reranker already beats both the visible reranker and the frozen-head repair under mixed attacker transfer, stop treating scale and disentangling as competing stories. At the current stage they compound: the best deployment model is the stronger model in the masked condition.
- Reproduction matters even for apparently clear frontiers. On the current naturalized main slice, the masked reranker remains strongest after `3` frozen-head seeds and paired bootstrap, so the deployment conclusion is no longer a one-run artifact.
- Blind-audit packets can silently degenerate if they are exported from quartet records without explicit sampling controls. Use domain-balanced quartet selection, randomized trace order, and randomized verbalizer choice; otherwise the packet can collapse to the first few quartets and a single verbalizer family.
- A well-packaged blind-audit packet can still fail the substantive audit if invalid traces keep the clean closing line. Reviewers can spot copied endings much faster than deeper process artifacts, so benchmark hardening must edit or regenerate the ending consistently with the corrupted local step.
- Removing the copied-ending artifact is not enough by itself. Once the blatant ending mismatch is fixed, reviewers can still detect algebra and graph invalid traces through repeated single-step numeric corruption under nearly identical templates; planning traces improve earlier than arithmetic traces.
- `LLM-assisted` surface generation is viable but does not automatically pass artifact audit. On the current pilot, API rewriting solved the engineering problem (`72/72` rewritten traces) while leaving a domain split: blocksworld benefits most, algebra stays visibly scaffolded, and graph/path still exposes patched comparison totals.
- Reviewer-in-the-loop filtering for counterfactual families should operate on masked traces, not answer-visible traces. Otherwise swapped-answer contradictions dominate the reviewer signal and corrupt family selection before blind audit even begins.
- A strict multi-candidate benchmark constructor can still select the wrong families if its score mixes detectability and label-alignment too loosely. The first `benchmark-v3` smoke selected graph/path families where the valid trace looked more synthetic than the invalid trace.
- A real hard detectability gate can zero out coverage entirely. On the current `benchmark-v3` smoke with `k=2` and `max_pair_detectability = 0.8`, all three domain-balanced quartets are rejected, so strict benchmark work now bottlenecks on candidate quality rather than selector bookkeeping.
- Unary single-trace reviewer pruning can be too blunt to be useful. On the current benchmark-v3 cascade pilot, it labeled almost every candidate as artificial, while within-role pairwise pruning preserved coverage and exposed a more meaningful bottleneck.
- Pairwise candidate pruning and pairwise family gating diagnose different things. In the current benchmark-v3 smoke, within-role pairwise pruning can identify cleaner candidates, but the final valid/invalid pair still fails because side-by-side contrast exposes patched seams that single-candidate ranking misses.
- Pair-conditioned generation can improve candidate survival without fixing pair-level detectability. In the current benchmark-v3 smoke, joint valid/invalid rewriting lowers cost and makes within-role candidate rankings cleaner, but the final pair gate still rejects all quartets.
- Preserving joint pairs during pruning is cleaner than role-wise top-1 pruning, but it still may not be enough. In the current benchmark-v3 smoke, pair-level adversarial prune improves the abstraction and slightly improves some pair scores, yet still accepts `0` quartets under the strict gate.
- Contrast-aware pair prompts can saturate quickly. In the current benchmark-v3 smoke, explicitly asking for balanced side-by-side naturalness does not materially improve the final pair-detectability scores over the simpler pair-conditioned setup.
- Full-trace rewrite with stepwise fallback needs an explicit post-pass sanity check on step counts and quartet completeness. A fallback assembly bug can silently produce plausible-looking truncated traces and poison a benchmark slice before any headline metric fails.
- Full Cartesian family search gets expensive immediately. Even `2` candidates per role over `3` quartets already costs `216` API calls in the current `benchmark-v3` strict smoke, so any scale-up needs pruning or cascaded reviewer filtering.
- A simple within-role pairwise cascade can cut benchmark-v3 reviewer cost sharply. In the current smoke it reduces usage from `226` calls to `70` while preserving one candidate per role, so it is a better pruning primitive than unary reviewer screening.
- Pair-conditioned generation compounds that cost win. On the current smoke, joint pair rewriting plus within-role pairwise prune lowers total usage further to `48` calls while preserving all roles, so it is a useful efficiency primitive even before it solves blind-clean selection.
- Pair-level adversarial prune preserves that efficiency while avoiding pair breakup. On the current smoke it uses `49` calls and keeps the best jointly generated pair intact, so future work should build on this primitive rather than reverting to role-wise pruning.
- Once pair-level adversarial prune and contrast-aware pair prompts both plateau at `0` accepted quartets, the next likely leverage is not more prompt wording. It is a stronger generator or a stronger reviewer adversary.
- Lower shallow style AUROC is not enough to declare a benchmark-v2 win. Human-style blind review still keys on local patched seams and overt discourse-marker scaffolding that a shallow classifier may not weight strongly.
- Global anchor validation is not enough for graph-style traces. If full-trace rewriting can copy edge weights from the problem back into a step, it can silently recreate a human-visible arithmetic seam while still passing coarse anchor-preservation checks.
- Benchmark-v2 hardening now has a real coverage-vs-naturalness frontier. Stricter algebra rewrite guards can materially improve shallow style leakage while collapsing algebra coverage, so coverage must be treated as a first-class benchmark metric alongside artifact audit.
- Benchmark-v2 also has a coverage-vs-blindness split. Recovering algebra coverage by rescuing malformed full-trace payloads can raise rewritten counts without improving blind review, and can even reintroduce shallow length leakage in other domains.
- Source-level realism and rewrite robustness can fight each other. On the current algebra branch, replacing patched endings with more local “cancel the coefficient” slips made the source traces look better in isolation but broke enough LLM rewrites that the branch lost all algebra blind-packet eligibility.
- Rewrite faithfulness and blind-audit quality can also fight each other. Once algebra and graph rewrites preserve the wrong local arithmetic exactly, human reviewers can detect the invalid traces more easily than before, even if coverage becomes perfect.

## Pitfalls

- Prompt-based judge pilots are easy to misread as verifier evidence. High answer sensitivity at the judge interface does not imply the trained verifier backbone is using the same shortcut.
- Naturalization prompts can drift labels if they are allowed to rewrite the final answer line. Keeping the original answer line fixed is safer for audited transfer pilots.
- Blocksworld generation can silently create noop or repeated states if table moves are not filtered; this hurts blind audit quality and can inflate spurious shortcuts.
- Model-generated traces can fail the audit gate long before scoring becomes the bottleneck. If executability or local conclusion alignment collapses, reporting only `ASS` hides the real failure mode.
- Hidden reasoning during constrained generation can waste the token budget inside `<think>` and leave no auditable visible trace. For small local generators, stronger output constraints plus `/no_think` can be more reliable than private reasoning.
- Repair objectives can overfit the synthetic validation slice even when the loss is conceptually aligned with the proposal. If a joint repair term looks much better in-domain, re-check it on the hardest generated OOD slice before upgrading it to the default method.
- Low-`ASS_total` repair wins do not automatically survive light language shift as `AMCD` wins. A naturalized re-evaluation can reveal that the method is buying faithfulness more reliably than local discrimination.
- Hard-negative replay can look faithful on synthetic validation and still fail the OOD faithfulness test if the mined pool is skewed or too easy. The replay construction itself needs audit, not just the downstream model.
- If targeted replay still fails after fixing domain coverage, stop treating mining skew as the main explanation. The underlying replay objective may itself be moving the model along a frontier rather than dominating it.
- Head-level frontiers can hide the answer-swap story. If pooled `ASS_total` looks ambiguous across methods, inspect quartet-level local relations; local answer sensitivity can still strongly predict lower local `AMCD` and worse utility.
- A weak consistency head can make a dual-head result look vacuous when it is actually diagnostic. If validation sets `alpha = 0`, that still tells you explicit separation alone is insufficient at the current model/data scale.
- Deployment-faithful local aggregation can still underperform a simpler visible process head. Matching the proposal more closely does not guarantee better frontier behavior if the local head is too weak.
- Better reranking scores can hide poor confidence quality. If a stronger reranker improves `selection_gain_at4` and lowers exploitability, still inspect `ECE / Brier / AURC` before treating it as a calibrated verifier.
- Cross-family attacker transfer is more informative than self-attack alone once several model families exist. On the current setup, the mixed visible-attacker ensemble exposes a larger gap between `reranker8_masked` and `reranker8_visible` than average-case metrics do.
- Stable hyperparameters are informative. On the current repair study, `visible_cond_swap` repeatedly selects `lambda_cond_swap = 2.0`, which means its remaining weakness is not obvious tuning instability.
- Quartet-level role coverage is not enough for blind-audit export. The exporter must verify that the same verbalizer still has both `valid_correct` and `invalid_correct`, or incomplete quartets can silently survive until packet construction time.
- Preserving the exact wrong equation fragment is effective for validator compatibility, but it can overexpose the benchmark to human reviewers. In arithmetic-heavy domains, “faithful invalidity” can be worse than lightly naturalized invalidity for blind audit.
- For reviewer-aware benchmark construction, low invalid-pick rate is not enough. If one trace is still obviously synthetic, the family is not blind-clean even if reviewers point at the valid trace instead of the invalid one.
- In strict benchmark construction, do not “fix” zero coverage by silently relaxing the gate. A `0-selected-family` smoke run is a useful result because it localizes the problem to upstream generation or reviewer mismatch instead of hiding it under wider acceptance.
- If pairwise cascade preserves role coverage but the final pair gate still rejects everything, stop treating selector tuning as the main path. The next leverage point is pair-conditioned generation or pair-level adversarial filtering.
- If pair-conditioned generation still leaves the final pair gate at `0` accepted families, stop treating joint rewriting alone as the next breakthrough. The next leverage point is explicit pair-level adversarial filtering or pair-conditioned contrast objectives.
- If pair-level adversarial prune still yields `0` accepted families, stop treating selector engineering as the main blocker. The next leverage point is stronger pair-contrast generation or a stronger reviewer adversary.
- If explicit pair-contrast prompts also yield `0` accepted families, stop treating prompt engineering as the next blocker. The remaining leverage is model strength or reviewer strength.
- A stronger reviewer ensemble can create a new failure mode if aggregation is too lenient. On the current benchmark-v3 smoke, `api + local Qwen3-8B` promotes `3/3` families under the scalar gate while blind-facing reviewers still reject the packet, because frequent `both` labels collapse pair detectability too aggressively.
- Adversarial max-penalty aggregation is a useful sanity check on reviewer ensembles. On the same benchmark-v3 candidate pool, switching from permissive union aggregation to max-penalty aggregation collapses `3/3` promoted families back to `0/3`, which is strong evidence that the earlier promotion was a reviewer-aggregation artifact.
- Cross-family reviewer confirmation matters. On the current benchmark-v3 candidate pool, a different critic-style reviewer family (`CriticLeanGPT-Qwen3-8B-RL`) also returns `0/3` promoted families under the strict gate, which means the bottleneck is not just one reviewer family's quirk.
- A locally cached stronger model is not automatically usable. `Qwen3.5-35B-A3B` exists on disk, but the current `infer` environment cannot load `qwen3_5_moe`; environment support can become the next blocker before model quality does.
- Stronger API usage can move the pool without breaking the conclusion. On the current benchmark-v3 smoke, `k=2 + 1` reviewer-guided regeneration round produces new candidate pairs and slightly lowers some algebra penalties, but still leaves every domain above the `0.8` strict gate. One feedback round is not enough.
- More regeneration depth alone is not enough either. On the current benchmark-v3 smoke, pushing the same API-native loop from `1` to `2` rounds drives cost from `58089` to `103912` total tokens while leaving all domains stuck at a `0.9` detectability plateau. Once that happens, the next lever should be broader search or different branching, not just more rounds.
- Width is now a better API-native lever than extra depth, but still not sufficient on its own. On the current benchmark-v3 smoke, moving from `top_k=1` to `top_k=2` in a single regeneration round costs `69618` total tokens, stays below the `2`-round cost of `103912`, and reaches the same `0.9` plateau. Once that happens, the next breadth move should happen earlier or with a larger pool, not by returning to deeper top-1 chaining.
- Larger initial pools are not automatically a better breadth move. On the current benchmark-v3 smoke, increasing `candidates_per_role` from `2` to `3` while keeping `top_k=2` raises cost from `69618` to `93053` total tokens and still leaves every domain at the same `0.9` plateau. If breadth continues, it should come with cheaper screening or a different branching policy, not just one more initial sample per side.
- A local-reviewer cascade can save strict-review budget without fixing quality. On the current benchmark-v3 smoke, local cheap pruning collapses each quartet to one surviving family under a looser `0.9` screen, and the final strict rerun then needs only `9233` tokens, but all domains still land at `0.9`. Treat cascade as an efficiency primitive unless the second stage also changes generation.
- Regenerated candidates must not receive `..._b3_b3` verbalizer ids. If pair-conditioned regeneration uses a `CandidateTrace` as source, the benchmark verbalizer suffix has to be normalized or candidate-input reuse silently breaks.
- Survivor-conditioned second-stage regeneration is not automatically a better cascade. On the current benchmark-v3 smoke, regenerating from the locally pruned survivors still leaves all domains at `0.9` and raises total cost above both one-shot breadth baselines. If the next stage continues, it must change the feedback objective, not merely regenerate from cleaner inputs.
- Proposal-aligned benchmark quality is weaker than human-indistinguishable pair quality. For this project, the hard requirement is artifact-clean audited counterfactuals: preserve quartet semantics, pass shallow/length audits, avoid obvious templating or patched seams, and survive blind audit against “artificially edited trace” cues. It is *not* necessary to make structured algebra invalid traces visually indistinguishable from valid ones under a universal strict pairwise detectability gate.
- Prompt optimization should follow the same standard. In benchmark-v3, regeneration feedback should target surface artifacts only; comments like “incorrect arithmetic” or “illegal move” are not valid prompt objectives, because they measure semantic wrongness visibility rather than non-semantic benchmark corruption.
- Once the benchmark target is corrected, re-score old runs under acceptance-aligned modes before spending more API budget. On the current smoke, the universal strict gate stays at `0/3`, but ignoring `semantic_only` reviewer penalties already yields `2/3`, and counting only explicit surface-or-mixed penalties yields `3/3`. That kind of split can reveal that the main remaining bottleneck is one domain's surface artifacts, not universal benchmark failure.
- In blocksworld, a narrow canonical move-report surface is enough to kill most blind-facing artifact cues. Standardizing every step to the exact source move action plus resulting state turns reviewer reasons from `templated phrasing / system moves / four separate stacks` into mostly `extra move / more direct` comments. Under the proposal-aligned acceptance rule, that is a real improvement even if the universal strict scalar stays at `0.9`.
- In algebra, compact parallel operation framing helps more than trying to sound varied. For the current benchmark-v3 smoke, fixing every step to the same operation-driven register does not reduce the universal strict penalty, but it does move same-quartet acceptance from `0.0 / 0.45` to `0.0 / 0.0` under the proposal-aligned rescore. That means the remaining reviewer signal is mostly semantic wrongness visibility, not surface patchiness.
- Once the proposal-aligned acceptance rule reaches `3/3` on the integrated smoke, stop optimizing benchmark-v3 against the universal strict scalar gate. At that point the strict `0/3` should be treated as a semantic-visibility diagnostic, not as evidence that the benchmark is still surface-artifact dirty.
- The same stopping rule survives initial scale-up. Once the proposal-aligned acceptance rule holds at `6/6` on a mini-benchmark rerun, the remaining benchmark-v3 bottleneck is cost/diversity rather than surface-artifact cleaning; further prompt tuning against the universal strict scalar is wasted budget.
- If benchmark-v3 promotion and acceptance analysis disagree, first check whether the builder is re-reviewing already-pruned pairs under a different stochastic path. Reusing pair-prune reviewer results inside family selection both stabilizes export and cuts candidate-pool reuse cost substantially.
- At mid-scale, blocksworld is the dominant failure slice even after proposal-aligned acceptance correction. In the current `6 quartets/domain x 3 verbalizers` export, algebra accepts cleanly, graph/path mostly accepts, and most remaining failures come from blocksworld pair-prune rejections rather than global benchmark invalidity.
- Acceptance analysis should ignore non-pair rows in export review logs. Once family-selection rows are mixed into the same JSONL, the analyzer must filter to rows with `answer_variant_group` and `candidate_index`, otherwise it will over-assume the review schema and crash.
- In planning domains, words like `natural` and `structure` are too broad for acceptance taxonomy. Comments such as `forced sequence`, `more direct`, `table clearing`, or `unnecessary intermediate steps` usually describe semantic/task-structure visibility, not surface artifact; treating them as artifact cues can materially undercount acceptable benchmark families.
- When acceptance taxonomy changes, re-export from the existing candidate pool deterministically instead of rerunning reviewer selection. A stable `acceptance_analysis -> export` path avoids stochastic drift and cleanly separates benchmark-quality correction from generation-quality changes.
- If benchmark-v3 verbalizer ids carry an extra generation suffix like `_b3`, split utilities must parse the slot before the suffix. Otherwise verbalizer-holdout evaluation silently stays tied to the legacy benchmark naming scheme and fails on the first real rerun.
- Benchmark-v3 mid-scale does not behave like the old full-hybrid OOD slice. On the same `Qwen3-Reranker-8B`, masked still clearly improves `ordinary_auroc / AMCD / ASS_total`, but visible no longer buys extra selection utility. Treat benchmark-v3 as a cleaner process-faithfulness benchmark, not as a drop-in replacement for deployment-style exploitability stress.
- Once benchmark-v3 is used as the default dataset, do not assume the old weak slice persists. In the first same-dataset robustness summary, blocksworld stops being the dominant failure source and graph_path becomes the main weak group for both rerankers and frozen-head baselines.
- Executable synthetic generators are safer as validators than as primary source-benchmark constructors. The current blocksworld source bug shows that planning-domain `invalid` generation can silently collapse into “alternative legal plan” rather than “wrong process”; main benchmark source data should therefore be anchored in external step-level datasets, with synthetic generators downgraded to auxiliary stress suites.
- `ProcessBench` confirms why whole-trace external data should be handled differently from quartet benchmarks. `label=-1` forms a strict subset of `final_answer_correct=true`, while many answer-correct traces still carry an earlier wrong step. This makes it a strong external source for process-local supervision, but not a drop-in replacement for matched counterfactual metrics such as `AMCD/ASS`.
- Prefix construction on external step-level data gives a natural replacement for synthetic local counterfactuals. On `ProcessBench`, treating prefixes before the first error as valid and prefixes from the first error onward as invalid yields a usable `PB-Prefix` benchmark with measurable boundary drops, without inventing new latent semantics.
- On external-source whole-trace benchmarks, stronger rerankers can be rank-informative but badly miscalibrated. The first `ProcessBench` run of `Qwen3-Reranker-8B` keeps `ordinary_auroc` in the low `0.7`s while threshold accuracy collapses to about `0.35` because nearly every trace receives a high score. For this regime, report AUROC and calibration together; raw `0.5`-threshold accuracy alone is misleading.
- Validation-threshold diagnostics matter on external-source rerankers. On `ProcessBench`, selecting a threshold on the grouped validation split raises `Qwen3-Reranker-8B` test accuracy from about `0.35` to about `0.70` without changing AUROC, confirming that the big collapse was mostly calibration rather than ranking failure. Use `val -> test` threshold transfer before calling a reranker result fully negative.
- External-source `ASS` can be built without synthetic quartets if the dataset is maskable. On `ProcessBench`, answer-only swaps over the masked answer span are enough to create observed/swapped pairs; the visible API judge then shows a large score delta while the masked view collapses close to a noise floor. This is a practical way to recover answer-sensitivity evidence on real step-level traces before attempting external-source `AMCD`.
- External-source local pair construction is much harder than external `ASS`. On `ProcessBench`, answer-matched repair pairs from `invalid_correct` traces remain coverage-limited even after switching from whole-trace repair to suffix-only repair and excluding `audited_locus = 0`. Treat external local `AMCD` as a separate benchmark-construction problem, not as a quick extension of answer-swap stress.
- For external local-pair construction, parser hardening can buy real coverage before any prompt redesign. On `ProcessBench`, mixed prose + trailing JSON, escaped-JSON-as-string responses, and lone LaTeX backslashes inside JSON were a larger failure mode than repair semantics; fixing those parser/runtime issues raised repair-pair coverage from `4 / 8` to `8 / 8` on the same pilot selection.

## Heuristics

- Use `AMCD + ASS + same-backbone masked baseline` together as a minimal audit bundle. Any single metric is too easy to overinterpret.
- When testing natural transfer cheaply, start from held-out synthetic test quartets and rewrite only the language surface. This preserves executable labels while exposing judge-style leakage under distribution shift.
- If a naturalization model is unstable, prefer deterministic fallback templates over silently accepting semantic drift.
- For model-generated traces, deployment-style answer-swap tests are much easier to scale than full `AMCD`, because local validity labels do not come for free once the trace is genuinely free-form.
- When pushing `AMCD` onto model-generated traces, first report audit coverage and rejection reasons by domain. The coverage profile itself is a result and can reveal that planning domains fail earlier than algebraic ones.
- Route-style natural traces can often be recovered with exact conclusion-format constraints, but planning-style traces may need a stronger scaffold than prompt-only tightening once executability becomes the bottleneck.
- A legality-aware scaffold can recover planning coverage without making answer sensitivity explode. On the recovered blocksworld slice, the main shift was lower `AMCD` for everyone, not higher trained `ASS_total`.
- If repair gains appear first on planning-heavy OOD slices, inspect per-domain AUROC before declaring a global win. On the current full hybrid slice, `cond-swap` repair mostly helps blocksworld while giving back some algebra ordinary AUROC.
- If a naturalized OOD slice turns `masked_bce` answer-sensitive again, treat language shift as a real stressor rather than a presentation tweak. It can move the whole frontier, not only the prompt judge.
- If self-mined hard negatives are heavily skewed away from the hardest domain, do not overinterpret the replay result. The mining distribution can determine whether `hard-neg` behaves like useful replay or like shortcut amplification.
- For mechanism analysis, a lightweight quartet-level mediation is already informative. On the current setup, most of the effect of local answer sensitivity on selection gain/exploitability runs through local `AMCD`, not a large residual direct path.
- For Qwen3 rerankers, treat visible vs masked as a document-side ablation. Keeping the query fixed and swapping only the answer-bearing trace text gives a clean Week 4 test of whether direct answer access helps or hurts ranking.
- Once worst-group analysis is available, use it to choose the next realism target rather than adding more average-case plots. On the current project, naturalized blocksworld is the clearest remaining weak slice for the best masked reranker.
- If a baseline is seed-unstable, do not let a single lucky run define the story. `pairwise_visible` looked competitive on some slices, but its Week 6 seed spread is large enough that it should stay a baseline, not the deployment target.
